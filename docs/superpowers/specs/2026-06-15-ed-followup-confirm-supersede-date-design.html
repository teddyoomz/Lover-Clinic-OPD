ED Follow-up Assessment v2 — Confirm-Identity · Latest-Link-Only · Round Dates

Date: 2026-06-15 (EOD+2) · Status: Approved by user (Q1–Q4 locked) · Builds on: ED Score feature (2026-06-15, prod 019df953) · Plan: see docs/superpowers/plans/2026-06-15-ed-followup-confirm-supersede-date.html

Purpose / Problem

The ED follow-up assessment link is generated from a specific customer's page (CDV → EDFollowupModal), so the clinic already knows who it's for. Four user-reported problems + one bug surfaced while reading the real submit path:

  • R1 — The follow-up link asks the customer to re-type their name/age/blood-type/date. Pointless (we know the customer) and error-prone. Change to a read-only identity card so the customer confirms it's the right person, then assesses.
  • R2 — A meaningless token chip FW-ED-db69fb0d… shows in the header. It means nothing to a human → remove it from every PatientForm mode.
  • R3 — Generating a new QR/link while an old one is still unanswered leaves multiple live links → duplicate customer submissions. Only the latest link per customer+branch may be valid; older pending ones are deleted on generate.
  • R4 — The TFP "หมายเหตุทั่วไป" ED-Score strip shows each round's scores but not when each round was assessed → a doctor can't tell which round was done today. Add the date per round + a "วันนี้" badge when the round's date is today.
🔴 Bug found (live on prod, in-scope fix): PatientForm.jsx:350 validates if (!formData.province) with no mode gate. The follow-up form never renders a province field and defaultFormData.province === '' → a customer who fills the follow-up assessment and presses Send is blocked with "กรุณาเลือกจังหวัด" and cannot submit. The existing L2 e2e wrote Firestore directly (bypassing this client-side validation) and L1 was deferred, so it shipped unnoticed. Fix: gate to if (isIntake && !formData.province) (province is an intake/deposit address field) — unblocks follow-up and custom submit.

Decisions (locked via Q&A with user)

  • R1 fields: ชื่อ-สกุล + อายุ + เบอร์โทร (NO HN — customers don't know it).
  • R1 phone: masked middle 081-•••-5678 (safer on a public anon link, still recognizable).
  • R1 interaction: passive read-only card (assessment directly below) — no confirm-gate button.
  • R1 fallback: no confirmInfo on the session → render the original editable fields (already-sent links keep working).
  • R2: remove the session pill from every PatientForm mode (intake/deposit/follow-up); sessionId stays as the doc-id internally.
  • R3: hard-delete prior pending follow-up sessions + their linked pending rounds for the same customer+branch, before creating the new one ("ลบทิ้งไปเลย").
  • R4: date per round, dd/mm/yyyy พ.ศ., + "วันนี้" badge when date == today; intake date from patientData.assessmentDate → fallback customer.createdAt.
  • 🔴 Bug-fix: province validation gated to isIntake — folded into this spec because it's the submit path of the form being redesigned.

Acceptance Criteria

  1. Generating a follow-up link snapshots confirmInfo; the customer link renders a read-only card (name neutral, phone masked) and NO editable name/age/date inputs and NO token pill.
  2. A follow-up link generated BEFORE this ships (no confirmInfo) still renders the editable fields (no breakage).
  3. A customer can fill the assessment and submit successfully (province no longer blocks) → a be_assessments round is materialized.
  4. Generating a 2nd link for the same customer+branch while the 1st is still pending deletes the 1st session + its pending round; opening the 1st link shows "ไม่พบ/หมดอายุ"; the 2nd link works.
  5. The TFP ED-Score strip shows each round's date (dd/mm/yyyy พ.ศ.); a round assessed today shows the "วันนี้" badge; the intake round shows its admission/assessment date (not blank).
  6. No token pill anywhere in PatientForm (intake/deposit/follow-up).
  7. Verify bar (user: "perfect 100%"): pure-unit (maskPhone/buildConfirmInfo/formatRoundDate/supersede-predicate) + RTL + Rule I flow-simulate + Rule Q L2 on real prod (real client-SDK supersede + submit) + Rule Q L1 real browser (real link → card → submit passes → materialize → TFP date+วันนี้, both themes). Honest disclosure of any L1 gap if no browser is connected. Full vitest green + build clean.

Risks / Open Questions

  • Concurrent generate (Rule T): two admins generating simultaneously could both pass supersede and create two pending links. Manual once-in-a-while flow → negligible; documented with a ponytail: comment, no lock.
  • Supersede partial failure: a delete failing mid-batch could leave one old pending link. Best-effort per-item + proceed-to-create keeps the clinic able to send a link; common case (deletes succeed) meets the goal.
  • L1 browser availability: real-browser L1 needs a connected Chrome MCP browser; if none is connected at execution, the L1 step is disclosed as a gap and handed to the user (Rule Q-honest) — never claimed as done.
  • Date era: spec uses พ.ศ. to match the feature's existing 2569 surfaces; switchable to ค.ศ. with a one-line change if the doctor prefers.
  • CDV consistency (out of scope): the CDV EDScoreBox history could show round dates too for parity, but only the TFP note was requested — left as an optional follow-up (YAGNI).
