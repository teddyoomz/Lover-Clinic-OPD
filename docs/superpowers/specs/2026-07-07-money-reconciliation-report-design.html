💰 Money Reconciliation Report — ตรวจความครบธุรกรรม

Origin: V155/V157 residual — a sale/TFP auto-sale writes ~11 collections non-atomically. V157 SURFACES failures at save-time (sideEffectWarnings + alert), but there is no retro verification. This feature closes the last unguarded corner of the money system.

User decisions (AskUserQuestion, 2026-07-07): ทำทั้งคู่ — (a) on-demand tab in หน้ารายงาน Backend + (b) nightly cron + admin notice. Read-only; no new Firestore collection (Rule C3 — cron result reuses be_admin_audit).

Flow

Tab (on-demand):

เลือกช่วงวันที่→ loadSalesByDateRange({branchId})→ ต่อใบ: fetch deposit/wallet/points/courses/movements ที่อ้าง saleId→ pure checker: reconcileSale(sale, evidence) → {channel: verdict}→ ตาราง + drill-down (READ-ONLY)

Cron (nightly, หลัง 03:00 backup):

api/cron/money-reconciliation-sweep.js (CRON_SECRET)→ admin SDK: sales ของเมื่อวาน (ทุกสาขา)→ SAME pure checker (shared src/lib/reconcileSaleCore.js)→ เขียนสรุปลง be_admin_audit/recon-daily-YYYYMMDD-*→ tab แสดง banner ผลล่าสุด (discrepancy > 0 = amber)

SSOT: ONE pure module src/lib/reconcileSaleCore.js (no Firebase imports) consumed by BOTH the tab (client SDK data) and the cron (admin SDK data) — no drift between surfaces. Cron writes ONLY the summary + offending saleIds (no new collection; be_admin_audit create-prefix pattern ala system-config audit). Zero Firestore rules change expected (be_admin_audit read already isClinicStaff for the audit panel; VERIFY at impl — if recon docs aren't readable, surface the banner from the tab's own on-demand re-check of yesterday instead of reading the cron doc).

Guardrails

  • READ-ONLY everywhere — no fix-buttons in v1 (พบปัญหา → admin ตามลิงก์ไปแก้ที่หน้าจริง)
  • Deterministic-only discrepancies (deposit/wallet/points/courses + cancelled-reversal); stock = INFO
  • Branch scope: tab follows BranchSelector (BS-11 pattern, registered in the audit); cron = all branches
  • Money numbers via roundTHB comparison tolerance 0.01 (V156)
  • Tests: pure-checker unit bank (per-channel matrix + adversarial) + Rule I flow-simulate + Rule Q L2 against real prod sales
