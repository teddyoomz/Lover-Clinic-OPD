Stock — ค้นหา order ตามชื่อสินค้า (line item) + แก้วันหมดอายุต่อ batch/lot

วันที่: 2026-06-03 · สถานะ: Approved by user (design) · Scope: per-branch tab=stock (+ central order-search ตาม Rule P) · Tag: V159 (สองงานในชุดเดียว) · Deploy: frontend + backendClient เท่านั้น — ไม่แตะ firestore.rules → ไม่ต้อง Probe-Deploy-Probe

Purpose / Problem

  1. ค้น order นำเข้าไม่เจอชื่อสินค้าที่อยู่ "ไส้ใน": ฟิลเตอร์หน้า order (tab=stock ของแต่ละสาขา) จับแค่ vendorName + orderId ไม่จับ items[].productName. เช่น "elonza" เป็น 1 ใน 10 รายการของ order หนึ่ง → ค้นแล้วไม่เจออะไรเลย. (บั๊คเดียวกันที่คลังกลาง CentralStockOrderPanel.)
  2. แก้วันหมดอายุของ batch/lot ไม่ได้ถ้าไม่ไปแก้ที่ order: admin คีย์วันหมดอายุผิดได้ง่าย ต้องแก้ได้ตลอด ต่อ batch/lot ทันที แม้ lot นั้นถูกใช้ไปบางส่วนแล้ว — โดยไม่ต้องย้อนไปแก้ที่ order นำเข้า.

Flow

Feature 1 — search predicate (per order)

ขั้นเกิดอะไร
1admin พิมพ์ q = "elonza" ในช่องค้นหา
2filteredOrders filter ต่อ order: match ถ้า vendorName.includes(q) OR orderId.includes(q) OR items.some(it => it.productName.includes(q)) ← เพิ่มใหม่
3ถ้า match ที่ line item → formatOrderItemsSummary(items, q) ดันตัวที่ match ขึ้นก่อน + ไฮไลต์ (Q1=B) → แถวโชว์ "Elonza ×10, …"
4เหมือนกันที่ CentralStockOrderPanel (Rule P). productName ใน line = snapshot ตอนนำเข้า → ค้นตาม snapshot (ไม่ต้อง live-resolve)

Feature 2 — expiry edit cascade (1 runTransaction, atomic)

กดบันทึก ในฟอร์ม (เลือก batch แล้ว, newExpiresAt ≠ batch.expiresAt)
▼ ถ้า qty > 0 → createStockAdjustment() เดิม (transaction แยก: batch qty + movement + adjustment 'add'/'reduce')
▼ ถ้า expiry เปลี่ยน → updateStockBatchExpiry({batchId,newExpiresAt,note,branchId},{user})
runTransaction:
  1. tx.get(batch) → จับ oldExpiresAt จาก server
  2. tx.update(batch, {expiresAt:new, expiresAtLegacyValue:old, expiresAtEditedAt, expiresAtEditedBy, status:normalize(remaining), updatedAt}) — status normalize: ถ้า remaining>0 → ACTIVE (ล้าง EXPIRED ที่ค้างจากวันเก่า)
  3. tx.set(be_stock_adjustments/{id}, {type:'expiry', batchId, productId, productName:liveResolve(productId), oldExpiresAt, newExpiresAt, note, branchId, user, createdAt, linkedMovementId:null}) — Q3=C; ไม่เขียน be_stock_movements
  4. Q4=B sync: ถ้า batch.sourceOrderId + batch.orderProductId มี → resolve order collection จาก locationType (branch→be_stock_orders / central→be_central_stock_orders) → tx.get(order) → map items[] set expiresAt ที่ orderProductId ตรงกัน → tx.update(order, {items}). ไม่มี order ต้นทาง → ข้าม (เช่น batch จาก AUTO-NEG/transfer)
▼
มีผลทันที: FEFO (batchFifoAllocate/hasExpired/pickNegativeTargetBatch) อ่าน batch.expiresAt สดทุกครั้ง → การตัดสต็อครอบถัดไปใช้ค่าใหม่เลย; lot ที่เคยติด "หมดอายุเพราะคีย์ผิด" กลับมาใช้ได้

Decisions (locked via Q&A)

  • Q1 = B — order match line item → แถวโชว์ + ดันรายการที่ match ขึ้น/ไฮไลต์ (กันปัญหา summary โดนตัดก่อนถึงตัวที่ค้น)
  • Q2 = แก้ในฟอร์มปรับเดิม (user directive) — ไม่ทำปุ่ม ✎ บนตาราง; ใช้ batch selector ของ AdjustCreateForm ที่มีอยู่ → lot เดียว/หลาย lot เลือกใน dropdown เหมือนกัน → ครอบ 2 หน้าด้วย control เดียว
  • Q3 = C — audit doc ใน be_stock_adjustments type 'expiry' (ครบประวัติ) + ฟิลด์ forensic บน batch (โชว์ inline); ไม่ เขียน be_stock_movements
  • Q4 = B — sync สองทาง batch ↔ บรรทัด order ต้นทาง (branch/central) ใน transaction เดียว
  • Edge — dropdown รวม lot ที่ "หมดอายุเพราะคีย์ผิด" (remaining>0); status normalize ตอนแก้ → วันใหม่ในอนาคต = FEFO ใช้ได้กลับมา; ตัด cancelled/depleted ออก
  • No rule change → no Probe-Deploy-Probe; deploy รอ user สั่ง "deploy" (V18)

Acceptance Criteria

  1. ค้น "elonza" ที่ OrderPanel + CentralStockOrderPanel → เจอทุก order ที่มี elonza เป็น line item (รวมตัวที่อยู่ลำดับท้ายๆ ของ items[]); แถวแสดงให้เห็นว่า match เพราะรายการไหน
  2. ค้น vendor / ORD-... ยังทำงานเหมือนเดิม (ไม่ regress)
  3. ในฟอร์มปรับ (ทั้งจากหน้ายอดคงเหลือ + หน้าปรับสต็อค): เลือก batch → แก้ "หมดอายุ" → บันทึก → be_stock_batches.expiresAt เปลี่ยนทันที
  4. แก้ expiry อย่างเดียวได้ (ไม่กรอก qty) และ แก้ qty+expiry พร้อมกันได้
  5. แก้ได้แม้ lot ถูกใช้ไปบางส่วน (remaining<total) และ แม้ lot ติด "หมดอายุเพราะคีย์ผิด"
  6. วันใหม่เป็นอนาคต + remaining>0 → FEFO กลับมาตัด lot นั้นได้ (status ไม่ค้าง EXPIRED)
  7. เขียน audit doc type 'expiry' (old→new+เหตุผล+ผู้ทำ) + ฟิลด์ forensic บน batch; conservation/รายงานสต็อคไม่เพี้ยน (ไม่มี movement)
  8. บรรทัดสินค้าใน order ต้นทาง sync วันใหม่ด้วย (ถ้ามี); ไม่มี order ต้นทาง → ไม่ error
  9. idempotent + concurrency-safe (สองคนแก้ expiry batch เดียวกันพร้อมกัน → last-write-wins ถูกต้อง, audit ครบ, ไม่ทำ qty/conservation พัง)
  10. full vitest เขียว + build clean + real-prod Rule Q L2 e2e ผ่าน

Risks / Open Questions

  • ตำแหน่ง formatOrderItemsSummary: ต้องหาไฟล์จริง (import ใน OrderPanel) + ดูพฤติกรรม truncation ตอนใส่ highlight (plan จะ locate)
  • status EXPIRED ถูก persist หรือ derive? ต้องยืนยันว่ามี cron/field set status=EXPIRED จริงไหม → ถ้ามี ต้อง normalize ตอนแก้ (รวมไว้ใน transaction แล้ว); ถ้า derive จาก expiresAt ล้วน ก็ไม่ต้องทำอะไรเพิ่ม
  • batch dropdown ขยาย scope: listStockBatches({status:'active'}) ปัจจุบันตัด expired — ต้องดู semantics ของ status param เพื่อรวม expired-remaining>0 โดยไม่ดึง cancelled/depleted; กระทบ qty-adjust ของ lot ที่หมดอายุด้วย (เพิ่ม/ลด lot หมดอายุ = write-off ปกติทำได้)
  • คลังกลาง — ฟอร์มปรับ: ยืนยันใน plan ว่า central warehouse adjust ใช้ AdjustCreateForm ตัวเดียวกันไหม (ใช้ร่วม → ครอบฟรี) หรือมีฟอร์มแยก (→ flag เป็น Rule-P follow-up; primary scope = per-branch ตามที่ user ระบุ)
  • order-line array RMW: ทำใน transaction (Rule T) ปลอดภัย; order ใหญ่ (หลายร้อย items) = array เขียนทั้งก้อน — ปกติ items ต่อ order ไม่เยอะ
