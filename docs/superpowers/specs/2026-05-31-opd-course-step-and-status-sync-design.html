OPD "คอร์ส" Step + Appointment Status ↔ Tab Real-time Sync

Date: 2026-05-31 · Status: Pending user approval · Surface: Frontend "นัดหมาย วันนี้" card list (+ cross-surface real-time) · Theme: Dark + Light (เข้าตีมเดิม)

Purpose / Problem

Feature 1 — ขั้น "คอร์ส" ในสเต็ปเปอร์ OPD: พนักงานชอบ "ซื้อคอร์สทิ้งไว้แล้วลืมกดตัด". เพิ่มขั้น "คอร์ส" ระหว่าง ② แพทย์ ↔ ③ เสร็จ บน card "นัดหมาย วันนี้" — เป็นสีม่วง "สำเร็จ" เฉพาะเมื่อ OPD นั้นตัดคอร์สจริง (ซื้ออย่างเดียวไม่นับ); ถ้า OPD เสร็จแล้วแต่ไม่ได้ตัด → เตือนเหลือง "ยังไม่ตัด" ให้ผู้จัดการ/พนักงานเห็นทันที.

Feature 2 — กดเสร็จ/แก้สถานะ → ย้าย tab อัตโนมัติ + real-time ทุกที่: ปัจจุบันแท็บ "กำลังรอ/✓ เสร็จแล้ว" ตัดสินด้วย serviceCompletedAt (timestamp) ซึ่งแยกขาดจาก appt.status (ยืนยันแล้ว/เสร็จแล้ว ใน dropdown แก้ไขนัด) → กดเสร็จแล้ว status ไม่เปลี่ยน, แก้ status เป็น "เสร็จแล้ว" ใน modal แล้ว card ก็ยังค้าง tab กำลังรอ. ต้องผูก 2 อย่างให้ sync กัน และเด้ง tab แบบ real-time จากทุก surface (Frontend + Backend) ทุกอุปกรณ์.

Architecture

Feature 1 (display-only, อ่านจาก live data):

  • SSOT helper ใหม่ resolveCourseDeducted(treatment) ใน src/lib/treatmentDisplayResolvers.js = !!(t?.detail?.courseItems?.length) || !!(t?.detail?.treatmentItems?.length) (logic เดียวกับ V136 loadedHasNoCourseUsage แต่บน raw live doc → reuse/SSOT, กัน V12 drift).
  • Stepper opt-in: TreatmentLifecycleStepper รับ prop ใหม่ withCourseStep + courseDeducted. เมื่อ withCourseStep → STEP_KEYS = ['vitalsigns','doctor','course','completed']; ขั้น 'course' มี state done(violet) / warn(amber "ยังไม่ตัด") / pending. ไม่แตะ getTreatmentLifecycle → CDV treatment-history stepper (Phase 28) คงเป็น 3 ขั้นเดิม (scoped; ขยายไป CDV ทีหลังได้ถ้าต้องการ).
  • AppointmentOpdStepperRow ส่ง withCourseStep + courseDeducted={resolveCourseDeducted(latestTreatment)}.
  • Theme: violet + amber ต้องผ่าน WCAG-AA ทั้ง dark + light — ใน light ดันเป็นเฉดเข้ม (เช่น violet-700 / amber-700) ตาม V124/V125. ปลอดภัยกับกฎไทย (ไม่ใช่แดงกับชื่อ/HN; warn เป็นเหลืองไม่ใช่แดง).

Feature 2 (coupling chokepoint, 3 จุดใน src/lib/backendClient.js):

  • markAppointmentServiceCompleted → เพิ่ม status:'done' ลงใน updateDoc.
  • unmarkAppointmentServiceCompleted → เพิ่ม status:'confirmed'.
  • updateBackendAppointment → ใช้ oldData (pre-read ที่มีอยู่แล้ว) ตรวจ transition: done↔non-done แล้ว set/clear serviceCompletedAt (+ serviceCompletedBy/wasServiceCompleted) ให้ตรง. แตะเฉพาะตอน data.status มีค่าและข้ามเส้น done เท่านั้น (ไม่ clobber การ save ที่ไม่เกี่ยว status).
  • ทุก caller (Frontend AppointmentFormModal, Backend appt tab, legacy form, confirm/queue-auto-confirm) inherit อัตโนมัติ. serviceCompletedAt ยังเป็น SSOT ของ tab filter → ไม่แตะ filter, ไม่ต้อง migrate legacy.

Decisions (locked via Q&A with user)

  • Q1 = B: OPD เสร็จแต่ไม่ตัดคอร์ส → ขั้น "คอร์ส" แสดง เตือนเหลือง "ยังไม่ตัด" (ไม่ใช่ขีดเงียบ) — เพื่อจับ "ลืมตัด" ชัดๆ (ยอมรับว่าจะเตือนกับเคสไม่มีคอร์สให้ตัดด้วย).
  • Q2 = A: สีขั้น "คอร์ส" ตอนสำเร็จ = ม่วง violet.
  • Q3 = A: ผูก status ↔ tab สองทางเต็ม (symmetric) — รวมการแก้ status กลับจาก done ใน modal ก็เด้งกลับ "กำลังรอ".
  • Real-time (user เพิ่ม): ตัดคอร์ส / แก้สถานะ จาก surface ไหน (Frontend หรือ Backend) ก็แสดง/เด้ง tab ทันที ทุกอุปกรณ์ — ใช้ onSnapshot listener ที่มีอยู่แล้ว (ไม่เพิ่ม listener ใหม่).
  • Scope = A: เพิ่มขั้น "คอร์ส" เฉพาะ stepper บน card "นัดหมาย วันนี้" (opt-in prop); CDV treatment-history คง 3 ขั้นเดิม.

Acceptance Criteria

  1. Card "นัดหมาย วันนี้" แสดง stepper 4 ขั้น: ซักประวัติ → แพทย์ → คอร์ส → เสร็จ ทั้ง dark + light (AA ทั้งคู่).
  2. OPD ที่ตัดคอร์ส (detail.courseItems/treatmentItems ไม่ว่าง) → ขั้น "คอร์ส" = ม่วง ✓ ; ตัดเสร็จปุ๊บขึ้นทันที (live) เหมือนขั้นอื่น.
  3. OPD เสร็จแต่ไม่ตัด → ขั้น "คอร์ส" = เหลือง "ยังไม่ตัด"; OPD ที่ตัดเฉพาะ "ซื้อ" (purchasedItems) แต่ไม่ตัดใช้ → ยังเตือน (ซื้ออย่างเดียว ≠ ตัด).
  4. กด "✓ ลูกค้ารับบริการเรียบร้อย" → status = "เสร็จแล้ว" + ย้าย tab "✓ เสร็จแล้ว" ทันที; badge เป็น "เสร็จแล้ว".
  5. แก้ status เป็น "เสร็จแล้ว" ใน modal นัด (Frontend หรือ Backend) → card เด้งไป tab "✓ เสร็จแล้ว" ทันที ทุก surface/อุปกรณ์.
  6. กด "↩ กลับไปคิวรอ" หรือแก้ status กลับเป็น "ยืนยันแล้ว" → status = "ยืนยันแล้ว" + เด้งกลับ tab "กำลังรอ" ทันที.
  7. ไม่มี regression: appt ที่ไม่เกี่ยว status/serviceCompletedAt (แก้เวลา/หมอ/ห้อง) ไม่โดน clobber; legacy done-without-serviceCompletedAt ไม่ถูกย้าย tab ย้อนหลัง.

Risks / Open Questions

  • #1 (correctness — verify ก่อน claim done, Rule Q): field path detail.courseItems / detail.treatmentItems ยืนยันจากโค้ด (TFP:1029-1037 + backendDetail write) แล้ว — แต่ต้อง Rule R diag อ่าน be_treatments doc จริงบน prod (เคสตัดคอร์สจริง) ยืนยันว่า predicate ติดสว่างถูกต้อง ก่อน claim (กันบั๊ก V104/V136-class "อ่านผิด field → ขั้นไม่สว่าง/เตือนผิด").
  • multi-treatment/day: stepper ใช้ latestTreatment (OPD ล่าสุดของวันนั้น) — ถ้าตัดคอร์สใน OPD ก่อนหน้าแต่ OPD ล่าสุดไม่ตัด จะเตือน. ตรงกับ spec "ในบันทึก OPD นั้นๆ" (per-record). ยอมรับเป็น v1.
  • warn false-positive: เคสมาปรึกษา/ซื้อสินค้าเฉยๆ (ไม่มีคอร์สให้ตัด) จะขึ้น "ยังไม่ตัด" ด้วย — user เลือก B แล้ว (ยอมรับ). ถ้ารกเกินไปในอนาคต → upgrade เป็น "เตือนเฉพาะลูกค้าที่มีคอร์สเหลือ" (option C เดิม) ได้.
  • optimistic UI: ปุ่มกดเสร็จ optimistic แค่ serviceCompletedAt (ย้าย tab ทันทีสำหรับคนกด); badge status ตามมาจาก listener (~ms). อาจ optimistic status ด้วยได้ (เสริม, ไม่บังคับ).
  • เลขขั้นในธีมสว่าง: warn amber + violet ใน light footer band (red-tint) ต้องผ่าน AA — ใช้เฉด -700 ใน light (V125 aaAccent pattern).
  • No deploy/rules/storage: frontend/lib เท่านั้น → ไม่มี Probe-Deploy-Probe; deploy รอ user สั่ง (V18).
