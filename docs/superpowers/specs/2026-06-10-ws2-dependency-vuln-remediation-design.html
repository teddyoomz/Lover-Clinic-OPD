jsPDF API surface used (ALL 4 sites) — proves 3→4 compat (every call is stable across the major)
  new jsPDF({ orientation, unit, format })        // constructor — stable
  pdf.addImage(imgData, 'JPEG'|'PNG', x,y,w,h, alias?, 'FAST'?)  // stable
  pdf.addPage()                                   // stable
  pdf.internal.pageSize.getWidth()                // stable

  NOT USED: pdf.text() · addFont/addFileToVFS (Thai fonts go through html2canvas → image)
            · AcroForm* · loadFile · setLanguage   ← the CRIT advisory's attack surfaces

The jspdf 4.0 breaking changes (ESM module shape, deprecated text/font internals) touch APIs we never call → low real risk despite being a "major". No jspdf-autotable dep (confirmed) → no plugin compat to manage.

Flow

Upgrade → verify → deploy (with Rule A safe-revert fallback)

edit package.json (remove html2pdf, bump jspdf/vite/postcss)→ npm install (lockfile updates)→ npm run build (clean — vite 8.0.16 + jspdf 4)

→ full vitest (V32 source-grep "no html2pdf import" stays green; no PDF-test regression)→ node API-compat script: exercise EVERY jsPDF call our code makes on v4 (new/addImage/addPage/output/internal) → no-throw

→ L1 (Claude Preview, real browser): run the REAL exportDocumentToPdf engine → valid non-empty PDF blob renders→ npm audit: the 4 targeted advisories cleared→ deploy vercel (build-only; no rules)

Fallback (Rule A): if build OR jsPDF render breaks → revert the jspdf bump to 3.x + document not-reachable (safe — the CRIT is not actually exploitable today). vite/postcss/html2pdf-remove stay (independent, safe).

Architecture

Pure dependency-manifest change — no source logic touched (except possibly fixing the stale "Lazy-imports html2pdf.js" comment in documentPrintEngine.js:477, which already describes removed behavior). Units: package.json + package-lock.json + an optional scripts/diag-ws2-jspdf4-compat.mjs API-exercise. The 4 PDF-generation call sites (documentPrintEngine.js, AppointmentHubView.jsx, ClinicReportTab.jsx, ExpenseReportTab.jsx) are NOT edited — their API usage is already v4-compatible.

Decisions (locked via Q&A with user)

  • Q1 = A: Bump jspdf 3→4.2.1 + FULL verification (build + node API-compat + browser L1 render). Rule A revert to 3.x if rendering breaks (safe — vuln not reachable).

Acceptance Criteria

  1. npm run build clean with vite 8.0.16 + jspdf 4 + postcss 8.5.15.
  2. Full vitest green (V32 "no html2pdf import" guards still pass; no PDF-test regression).
  3. node API-compat script exercises every jsPDF call our code makes on v4 with zero throw + valid PDF bytes.
  4. L1 (real browser) — the actual exportDocumentToPdf engine produces a valid, non-empty, renderable PDF.
  5. npm audit no longer lists html2pdf.js / jspdf / vite / postcss advisories.
  6. No new advisory introduced by the bumps.
