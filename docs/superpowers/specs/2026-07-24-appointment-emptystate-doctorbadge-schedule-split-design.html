Appointment empty-state · doctor-badge nav · doctor/staff schedule split

Date: 2026-07-24 · Status: Approved by user (Q1=A · Q2=A · Q3=A) · Scope: frontend-only, no data migration, no rules change · Plan: TBD (writing-plans next)

Purpose / Problem

Three UX/data fixes on the mobile appointment hub (AppointmentHubView) surfaced by the user's screenshot (วันนี้ · กำลังรอ 0 · เสร็จแล้ว 2):

  1. Empty-state lies. When the day's queue is genuinely finished (0 waiting, 2 completed), the list shows "ลองเปลี่ยน tab หรือ ปรับตัวกรอง" — nagging the user to adjust filters that cannot reveal anything, because the work is simply done. The message must reflect why it's empty.
  2. Doctor badge is a dead end. The 🩺 doctor badge (or "ไม่มีแพทย์เข้า") is pure display. The user wants to tap it → confirm → jump to the editable doctor-schedule page in a new tab; users lacking permission see the destination's own "can't open".
  3. ตารางแพทย์ mixes roles. listDoctors() returns both แพทย์ and ผู้ช่วยแพทย์ (they share be_doctors via a position field), so the doctor-schedule tab shows assistants too. Assistants are staff — they belong in ตารางพนักงาน. Real prod data (Rule R): exactly 2 แพทย์ + 2 ผู้ช่วยแพทย์, clean values.

Architecture

  • ① Empty-state: new pure helper deriveEmptyStateReason() (small module or co-located, exported for tests) + rewrite the empty block in AppointmentHubView.jsx:834-843 to render per reason. CTAs reuse existing handlers (setTodaySubPill, onAddAppointment).
  • ② Badge: AppointmentHubDoctorCards.jsx becomes clickable (role=button + aria-label on both chip-list and empty states) + a small themed ConfirmModal (reuse an existing confirm-modal shape if one fits; else a minimal new one following AV78 — backdrop does NOT close). Navigation via window.open('?backend=1&tab=doctor-schedules','_blank').
  • ③ Split: one shared predicate isDoctorAssistant(person) (Rule of 3 — used by both tabs, prevents drift) in a shared lib (e.g. staffScheduleValidation.js or a small role util). DoctorSchedulesTab filters it OUT; EmployeeSchedulesTab merges it IN (as staffKind='assistant') + extends its schedule staffIdSet + branch filter (assistants use filterDoctorsByBranch).

Decisions (locked via Q&A)

  • Q1 = A: context-aware empty-state — 3 reasons (filtered / all-done / no-appts), each with its own copy + CTA; only the filtered case mentions adjusting the filter.
  • Q2 = A: tap doctor badge → themed confirm modal → open ตารางแพทย์ in a new tab; permission enforced at the destination (no pre-check); both badge states clickable.
  • Q3 = A: display re-partition only, NO data migration; scope = the two schedule tabs; move position==='ผู้ช่วยแพทย์' out of ตารางแพทย์ into ตารางพนักงาน; everyone else stays a doctor.

Acceptance Criteria

  1. Today · กำลังรอ · 0 waiting + ≥1 completed → shows "เสร็จหมดแล้ววันนี้" + a working "ดู N รายการที่เสร็จแล้ว" that switches to the เสร็จแล้ว sub-pill. NEVER "ปรับตัวกรอง".
  2. A tab with genuinely zero appts (no filter) → "ยังไม่มีนัดหมาย" + working "+ เพิ่มนัดหมาย".
  3. With a search/type/status active AND the tab has data → "ไม่พบนัดหมายที่ค้นหา" + "ลองล้างคำค้นหา หรือปรับตัวกรอง".
  4. Tapping the 🩺 chip OR "ไม่มีแพทย์เข้า" opens the themed confirm; confirming opens ?backend=1&tab=doctor-schedules in a new tab; a no-permission account sees the backend's access-denied there; cancel/ESC/backdrop = no navigation.
  5. ตารางแพทย์ lists only แพทย์ (real: 2). ตารางพนักงาน lists be_staff + the 2 ผู้ช่วยแพทย์, managed as staff (no room selection). Existing schedule entries + treatment/DF doctor-pickers unchanged.

Risks / Open Questions

  • Assistant branch filter: the 2 assistants have empty branchId in prod — must confirm filterDoctorsByBranch includes them under the selected branch (else they'd vanish from ตารางพนักงาน). Verify during plan (Rule R).
  • Existing assistant schedule entries with roomIds (set while they lived in ตารางแพทย์) stay as-is (harmless); new/edited entries won't set rooms. No cleanup needed.
  • Predicate strictness: classify by exact position==='ผู้ช่วยแพทย์' (data is clean). A future variant/typo would default to "doctor" (kept in ตารางแพทย์) — acceptable + safe.
  • Verification (Rule Q): pure-helper units + L1 real-browser for all three (empty-state message per state, badge→modal→new-tab + no-permission, schedule split counts). Frontend-only → no rules deploy, no data migration.
