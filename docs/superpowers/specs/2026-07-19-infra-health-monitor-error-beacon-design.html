Infra Health Monitor + Client Error Beacon — Design Spec

Date: 2026-07-19 (EOD+2 evening) · Status: Approved by user ("ลุยต่อเลย") · Constraint (user, verbatim): "ทำหมดเลย แต่ห้ามเพิ่มบั๊ค อย่างเช่นที่ noti หายไป เพราะนายไปทำอะไรสักอย่าง" · Implementation plan: 2026-07-19-infra-health-monitor-error-beacon.html

Purpose / Problem

สองเดือนที่ผ่านมาระบบเบื้องหลัง ตายเงียบ 3 ครั้ง: push outage เงียบ 12 วัน (AV210) · whole-system backup NO_MANIFEST เงียบ 5 วัน (V122) · chat-history retention cron ตาย 46 รอบรัน (type mismatch) ทุกครั้ง user เป็นคนเจอเอง ปัจจุบันมี 13 crons + push + backup ที่ fail ได้โดยไม่มีใครเห็น และ error ฝั่งเครื่อง user (จอดำ V163-class ฯลฯ) มองไม่เห็นเลยจนกว่า user จะรายงาน Spec นี้ทำให้ "ความพังมองเห็นได้" ภายใน 24 ชั่วโมง (infra) / ภายในนาที (client error) โดย ไม่แตะ CSP / SW / push path / firestore.rules เลย (blast-radius = ศูนย์ต่อระบบเดิม — บทเรียน AV210)

Architecture

หลักการ: ต่อรางเดิมทั้งหมด ไม่สร้างรางใหม่ (แนวทาง 1 ที่ user เลือก)

  • Pure SSOT — src/lib/infraHealthCore.js: ตาราง EXPECTED_MAX_AGE_HOURS ต่อ cron (ครบ 13 + รายการ UNMONITORED ที่ประกาศชัด) + evaluateInfraHealth() (pure, รับ doc snapshots ที่อ่านมาแล้ว) + buildInfraAlertText() (ไทย) + buildInfraChatCardDoc(). Cron / UI / test ใช้ตัวเดียวกัน — ไม่มี drift
  • Cron — api/cron/infra-health-sweep.js: copy skeleton จาก money-reconciliation-sweep.js + auth แบบ bearer-or-x-cron-secret + writeScheduledTaskStatus + ลงทะเบียนใน scheduledTasksRegistry.js + run-scheduled-task.js (ปุ่ม "ตรวจตอนนี้" ได้ฟรีจาก ScheduledTasksTab)
  • Alert ช่อง 1: staff chat — เขียนผ่าน admin SDK (bypass rules — pattern เดียวกับ intake/followup ใน functions/staffChatNotify.js) · kind ใหม่ 'infra-health' · StaffChatSystemCard.jsx เพิ่ม HEADLINE['infra-health'] + early branch แบบ plain-text (ไม่มี customer row) — จำเป็น เพราะ kind ที่ไม่รู้จักจะ fallback เป็น intake card เพี้ยน
  • Alert ช่อง 2: LINE OA — reuse pushLineMessage (รับ {type:'text',text} ตรงๆ ไม่ผูก flex) + getLineConfigForBranch · target = {branchId, lineUserId}[] (ต้องเป็น OA ของสาขาที่ผู้รับแอดเป็นเพื่อน — ข้อจำกัด per-branch OA, ไม่มี global token) · เช็ค statusCode===200 เอง; 410 = unfollowed → จดผลใน run doc + โชว์ในการ์ด
  • Config — key ใหม่ infraHealth ใน clinic_settings/system_config (ตาม precedent v86Glow/scheduledTasks): {enabled, lineTargets:[], staffChatBranchId, errorThreshold24h} — ได้ validated+audited write + live listener ฟรี · run-state แยก ไปอยู่ be_admin_audit/infra-health-latest (doc ตายตัว อ่านด้วย getAdminAuditDoc แบบ recon-banner — be_admin_audit อ่านได้ด้วย isClinicStaff อยู่แล้ว ไม่ต้องแก้ rules)
  • Beacon client — src/lib/errorBeacon.js (helpers เป็น pure exports เพื่อเทส) ติดตั้งบรรทัดแรกๆ ของ main.jsx + AppErrorBoundary.jsx ครอบ <App/> (จอดำ → หน้า "โหลดหน้าใหม่")
  • Beacon server — api/client-error.js (anon POST — mirror api/patient-view.js: CORS, cached getDb, validate, generic 500) + api/admin/client-errors-list.js (verifyAdminToken) · collection client_error_log default-deny (ไม่มี rule ใหม่ = client SDK แตะไม่ได้ = ปลอดภัยสุด) · Rule C3 ครบ: writer=endpoint, reader=admin endpoint, ขนาดถูก cap 500/วัน + retention 30 วัน
  • UI — src/components/backend/InfraHealthSection.jsx (ไฟล์แยก กัน SystemSettingsTab บวม) ประกอบใน SystemSettingsTab: status rows + config form (saveSystemConfig) + LINE-target picker (list จาก chat_conversations ล่าสุด — staff อ่านได้อยู่แล้ว) + error viewer + ปุ่มทดสอบแจ้งเตือน (api/admin/infra-health-test-alert.js)

Decisions (locked via Q&A with user)

  • Q1 = Staff chat + LINE OA: ช่องแจ้งเตือนคู่ — อิสระจาก FCM ทั้งสองช่อง (FCM แจ้ง "FCM ตาย" ไม่ได้)
  • Q2 = ทุกหน้า: beacon ครอบ staff app + หน้าลิ้งลูกค้าทั้งหมด (PatientForm / ดูนัด / ดูข้อมูล / ED scoring)
  • Approach = 1 "ต่อรางเดิม": ปฏิเสธ tab ใหม่+listener (ต้องแก้ rules) และ alerts-only (เสีย history)
  • ไม่แก้ firestore.rules เลย: error log อ่าน/เขียนผ่าน endpoint (admin SDK) — collection default-deny
  • แจ้งเตือนวันละ ≤1 ครั้ง/ปัญหา (doc id ต่อวัน) — ไม่มี recovery-spam, สถานะดูได้ในการ์ดตลอด

Acceptance Criteria

  1. จำลองเหตุการณ์จริง 3 แบบ (backup stale >36h / push token สด 0 / cron ok=false) ผ่าน fixture → evaluateInfraHealth ให้ RED/WARN ถูกต้อง + alertText ไทยครบทุกปัญหา
  2. กด "ทดสอบแจ้งเตือน" → การ์ดขึ้น staff chat จริง + LINE เด้งเข้ามือถือ user จริง (Rule Q L1/L3)
  3. โยน error ใน browser จริง (dev) → ผ่าน sanitizer → ค่า query param ถูก strip → โผล่ใน viewer (L1 เท่าที่ทำได้ pre-deploy; full round-trip = post-deploy)
  4. cron ใหม่ที่เพิ่มใน vercel.json โดยไม่ประกาศฝั่ง health → classifier test แดง
  5. AV210 bank green + git diff ยืนยัน 0 การเปลี่ยนแปลงใน headers/CSP/SW/rules/push
  6. เขียน error เกิน cap → doc ที่ 501 ของวันถูก drop เงียบ (cost bounded)

Risks / Open Questions

  • LINE target ยังว่างตอน ship — จนกว่า user จะเลือก lineUserId ในการ์ด → LINE channel เงียบ (staff-chat card ยังทำงาน) การ์ดจะโชว์คำเตือน "ยังไม่ได้ตั้ง LINE target"
  • ผู้รับต้องเป็นเพื่อนกับ OA ของสาขานั้น (ข้อจำกัด per-branch token) — 410 จะถูกจดและโชว์ในการ์ด
  • Beacon endpoint บน prod จะมีจริงหลัง deploy เท่านั้น — pre-deploy L1 ทำได้แค่ฝั่ง capture+payload; full round-trip verify post-deploy (จดเป็น honest gap)
  • Anon endpoint = spam vector — ไม่มี shared rate-limiter ใน repo; ป้องกันด้วย size cap + field truncation + daily cap 500 (โกงสุด = 500 junk doc/วัน, bounded, ยอมรับ)
  • ErrorBoundary เปลี่ยน UX ตอน crash (จอดำ → หน้า reload) — ตั้งใจ และโผล่เฉพาะกรณีที่เดิมคือจอดำอยู่แล้ว
