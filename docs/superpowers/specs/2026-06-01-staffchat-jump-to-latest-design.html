Staff Chat — Jump-to-Latest Button Design Spec

Date: 2026-06-01 · Status: Approved by user · Implementation plan: docs/superpowers/plans/2026-06-01-staffchat-jump-to-latest.html (next)

Purpose / Problem

In the staff-chat panel, once you scroll up to read older messages there is no fast way back to the latest message — you must scroll manually, and if new messages arrive while you're scrolled up there is no signal. User ask (verbatim): "ทำให้ staff chat box มีปุ่มเล็กๆเพื่อกดลงไปแชทล่าสุด เหมือนแอปแชทดังๆที่เขามีกัน". Add the familiar floating "jump to latest" affordance (Messenger / LINE / Slack), with an unread-count badge for new messages below the fold. Purely additive — the existing send / receive / day-grouping / read-cursor (V82) flow is untouched.

Architecture

Two source files. No new module, no new observer, no Firestore/rules/storage/cron.

FileChange
src/components/staffchat/StaffChatMessageList.jsx(a) Wrap the existing scroll <div> in a relative flex-1 min-h-0 parent so the button floats over the list (the scroll div, endRef, bottomSentinelRef stay verbatim inside, made h-full overflow-y-auto). (b) Add isAtBottom state, set inside the EXISTING observer callback via setIsAtBottom(entry.isIntersecting) — onScrolledToBottom() still fires only on intersect (V82 unchanged); relax the effect guard so the observer runs for the boolean even if no callback passed. (c) New prop unreadCount = 0. (d) Render the floating jump button (circle + ChevronDown + conditional rose badge) when !isAtBottom; onClick scrolls endRef to bottom. data-testid="staff-chat-jump-latest" + staff-chat-jump-latest-count.
src/components/staffchat/StaffChatWidget.jsxOne line: pass unreadCount={chat.unreadCount} to <StaffChatMessageList>.

Component boundary: the jump button is internal to StaffChatMessageList (a small inline element, not a new exported component — it has no reuse elsewhere). It depends only on local isAtBottom + the unreadCount prop + endRef. Consumers (Widget) are unaffected beyond the one new optional prop (defaults to 0 → backward-compatible).

Decisions (locked via Q&A with user)

  • Q1 = C: Circle icon + unread-count badge (Messenger style). Superset of a bare circle — degrades to plain circle when unreadCount === 0.
  • Q2 = A: Button appears whenever scrolled up from bottom (regardless of new messages); hidden at bottom. Matches Messenger/LINE/Slack. (Rejected B = only-when-new — would strand the user with no jump-back after reading history.)
  • Reuse the EXISTING bottomSentinelRef IntersectionObserver (piggyback the boolean) rather than adding a second observer/scroll-listener → zero change to V82 read-cursor timing + minimal code.
  • Count source = chat.unreadCount (already computed by V82 cursor) — no new counting logic.
  • Button is neutral (--bg-hover + --bd-strong) so it doesn't compete with the rose send button / rose header; only the count badge is rose. Auto dark+light via tokens.

Acceptance Criteria

  1. When the list is scrolled to the bottom, no jump button is rendered (staff-chat-jump-latest absent).
  2. When scrolled up (bottom sentinel not intersecting), the jump button renders.
  3. When unreadCount > 0 AND scrolled up, the rose count badge renders with the count; caps display at "9+" for counts > 9. When unreadCount === 0, no badge (bare circle).
  4. Clicking the jump button calls endRef.scrollIntoView({behavior:'smooth', block:'end'}) (smooth scroll to latest).
  5. After reaching the bottom, the existing onScrolledToBottom() still fires (V82 read-cursor advances) and the button hides — verified unchanged.
  6. StaffChatWidget passes chat.unreadCount to StaffChatMessageList.
  7. Button is keyboard-focusable with aria-label="ลงไปข้อความล่าสุด"; theme tokens render correctly in dark AND light.
  8. No regression: full vitest suite green; build clean; existing staff-chat tests (message list, read cursor, panel) unchanged-or-green.

Risks / Open Questions

  • Edge flicker at the very bottom: the sentinel is 1px @ threshold 0.5, so the button toggles right at the bottom edge. Acceptable (standard behavior); if it ever feels jumpy, a future tweak could add a small rootMargin — but that would shift V82 read timing, so deliberately NOT done now.
  • jsdom has no IntersectionObserver: RTL tests mock it (as existing MessageList tests already do) and drive isIntersecting true/false to exercise show/hide.
  • Honest Rule Q gap: assembled real-browser L1 inside the admin-gated AdminDashboard panel (live unreadCount + real smooth-scroll + reset) = USER after deploy. Pre-deploy L1 = sandbox-mounted real StaffChatMessageList in Chrome (real scroll container + real IntersectionObserver), like the EOD+6 resizable-panel L1.
  • Mobile: same MessageList drives the fullscreen mobile overlay → button works there too; verified via the shared component (no separate mobile path).
