Instant Staff App — Cold-Start <1s Design Spec

Date: 2026-07-07 EOD+2 · Status: Design approved by user ("ok") · Implementation plan: 2026-07-07-instant-staff-app-cold-start.html

Purpose / Problem

User report (screen recording, iPhone PWA บน 5G): เปิดแอปครั้งแรกหลังเว้นไปนาน → จอดำ ~2 วิ → shell ของหน้า นัด (AppointmentHubView) ขึ้นแต่ทุก count = 0 + "กำลังโหลด…" ค้าง ~7-10+ วิ กว่าข้อมูลจะมา (บางครั้งนานกว่านั้นมาก). Root cause ที่พิสูจน์จากโค้ด:

  • firebase.js:27 — NO offline persistence (การตัดสินใจ Q1 fresh-always จาก Mobile-Load Reliability 2026-06-16) → ทุก cold start คือ network round-trip ล้วนๆ บน connection ที่ต้อง handshake + auth-token refresh ใหม่
  • AppointmentHubView.jsx:174 — loadAll รอ 7 dataset ใน Promise.all + 1 sequential (รวม getAllCustomers() ทั้ง collection + getAllSales()) ก่อน paint อะไรเลย ทั้งที่ list นัด + counts ใช้แค่ 2 dataset — อีก 5 ก้อนใช้ทำ chip เสริมบนการ์ดเท่านั้น
  • ทุก getter เป็น one-shot getDocs = server-first เสมอ แม้เปิด cache ก็ไม่ auto-ได้ประโยชน์ (ต้องเปลี่ยน read strategy คู่กัน)

Goal: เปิดแอปซ้ำ (เว้นนานแค่ไหนก็ตาม ตราบที่ cache ยังอยู่) → เห็นหน้า นัด พร้อมข้อมูลล่าสุดที่เคยเห็น < 1 วิ → ระบบ sync สดตามมาแก้ให้เองใน ≤ 3 วิ (เน็ตปกติ) พร้อม indicator "กำลังซิงค์…" · เครื่องใหม่/ไม่มี cache → เร็วขึ้นจาก staged load + Service Worker · หน้าลูกค้าคง fresh-always 100% เหมือนเดิม.

Decisions (locked via Q&A with user, 2026-07-07)

  • Q1 = A: เปิด persistentLocalCache ทั้งแอป → staff surfaces = stale-while-revalidate (paint จาก cache ทันที + sync สดตาม + indicator); customer-facing = fresh-gate รอ server ก่อน render เหมือนเดิม. REVERSES Q1 fresh-always (2026-06-16) เฉพาะฝั่ง staff — เหตุผลเดิม ("ลูกค้าห้ามเห็นยอดคอร์ส/เวลานัดเก่า") ยังถูก honor ด้วย fresh-gate.
  • Q2 = A: Hub แยก 2 จังหวะ — นัด+counts+แพทย์ paint ก่อน; chip เสริม (กระเป๋าเงิน/มัดจำ/membership) โหลดตามแบบ skeleton.
  • Q3 = B: ขอบเขต = ทั้ง staff app (AdminDashboard + BackendDashboard ทุก tab) — listener tabs ได้ฟรีจากชั้น 0; one-shot tabs sweep ด้วย useSwrLoad + classifier.
  • Q4 = B: Service Worker precache app shell เอาด้วย — พร้อม update flow + kill-switch.

Acceptance Criteria

  1. Repeat cold-start (มี cache): เปิดแอป → หน้า นัด แสดงรายการนัด + counts (ข้อมูล cache) ภายใน < 1 วิ หลัง shell ขึ้น (Playwright วัดจริง, CDP network throttle Fast 3G ก็ยังผ่าน)
  2. Server sync: เน็ตปกติ — ข้อมูลสดแทนที่ cache ≤ 3 วิ; indicator "กำลังซิงค์…" แสดงระหว่าง fromCache และหายเมื่อ server ยืนยัน; ข้อมูลที่เปลี่ยนบน server ระหว่างที่แอปปิด ปรากฏถูกต้องหลัง sync
  3. First-ever (ไม่มี cache): จังหวะ 1 paint โดยไม่รอ 5 dataset เสริม (เร็วขึ้นกว่า baseline ที่วัดไว้); chip skeleton → เติมเมื่อจังหวะ 2 มา; ไม่มี layout shift
  4. Customer fresh-always: ?patient= / ?session= / ?schedule= / ED — ไม่ render ข้อมูล fromCache เลย (Playwright: seed cache → เปลี่ยนข้อมูลบน server → เปิดหน้าลูกค้า → เห็นค่าใหม่เท่านั้น ไม่มี flash ค่าเก่า)
  5. Write semantics ไม่เปลี่ยน: การขาย/ตัดคอร์ส/ตัดสต็อคยัง await server ack; ปิดเน็ตแล้วกดบันทึก → พฤติกรรม error/รอ เหมือนเดิม (ไม่ silent-queue เงิน)
  6. SW: เปิดแอปแบบ offline/เน็ตช้ามาก → shell ขึ้นจาก precache; deploy ใหม่ → client ได้เวอร์ชันใหม่ภายใน 1 รอบ visibilitychange + ไม่มี white-screen จาก chunk เก่าหาย; /api/* + Firestore ไม่ถูก SW แตะ (network log พิสูจน์)
  7. Regression: full vitest 0 fail · build clean · pixel-parity ทุก surface (นอกจาก indicator/skeleton ใหม่) · V17 reconnect + useResilientLoad + Mobile-Load L1 specs ยังเขียว · multi-tab (2 แท็บพร้อมกัน) ไม่พัง persistence

Risks / Open Questions

  • Cache query ไม่ครบ: counts จาก cache อาจต่ำกว่าจริงชั่วครู่แล้วเด้งขึ้น — mitigated ด้วย indicator; ห้าม decision-write จาก fromCache (audit invariant ใหม่ AV206)
  • iOS eviction: ไม่ใช้แอปนานมาก → IndexedDB อาจโดนล้าง → degrade เป็น first-ever path (ไม่แย่กว่าปัจจุบัน); storage.persist() ลดโอกาส
  • persistentLocalCache perf: ecosystem เคยมีรายงานช้าใน SDK เก่า (2023) — เราอยู่ firebase 12.11; Phase E วัดจริง ถ้า cache leg ช้าผิดปกติ → escalate
  • SW stale-deploy: update flow + no-cache sw.js + kill-switch (ตาม §5); Playwright update-flow spec บังคับ
  • Firestore listener โหลดถี่: persistence ไม่เพิ่ม read cost (cache hits ไม่คิดเงิน; server sync = delta) — ค่า Firestore ไม่ขึ้น
  • Classifier ครอบไม่หมด: tab ใหม่ในอนาคตลืม SWR → classifier test บังคับ SWR-adopted-or-sanctioned (แบบ AV205)
