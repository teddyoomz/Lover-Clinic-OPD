Light-Theme Audit & Fix — Design Spec

Date: 2026-05-27 (EOD+12) · Status: Approved by user (design "ok") · Scope: CSS/colour theming only — zero wiring/flow/logic (cosmetic shell) · Implementation plan: TBD (next: writing-plans)

Purpose / Problem

The project was built dark-first for its whole life. With launch imminent, the light theme has accumulated whack-a-mole defects: any page or modal still rendering a hardcoded dark colour, or white text on a light surface, looks broken or is outright invisible in light mode (history: V107, V83-followup-11, the EOD+12 chart-button fix). The user wants a complete sweep of every page and every modal (App UI) in light theme, fixing everything to be clearly legible — touching only design CSS/colour, never wiring/flow/logic.

How the theme system works (recap):
  • data-theme on <html>: dark (default) · light · auto (= system pref).
  • Palette is CSS custom properties (--bg-card, --tx-heading…). But components mostly use hardcoded Tailwind arbitrary classes (bg-[#0f0f0f], text-white, text-gray-300).
  • src/index.css (~4005 lines) carries a giant [data-theme="light"],[data-theme="auto"] override layer that remaps each hardcoded dark class → its light value.
  • Failure rule: any hardcoded dark class / white text NOT covered by an override stays dark/invisible on the light surface.

Flow

Per-batch audit→fix→verify pipeline (repeated for B1–B5):

1 · GREPsweep src/ โซน batch หา FM-A..D candidates
→
2 · CLASSIFYเทียบกับ override ใน index.css → FM-A/B/C (central) หรือ FM-D (component)
→
3 · FIXcentral: เพิ่ม [data-theme=light],[data-theme=auto] rule · FM-D: สลับค่าสี inline → CSS-var
→
4 · LIVE-VERIFYเปิด surface ใน light theme (Chrome MCP/Preview) · screenshot · วัด AA
→
5 · GUARD/audit-ui-cultural-a11y + build + vitest ท้าย batch · dark theme regress = 0

Constraint gate (every fix): diff ต้องเป็น CSS / className / inline-color-value เท่านั้น — ถ้าจะแตะ handler/state/prop/hook/flow = STOP (ผิดกฎ cosmetic-shell).

Architecture

Fix mechanism — hybrid central-first

  • FM-A / FM-B / FM-C → central: append new [data-theme="light"],[data-theme="auto"] rules into the matching labelled section of src/index.css. Always pair light + auto. 0 .jsx edits.
  • FM-D → component (only when central can't reach): inline style={{…}} / colours computed in JS. Swap the colour value to a theme-aware CSS var (e.g. '#fff' → 'var(--tx-heading)'). Value only — never the surrounding logic.
  • Selector discipline: prefer narrow selectors; the file already shows !important specificity scars (V107 had to narrow a catch-all). A new broad rule must not flip an intended white-on-accent CTA to dark, or vice-versa.
  • Dark theme untouched: every change is light/auto-scoped; dark rendering must be byte-identical before/after.

Discovery (audit method)

  • Static grep sweep per zone → candidate list, cross-checked vs existing index.css coverage.
  • /audit-ui-cultural-a11y (WCAG AA contrast + Thai-culture: no red on names/HN) + /audit (technical) as automated backstops.
  • Live-browser light-theme pass (Chrome MCP / Claude Preview, per feedback_use_chrome_mcp_first + Rule S): open every page + modal, screenshot, eyeball + measure computed-style contrast → confirm AA (Rule Q-vis: real pixels, not grep alone).

Decisions (locked via Q&A with user)

  • Q1 = WCAG AA (4.5:1): "ชัด" = AA — body text ≥ 4.5:1, large text/icons ≥ 3:1, every interactive element clearly visible. Measurable; backed by /audit-ui-cultural-a11y.
  • Q2 = Hybrid (central-first): extend the central index.css override layer first (0 .jsx edits); component-level only for FM-D (inline / JS colours).
  • Q3 = App UI, exclude print: every page + modal of admin + backend + patient-facing; print/document views excluded (always-white paper).
  • Q4 = Full live-browser pass: grep candidates → open every page + modal in light theme via Chrome MCP/Preview → real-pixel verify + AA-measure. Slowest, most thorough — launch-grade.
  • Hard constraint: CSS/colour only — zero wiring/flow/logic (cosmetic shell, per feedback_cosmetic_shell_redesign_constraint).

Acceptance Criteria

  1. Every App-UI page + modal (admin + backend + patient-facing; print excluded), opened in light theme, has no invisible/illegible text, icon, button, or border, and all text meets AA (4.5:1 body / 3:1 large+icons).
  2. No leftover dark surface (dark box/panel) on a light background.
  3. Dark theme is byte-identical before/after (zero regression — verified by before/after dark screenshots on touched surfaces).
  4. Both light AND auto (=system light) modes render identically.
  5. Diff contains only CSS rules / className strings / inline colour-value swaps — no handler/state/prop/hook/flow/logic change (grep-verifiable).
  6. No red on patient names/HN in light theme (Thai culture, rule 04).
  7. npm run build clean + full vitest (14971+/0) green (FM-D className/inline swaps must not break source-grep/RTL tests, or those tests are corrected to match).
  8. Evidence: light-theme screenshots of every fixed surface + /audit-ui-cultural-a11y pass.

Risks / Open Questions

  • index.css specificity wars (V107 scar): a new broad override may flip an intended white-on-accent CTA to dark (or the reverse). Mitigation: narrow selectors + live-verify after every batch.
  • FM-D test fragility: component inline/className swaps can trip source-grep/RTL tests asserting exact classes → run affected tests + full suite at batch end (Rule N exception: this is a broad cross-cutting change).
  • Hard-to-reach modals: some deep modals need specific data/state to open. Where I can't drive the state, use a TEST fixture or fall back to user L1 hands-on for those few (documented per batch).
  • "auto" mode verification needs OS light-pref OR forcing prefers-color-scheme: light in devtools during the live pass.
  • Open (confirm in plan): add a Rule D regression artifact — an override-coverage test that flags dark classes used in components but lacking a light override, to prevent future drift? (Recommended.)
  • No deploy implied: all work is local + commits await explicit "push"/"deploy" (V18). This ships alongside the existing unpushed EOD+11/EOD+12 frontend stack.
