Stock-Movement Retention (V106) — Design Spec

Date: 2026-05-20 (EOD+4) · Status: Approved by user · Implementation plan: docs/superpowers/plans/2026-05-20-stock-movement-retention.html (pending writing-plans)

Purpose / Problem

be_stock_movements grows monotonically: every sale, treatment, adjustment, transfer, withdrawal and negative-repay writes one append-only doc (writers in src/lib/backendClient.js: _deductOneItem, createStockAdjustment, _repayNegativeBalances, updateStockTransferStatus, updateStockWithdrawalStatus). They are never deleted, so Firestore doc-count + read-cost climbs forever.

Re-grounding finding that reshaped the original brainstorm: stock balance is be_stock_batches-authoritative — StockBalancePanel sums batch.qty.remaining directly and never replays movements. So pruning movements cannot break the balance, and a "balance snapshot" step is YAGNI (the batches already are the snapshot). The original Q1 "hard delete + balance snapshot" therefore drops its snapshot half.

The remaining constraint is the audit/legal record: firestore.rules blocks client deletes (allow delete: if false) and V34 locked "movements are the legal record" (MOPH audit-trail). So old movements must be preserved permanently in a cheaper store before deletion, not hard-deleted outright.

Flow

Daily cron cascade. Each run is bounded (≤ 2000 movements) so the launch backlog drains incrementally over consecutive days without hitting Vercel maxDuration.

Cron run cascade — archive-before-delete (AV19-class capture-before-destroy)
⏰ cron 03:30 BKK
30 20 * * * → auth: CRON_SECRET Bearer
(else 401, no data touched) → admin-SDK · canonical path
artifacts/{APP_ID}/public/data/be_stock_movements
↳ query where createdAt < cutoffISO
orderBy createdAt asc · limit 2000 → in-memory re-gate
normalizeCreatedAtForCompare(d) < cutoff
guards mixed Timestamp/ISO → group by (branchId, YYYY-MM)
↳ per group: read Storage file
→ mergeArchive (dedup by movementId)
→ write Storage file → GATE: all Storage writes confirmed? → delete fetched docs
writeBatch chunks of 450
↳ write audit doc
be_admin_audit/stock-movement-retention-* → respond {scanned, archived, deleted, moreRemaining} ⟳ next day: repeat until backlog drained

Invariant: a movement is never deleted unless its archive file write was confirmed first. If any group's Storage write fails → that group's docs are not deleted this run; logged; re-attempted next run. be_stock_batches (balance source) is never read or written by this flow.

Stage table
#StageFailure handling
1Cron fires 03:30 BKK; verify CRON_SECRET401 → exit, zero data touched

Architecture

1. Pure helper — src/lib/stockMovementRetentionCore.js

No Firebase imports → fully unit-testable. Single source of truth for retention constants + transforms.

  • RETENTION_DAYS = 90 · RETENTION_BATCH_LIMIT = 2000
  • computeCutoffISO(now = new Date()) → (now − 90d).toISOString()
  • archiveStoragePath(branchId, yyyyMM) → stock-movements-archive/{branchId}/{YYYY-MM}.json
  • monthKeyFromISO(createdAtISO) → "YYYY-MM" · groupByBranchMonth(movements) → { "branchId|YYYY-MM": [...] }
  • mergeArchive(existing[], incoming[]) → union deduped by movementId (idempotent re-runs)
  • normalizeCreatedAtForCompare(value) → ISO string from ISO | Firestore Timestamp | {_seconds,_nanoseconds} | {seconds,nanoseconds} (reuses V105 _v105NormalizeCreatedAt shape)
  • buildArchiveFileBody({branchId, month, movements}) → {schemaVersion:1, branchId, month, archivedAt, count, movements}

2. Cron — api/cron/stock-movement-retention.js

Mirrors api/cron/whole-system-backup-daily.js (auth + admin-SDK init + PEM parse). Uses canonical artifacts/{APP_ID}/public/data/be_stock_movements path (Rule M). Per-run cap = RETENTION_BATCH_LIMIT. maxDuration: 300. Archive-before-delete ordering enforced.

4. storage.rules

Add admin-only match /stock-movements-archive/{branchId}/{file=**} (mirror backups/). No firestore.rules change (be_stock_movements delete already if false; admin SDK bypasses).

5. MovementLogPanel.jsx

Query unchanged. Add one static info line "แสดงย้อนหลัง 90 วัน — ข้อมูลเก่ากว่านั้นถูกเก็บถาวรใน archive". Informational only — does not violate cron-only (Q5).

Decisions (locked via Q&A with user)

  • Q1 = Archive→Storage then hard-delete (V81/AV64 pattern). Balance snapshot dropped — YAGNI (batches authoritative).
  • Q2 = 90-day live retention window. Older → archive only.
  • Q3 = Daily cron, monthly-file archive — 30 20 * * * (03:30 BKK), append-merge per (branch, month).
  • Q4 = All movement types uniformly (archive preserves everything, incl. adjustments → no audit loss).
  • Q5 = Cron-only — no CLI, no admin UI button. ⇒ cron must self-drain the launch backlog incrementally (≤2000/run).
  • Sub-1 = include MovementLogPanel static info line.
  • Sub-2 = include explicit storage.rules admin-only match for the archive path.

Acceptance Criteria

  1. Movements whose normalized createdAt < now−90d are written to stock-movements-archive/{branchId}/{YYYY-MM}.json then deleted from be_stock_movements.
  2. Archive file content = dedup-by-movementId union; running the cron twice yields 0 additional deletions (idempotent).
  3. No movement is deleted unless its archive write is confirmed first (archive-before-delete).
  4. Each run processes ≤ 2000 movements; a large backlog drains over consecutive daily runs without timeout.
  5. Stock balance (be_stock_batches.qty.remaining via StockBalancePanel) is byte-identical before vs after retention.
  6. A movement with a legacy Timestamp createdAt is age-gated correctly by normalized ISO compare (never wrongly deleted by Firestore type-ordering).
  7. Request without valid CRON_SECRET → 401, zero data touched.
  8. Each run writes an audit doc with {scanned, archived, deleted, cutoffISO, monthsTouched, archiveRefs, ranAt, moreRemaining}.
  9. Anon/client cannot read or write stock-movements-archive/ (Storage 403).
  10. MovementLogPanel renders the 90-day info line.

Testing (Rule Q V66)

  • Pure-helper units — cutoff math, grouping, mergeArchive dedup/idempotency, path builder, normalizeCreatedAtForCompare across ISO/Timestamp/_seconds shapes, batch-limit. Adversarial: empty, mixed createdAt types, NFC≠NFD, NUL byte, 10k-movement perf budget, concurrent.
  • Rule I flow-simulate — fixture run → archive files produced → docs deleted → re-run = 0 extra → balance/batches unchanged → backlog drains over N simulated runs.
  • NEW AV99 invariant — deletion MUST be preceded by a confirmed Storage archive (archive-before-delete); the cron is the ONLY deleter of be_stock_movements; age compared via normalized ISO. Source-grep guard + closed-deleter list. (AV98 is the current max.)
  • Rule Q L2 — admin-SDK e2e on real prod: seed TEST--prefixed old-dated movements (V33.11 stock-test prefix) → run retention logic → assert archived to Storage + deleted from Firestore + batches/balance untouched → cleanup zero orphans + audit doc. Live cron itself = L3 (curl with CRON_SECRET post-deploy, or user observes the scheduled run).

Deploy

Combined vercel --prod (cron + endpoint + vercel.json) + firebase deploy --only storage:rules with Probe-Deploy-Probe (new probe: anon write to stock-movements-archive/ → expect 403). No firestore.rules change. No data ops at deploy — the cron performs the first backlog drain over the days following deploy. User authorizes deploy explicitly per V18.

Risks / Open Questions

  • Vercel cron count — this is the 4th cron (3 already run: line-reminder ×2 + whole-system-backup-daily). Plan already supports ≥3 schedules; the 4th is within limits. Confirm at deploy.
  • Mixed createdAt types in legacy data — mitigated by the in-memory re-gate. If many legacy Timestamp docs exist, the Firestore range query (string compare) may over-fetch them, but the re-gate prevents wrong deletion; over-fetched docs simply consume part of the 2000-cap that run. Acceptable.
  • maxDuration — if 2000 docs × Storage round-trips exceed 300s, lower RETENTION_BATCH_LIMIT (single constant, tunable).
  • Archive Storage growth — permanent by design (it IS the legal record). JSON is far cheaper than Firestore docs; no archive-of-archive retention.
  • First-run backlog size — everything >90d since the stock system shipped; incremental drain handles it but full clearance may take several daily runs. Acceptable; moreRemaining in the audit doc tracks progress.
