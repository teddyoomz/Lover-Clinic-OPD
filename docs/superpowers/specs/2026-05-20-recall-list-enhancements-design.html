Recall List Enhancements — Design Spec

Date: 2026-05-20 (EOD+2) · Status: Approved by user (decisions Q1–Q3 locked) · Implementation plan: docs/superpowers/plans/2026-05-20-recall-list-enhancements.html (to be written by writing-plans)

Purpose / Problem

The Recall (patient follow-up) feature is shown on three surfaces — the Frontend "Recall" tab, the Backend Recall tab, and the customer-detail page — but the cards omit the customer's phone number and bury the free-text note. Staff calling patients have to open each customer to find the phone, and the reason/result of a call (which staff always fill in) is rendered small. There is also no record of which staff member logged a result, and the Frontend tab only surfaces today's recalls, so yesterday's missed follow-ups silently fall off the radar.

This spec makes every recall card show a tap-to-call phone number and a large, prominent note, captures and displays the staff member who recorded the result, and reworks the Frontend tab into "Recall วันนี้" with Today (top, most prominent) + all overdue + tomorrow — beautiful in both dark and light themes.

Architecture

Approach: single-source. All three surfaces (backend tab, frontend view, customer-detail) render through the shared src/components/backend/recall/RecallRow.jsx. Phone + prominent-note + logged-by land in that one file and propagate everywhere. No new card component, no per-surface duplication.

Decisions (locked via Q&A with user)

  • Q1 = A: Prominent note = outcomeNote when present, else fall back to reason (pending recalls show the recall reason; contacted recalls show the contact result). Label adapts: "📝 ผลการติดต่อ" vs "📝 เหตุผลนัด recall".
  • Q2 = B: Staff dropdown starts blank and is REQUIRED every time — Save is disabled until a staff member is picked. Stored in outcomeBy {id,displayName} and shown as the "บันทึกโดย" byline.
  • Q3 = A: Frontend past section = all overdue/past-due pending ("ค้าง/เลยกำหนด"), not just literally yesterday — so older forgotten recalls surface too. Today stays on top + most prominent; tomorrow below.
  • Theme: both dark + light, on-theme, via existing CSS vars. No red on customer name / HN (Thai-culture rule); phone uses the app's call-action accent.
  • YAGNI / out of scope: no staff picker on recall create (outcome only); no LINE-flow change; no new collection or schema field (reuse customerPhone + outcomeBy).

Acceptance Criteria

  1. Every recall card on all 3 surfaces shows the customer phone as a tel: link that initiates a call; cards with no phone degrade gracefully (no broken link).
  2. The note block is visually prominent (larger font + tinted box) and shows outcomeNote || reason; contacted recalls show outcomeNote, pending show reason.
  3. Recording an outcome is blocked until a staff member is selected; the selected staff is persisted to outcomeBy and rendered as "บันทึกโดย: {name}" on the card.
  4. The Frontend tab is titled "Recall วันนี้" and renders today (top, most prominent) + all overdue + tomorrow, in that order.
  5. All of the above render correctly and attractively in BOTH dark and light themes; customer name/HN are never red.
  6. Verified per Rule Q (L1 real browser OR L2 real client SDK) before "done" — not mock-only.

Risks / Open Questions

  • Phone format / missing phone: customerPhone is denormalized at recall create-time; some legacy recalls may lack it. Render falls back to hidden (no link) when empty; the tel: link strips non-digits. (Live-resolve from be_customers is out of scope — denormalized value is acceptable.)
  • outcomeBy overload: we reuse outcomeBy for the explicitly-picked staff and move the logged-in account to updatedBy. Confirm no existing reader expects outcomeBy to be the account (none found in explore; verify during implementation).
  • Customer-detail redundancy: phone is shown even on the customer's own detail page (user said "ทุกๆที่"); minor redundancy accepted intentionally.
  • Verification limitation: headless preview can't show visual flicker/theme nuance at tiny viewport — final theme + tap-to-call confirmation needs user L1 hands-on (Rule Q).
