Scheduled Tasks Tab — "งานอัตโนมัติ & ตารางเวลา"

Date: 2026-06-02 · Status: Approved by user (design) · Implementation plan: (to be written via writing-plans) · Full interactive mockup: public/brainstorm-scheduled-tasks.html (11 tasks)

Purpose / Problem

The app has 11 scheduled/automated tasks scattered across two runtimes (10 Vercel crons in vercel.json + 1 Firebase scheduled Cloud Function) — appointment reminders, a daily full-system backup, and a fleet of auto-delete/retention/sweep jobs. They are invisible, un-tunable at runtime, and "เยอะจนลืม" (so many the admin forgets them). The whole-app sweep also surfaced a silent conflict: staff chat is deleted by TWO jobs with different retention (Vercel 30d vs Firebase 7d), so the real retention is 7d, not the intended 30d — nobody knew.

This feature consolidates ALL 11 into one admin-only backend tab where each can be enabled/disabled, have its parameters tuned (retention days, grace, quiet hours), show its last-run status, and be manually triggered — all at runtime. The job schedule (when it fires) stays read-only because Vercel cron timing is deploy-time (vercel.json); changing it needs a deploy.

States covered: loaded (above); disabled task → toggle off + grey title + "ปิดอยู่ — ไม่ทำงาน"; safety-critical disable → confirm modal "ปิดแล้ว …จะ… แน่ใจไหม?"; never-run → "— ยังไม่เคยรัน"; failed run → red "✗ ล้มเหลว" + error tooltip; no-param tasks (stock-lot, chart-sweep) → only toggle + status + run-now.

Flow

Flow 1 — Toggle / tune param (runtime config)

Admin toggle off OR แก้ param ใน ScheduledTasksTab
→
Confirm? safety-critical + ปิด → modal บอกผล
→
saveSystemConfig patch scheduledTasks[id] + audit doc (atomic writeBatch — reuse เดิม)
→
system_config doc updated
→
Next cron run reads config → enabled:false = skip / ใช้ param ใหม่

Flow 2 — Cron run → last-run status (every scheduled fire)

Vercel/Firebase fires cron at its (fixed) schedule
→
readScheduledTaskConfig(id) admin SDK · fail-safe: error/missing → enabled+default
→
enabled? no → write status skipped + return · yes → do work w/ params
→
be_admin_audit (ledger, เดิม) + merge slice into scheduled_task_status doc
→
UI listener on status doc → badge updates live

Flow 3 — Run now (manual trigger)

Admin clicks ▶ รันตอนนี้ (destructive → confirm)
→
/api/admin/run-scheduled-task (verifyAdmin/perm) dispatch by taskId
→
shared core (same logic cron uses, apply:true)
→
status doc updated → UI badge "รันด้วยมือ · ✓"

Decisions (locked via Q&A with user)

  • Q1 = A (full): เปิด/ปิด + ปรับ param + last-run status + ปุ่ม run-now.
  • Scope = A: tab มีเฉพาะ 11 งานตั้งเวลา. งานลบแบบกดเอง 9 ตัว (delete-customer / cleanup-test-* / wipe-master / cleanup-phantom-branch / cleanup-orphan-stock) ไม่ย้ายมา — มี UI เดิมอยู่แล้ว.
  • Safety = A (warn-on-disable): งาน safetyCritical (backup / chatHistoryRetention / opdSessionCleanup) ปิดได้แต่เด้ง confirm บอกผล; งานทั่วไปปิดได้เลย. ไม่มี hard-lock.
  • Location = A: tab ใหม่ tab=scheduled-tasks ข้าง "ตั้งค่าระบบ" + permission scheduled_task_management เฉพาะ. ไม่ยัดเป็น section ในหน้าตั้งค่าเดิม (ยาวเกิน).
  • Schedule = read-only (Vercel cron timing = deploy-time; UI โชว์เวลาอ่านอย่างเดียว).
  • Fail-safe config read: cron config-read error/missing → default enabled + ค่าเดิม (งานสำคัญห้ามหยุดเงียบ).

Acceptance Criteria

  1. Tab scheduled-tasks แสดงครบ 11 งาน จัดกลุ่มตามหมวด พร้อม schedule (read-only), source badge (Vercel/Firebase), param inputs, last-run badge, ปุ่ม run-now.
  2. ปิดงาน (toggle off) + Save → system_config.scheduledTasks[id].enabled=false + audit doc; cron รอบถัดไปอ่านแล้ว skip จริง (เขียน status skipped). [Rule Q L2: client SDK เขียน config + รัน cron จริงบน prod ยืนยัน skip]
  3. แก้ param (เช่น retentionHours 24→48) + Save → cron รอบถัดไปใช้ค่าใหม่จริง. [L2]
  4. ปิดงาน safetyCritical → เด้ง confirm บอกผลก่อน; ปิดงานทั่วไป → ไม่เด้ง.
  5. ปุ่ม run-now (งาน destructive ต้อง confirm) → endpoint admin-authed รัน core จริง → status doc + badge อัปเดต. [L2/L1]
  6. last-run badge อัปเดต live หลัง cron/manual รัน (listener บน status doc).
  7. Config ว่าง/ก่อน migrate → ทุกงานทำงานเดิมเป๊ะ (enabled + ค่า hardcoded). [V14 lock]
  8. Config read fail (จำลอง) → cron ยัง enabled + ค่าเดิม (fail-safe). [unit + L2]
  9. เฉพาะ admin / perm scheduled_task_management เห็น+แก้ได้; คนอื่น 403/ซ่อน tab.
  10. Source-grep: ทั้ง 11 cron มี config guard + status write; registry param-default parity กับ _lib constants.

Risks / Open Questions

OPEN DECISION — staff-chat conflict (#5 Vercel 30d vs #6 Firebase 7d): แผนเสนอ ถอด Firebase function cleanupOldStaffChatMessages ทิ้ง เหลือ Vercel staff-chat-retention-sweep (30 วัน, ละเอียดกว่า มี orphan-sweep) เป็น single source → staff chat เก็บ 30 วันตามตั้งใจ. ถ้าจริง ๆ อยากเก็บแค่ 7 วัน ให้ตั้ง retentionDays=7 ที่ Vercel แทน. ต้องการ user ยืนยันก่อนแผนถอด function.
  • Deploy footprint (ใหญ่แต่ชัด): Vercel (frontend + 10 cron guards + run-now endpoint) + firebase deploy --only functions (ตัวที่ 11) + อาจ firestore.rules (status doc) → Probe-Deploy-Probe. ทำเป็น batch เดียว ตอน user สั่ง "deploy".
  • Run-now ของ Firebase fn (#6): เรียกข้าม runtime จาก Vercel ไม่ตรง ๆ — ถ้าถอด function ตาม open-decision ปัญหานี้หายไป (run-now ใช้ core ของ Vercel staff-chat แทน).
  • Param safety bounds: ทุก param มี min/max validate (กัน retention=0 ลบหมด / batchLimit ใหญ่จน timeout 300s). บาง param (batchLimit) อาจเลือกไม่ expose.
  • Wiring = deploy-coupled ครั้งเดียว: หลัง deploy guard แล้ว ทุกอย่างเป็น runtime. ก่อน deploy = พฤติกรรมเดิม (guard ยังไม่มี) — ปลอดภัย.
  • Cron core extraction: บาง cron orchestration ยังไม่อยู่ใน shared core เต็ม ๆ — run-now ต้อง extract (Rule C1); แผนจัดการต่อ task.
