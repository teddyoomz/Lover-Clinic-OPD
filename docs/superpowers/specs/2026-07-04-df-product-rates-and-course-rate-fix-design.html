DF Product Rates + Course Rate Fix — Design Spec

Date: 2026-07-04 · Status: Approved direction (Q1=A) — spec under user review · Implementation plan: 2026-07-04-df-product-rates-and-course-rate-fix.html (next step)

Purpose / Problem

User report (2026-07-04, with screenshot of DfEntryModal on mobile):

  1. บั๊ค: ค่ามือที่กรอกในหน้า "กลุ่ม DF (ค่ามือ)" ไม่ขึ้นใน modal "เพิ่มค่ามือแพทย์ & ผู้ช่วยแพทย์" ของ TFP — ทุกแถวขึ้น 0 บาท + "(ไม่มีอัตราในกลุ่มนี้)" ทั้งที่กรอกอัตราไว้แล้ว
  2. Gap: แถว "Shock wave" เฉยๆ (สินค้า/หัตถการ ที่เข้ามาทาง "+ เพิ่มรายการรักษา" หรือจากการติ๊กใช้สินค้าในคอร์ส) ไม่มีที่ให้ตั้ง master rate — หน้ากลุ่ม DF มี picker เฉพาะคอร์ส

Root cause ข้อ 1 — ยืนยันจากข้อมูล prod จริง (Rule R diag scripts/diag-df-rate-mismatch.mjs, 2026-07-04):

  • be_courses ทั้ง 405/405 docs ใช้ field courseName — 0 docs มี .name
  • TFP masterCourseIdByName (TreatmentFormPage.jsx:1991) อ่าน mc?.name → map ว่างเปล่า
  • → treatmentCoursesForDf fallback ใช้ "ชื่อคอร์ส" เป็น pseudo-id (customer.courses ไม่มี courseId)
  • → resolver getRateForStaffCourse เทียบ pseudo-name กับ rates[].courseId ที่เป็น id จริง (COURSES_1778150447655_AE530C40 ฯลฯ) → ไม่ match ทุกแถว → 0 บาท
  • อัตราที่กรอกไว้อยู่ครบ: กลุ่ม "แพทย์" (DFG-0626-ab5928b6) 120 อัตรา / กลุ่ม "ผู้ช่วยแพทย์" (DFG-0526-b1399741) 68 อัตรา รวม Shock Wave = 10 บาท ✓

Class-of-bug: V49-family (canonical-shape multi-reader-sweep) — จุดที่ V49 ตกสำรวจเพราะไม่ใช่ picker แต่เป็น name-map. เกิดตอน BSA/H-quater สลับแหล่งข้อมูล getAllMasterDataItems('courses') (legacy shape มี name) → listCourses() (canonical มี courseName) โดยไม่ได้กวาด reader ตัวนี้

  • สินค้าถูกลบทีหลัง: แถวอัตราแสดง courseName ที่ freeze ไว้ (พฤติกรรมเดิมของ rehydrate) — ลบแถวได้ปกติ

listCourses()→ map อ่าน mc.courseName || mc.name→ cid = COURSES_…AE530C40→ getRateForStaffCourse match rates[].courseId → 10 บาท (source: group)

Decisions (locked via Q&A with user)

  • Q1 = A: ค่ามือสินค้า/หัตถการ ตั้งที่หน้ากลุ่ม DF (section ที่ 2) — ไม่ทำ override รายบุคคลสำหรับสินค้าในรอบนี้ (เพิ่มทีหลังได้ ไม่ต้องแก้ schema)
  • Schema: ใช้ rates[] array เดิม + field kind:'product' — resolver ไม่แก้, ไม่ migrate ข้อมูลเก่า
  • ลำดับจับคู่ Source 2: course-name ก่อน (คงพฤติกรรม Phase 14.4 เดิม) → product-name ใหม่ → pseudo-name
  • Bugfix ไม่มีทางแยก: อ่าน courseName || name — canonical-first + legacy fallback

Acceptance Criteria

  1. เปิด TFP (นครราชสีมา) ลูกค้าที่ใช้คอร์ส Shock Wave → modal เพิ่มค่ามือ เลือกยาหยี + กลุ่มผู้ช่วยแพทย์ → แถวคอร์ส Shock Wave ขึ้น 10 บาท (จากกลุ่ม) อัตโนมัติ — ไม่ใช่ 0
  2. กรอกอัตรา "Shock wave" (สินค้า) ในหน้ากลุ่ม DF ได้ → แถว "Shock wave" ใน modal ขึ้นอัตรานั้นอัตโนมัติ
  3. อัตราคอร์สเดิมทั้ง 188 รายการ (120+68) ใช้ได้โดยไม่ต้อง migrate / กรอกใหม่
  4. Rule Q L2: diag script เรียก resolver จริงกับข้อมูล prod จริง → คืนอัตราตรงกับที่กรอก · Rule Q L1: ผู้ใช้ยืนยันบนหน้าจริง
  5. full vitest เขียว + build clean; แถวที่ไม่มีอัตรายังกรอกมือได้เหมือนเดิม

Risks / Open Questions

  • ชน id ข้าม namespace (คอร์ส vs สินค้า ใน rates[] เดียวกัน): id ProClinic เป็นตัวเลขทั้งคู่ (courses 56450+, products 38842+) — ช่วงไม่ทับกันในข้อมูลปัจจุบัน; DFG-4 (no-dup) กันซ้ำใน array อยู่แล้ว. ความเสี่ยงต่ำ, ยอมรับ
  • Double-pay: ติ๊กใช้คอร์สทำให้เกิดทั้งแถวคอร์ส + แถวสินค้า — ถ้าตั้งอัตราทั้งคู่จะ default-ติ๊กทั้งคู่ → แอดมินติ๊กออกเอง (พฤติกรรม enabled=value>0 เดิม). จะระบุใน docs/BUGS ไม่ทำ auto-dedupe
  • ขยะ be_df_staff_rates 2 docs (staffId 3841/3842 — 174 อัตราค่า 0, id ยุค ProClinic, ไม่มี branchId → ไม่ถูกโหลดอยู่แล้วเพราะ branch-scoped): เสนอ ลบทิ้ง (Rule M two-phase) — รอผู้ใช้อนุมัติตอน execute
