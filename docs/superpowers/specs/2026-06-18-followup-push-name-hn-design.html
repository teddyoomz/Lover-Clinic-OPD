Purpose / Problem

เมื่อลูกค้าส่งแบบประเมินติดตาม (ED follow-up) FCM push ที่เด้งหาแอดมินแสดง body แบบ generic "🔔 ได้รับข้อมูลผู้ป่วยแล้ว" — ไม่บอกว่าใครส่ง. แพทย์/แอดมินอยากรู้ทันทีว่า ชื่ออะไร · HN อะไร. สาเหตุ: follow-up opd_session เก็บ assessment answers ใน patientData (ไม่มี firstName) → patientName เป็น null → ตกไป body generic.

Architecture

  • NEW functions/customerDisplay.js (PURE CJS, no firebase) — CJS mirror of the ESM canonical src/lib/customerDisplayName.js: resolveCustomerName(customer) (walk patientData.firstNameTh/firstName → top-level firstname → customerName/name → nickname) + resolveCustomerHN(customer) (proClinicHN → pd.hn/HN → hn/HN/hn_no — V131: real HN lives in hn_no). The CF is a SEPARATE deploy package (Cloud Run, CJS) → cannot import the ESM src/lib; this mirror is the sanctioned duplicate (cross-link comment + a test asserting parity on key shapes). Drift risk acknowledged (resolvers rarely change).
  • NEW functions/notificationContent.js (PURE CJS, no firebase) — buildNotificationContent({ session, sessionId, customer, changedSections }) → { title, body }. Holds ALL the title/body branch logic (edit / follow-up / intake). Mirrors the pure-CJS-testable pattern of functions/assessmentMaterialize.js.
  • MODIFY functions/index.js — after the gates, read be_customers when linkedCustomerId present; replace the inline title/body block (lines ~66-84) with buildNotificationContent(...). Everything else (materialize, tokens, multicast, cleanup) UNCHANGED.
  • UNTOUCHED: firestore.rules, vercel, src/, createAssessmentSession (no session-shape change — uses existing linkedCustomerId + confirmInfo), intake + edit notification behavior.

Decisions (locked via Q&A with user)

  • Q1 = A: title = sessionName (form, เช่น "แบบประเมินติดตาม"); body = "🔔 {name} · HN {hn}". (ไม่ใช่ B ที่เอาชื่อขึ้น title.)
  • (defaulted) name = live-resolve จาก be_customers (V113) fallback confirmInfo.name snapshot; HN = resolveCustomerHN (hn_no จริง) fallback linkedCustomerId (รหัส LC-xxx = เลขลูกค้าที่ staff รู้จัก); intake คงเดิม (ไม่มี HN — ลูกค้ายังไม่ถูกสร้างตอน submit); edit คงเดิม.

Acceptance Criteria

  1. follow-up submit (linkedCustomerId + be_customers มี name+hn_no) → body = "🔔 {ชื่อจริง} · HN {hn_no}".
  2. customer ไม่มี hn_no (LC-customer) → HN = linkedCustomerId (เช่น "HN LC-26000082").
  3. อ่าน be_customers ไม่เจอ (ลูกค้าถูกลบ) → name = confirmInfo.name, HN = linkedCustomerId; ไม่ throw (push ยังส่ง).
  4. intake (pd.firstName, ไม่มี linkedCustomerId) → "🔔 ข้อมูลใหม่ · {name}" (เดิม, ไม่ regress).
  5. edit (updatedAt) → "✏️ แก้ไขแล้ว · {sections}" (เดิม, ไม่ regress).
  6. buildNotificationContent เป็น pure → unit-test ได้ทุก branch + edge (no name / no customer / long sessionName ตัด ≤28).
  7. resolveCustomerName/HN (CJS) ให้ผลตรงกับ ESM canonical บน shape ตัวอย่าง (parity test).
  8. be_customers อ่านด้วย canonical BASE_PATH (artifacts/{APP_ID}/public/data/be_customers) — Rule M path.
