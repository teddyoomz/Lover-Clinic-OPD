Recall Reason ทุกจุด + VIP ลูกค้า + Staff-Chat TFP Cards — Design Spec

Date: 2026-07-04 · Status: Approved by user ("อนุมัติ") · Implementation plan: 2026-07-04-recall-reason-vip-staffchat-cards.html (จะสร้างโดย writing-plans)

Purpose / Problem

4 feature requests (2026-07-04):

  1. Recall: พอบันทึกผลการติดต่อแล้ว เหตุผลแรกที่สร้าง recall หายไปจากจอ (ข้อมูล reason ยังอยู่ใน Firestore — recordRecallOutcome ไม่เคยทับ — เป็น UI conditional ใน RecallRow.jsx:126-129 ที่เลือกโชว์ outcomeNote แทน) → ต้องแสดงเหตุผล ทุกจุดที่แสดง recall
  2. VIP: toggle ลูกค้าเป็น VIP ในหน้าข้อมูลลูกค้า → ชื่อสีทอง + badge 👑 VIP ทุกที่ใน OPD system แบบ real-time ทันทีทุกจอ — ยกเว้นทุกช่องทางที่ลูกค้าเห็น (ลิ้ง/print/LINE) ห้ามแสดงเด็ดขาด
  3. Staff chat — ซักประวัติ: เมื่อกดปุ่ม teal "บันทึกข้อมูลซักประวัติ" ใน TFP → การ์ด "ระบบ" ใน staff chat: ชื่อลูกค้ากดได้ + ปุ่ม "เปิดบันทึกการรักษา" เปิด TFP ใบนั้นตรงๆ ให้แพทย์ลงบันทึกต่อ
  4. Staff chat — แพทย์บันทึกเสร็จ: เมื่อกดปุ่มม่วง "บันทึกสำหรับแพทย์" → การ์ดแบบเดียวกัน + แสดง "โดยแพทย์: {ชื่อแพทย์จากหัว TFP}" (detail.doctorName)
  5. (เพิ่ม 2026-07-04 รอบ 2) การ์ดรับเข้าเดิม: เพิ่มปุ่ม "📄 ดูข้อมูลรับเข้า" ใน bubble — เปิดดูข้อมูลรับเข้าที่ลูกค้ากรอก
  6. (เพิ่ม 2026-07-04 รอบ 2) การ์ดแบบประเมินติดตามเดิม: เพิ่มปุ่ม "📊 ดูแบบประเมิน" — เปิด EDDetailModal ตัวเดียวกับหน้าข้อมูลลูกค้า (modal สลับ/เปรียบเทียบรอบได้)

Flow

Flow ① — Recall render decision (RecallRow ใช้ร่วม 3 surfaces + 3 modals)
recall doc {reason, outcome, outcomeNote}→RecallRow Timeline→reason มี? → โหนดทอง "เหตุผลนัด" เสมอ
↓ outcome/outcomeNote มี?
มี → เส้นเชื่อม + โหนดฟ้า "ผลการติดต่อ" + chip ผลลัพธ์ ไม่มี → จบที่โหนดเดียว
RecallOutcomeModal / RecallSnoozeMenu / RecallLineTemplateModal→แถบ "🏷 นัดเพราะ: {reason}" ใต้หัว modal
ข้อมูลไม่แก้เลย — recordRecallOutcome ไม่เคยทับ reason (ยืนยันแล้ว backendClient.js:12991)
Flow ② — VIP toggle → real-time ทุกจอ (listener เดียว)
CDV: กดปุ่ม 👑 VIP→updateCustomer(id, {vip, vipAt, vipBy})→be_customers doc
VipProvider (mount ใน BackendDashboard + AdminDashboard เท่านั้น)→onSnapshot where('vip','==',true)→Set<customerId> ใน context
ทุก surface: useIsVip(customerId) / <VipName> / <VipBadge>→snapshot ยิง → ทุกจอเปลี่ยนทันที (denormalized name ไม่เกี่ยว — key ด้วย customerId)
EXCLUDE: PatientForm / PatientDashboard / ClinicSchedule / Sale+Quotation print / documentPrintEngine / LINE ทั้งหมด — VipProvider ไม่ mount + source-grep guard
Flow ③④ — TFP save → การ์ด staff chat → แพทย์เปิดกลับ
TFP handleSubmit('vitals' | 'doctor')→save สำเร็จ (ได้ treatmentId)→เขียนการ์ด client-side (try/catch ไม่กระทบ save)
id = CHAT-SYS-TFP-{treatmentId}-{vitals|doctor} (ตายตัว → save ซ้ำ = create ชน → เงียบ = ไม่มีการ์ดซ้ำ)
system: {kind, treatmentId, customerId, nameSnapshot, hnSnapshot, doctorName?} branchId = สาขาของการรักษา
firestore.rules→client เขียน system ได้เฉพาะ kind ∈ {tfp-vitals, tfp-doctor} + validators · intake/followup ยังปลอมไม่ได้
StaffChatSystemCard (kind ใหม่)→ชื่อกดได้ → /?backend=1&customer={id} ปุ่ม → /?backend=1&customer={id}&treatment={BT-id}
BackendDashboard deep-link useEffect (ขยาย)→?treatment= → getCustomer → setTreatmentFormMode({mode:'edit', treatmentId, ...})→TFP ใบนั้นเปิดทันที
เสียง + unread→อัตโนมัติตาม pipeline เดิม (V82 cursor)
Flow ⑤ — ปุ่ม "ดูข้อมูลรับเข้า" (fallback chain ใช้ได้เสมอแม้ session ถูกลบ)
กดปุ่มบนการ์ดรับเข้า→getDoc(opd_sessions/{sessionId})→มี doc → ใช้ session ตรง
↓ ไม่มี (booking flow ลบทิ้งแล้ว — AV131)
getCustomer(resolvedCustomerId) → synthesizeSessionFromCustomer() (__synthetic — pattern เดียวกับ AppointmentHub)
→portal modal ในแชท (precedent: StaffChatImageLightbox)→OpdIntakeDetailBody (shared — สกัดจาก modal คิว AdminDashboard → สอง view ไม่ drift)
Flow ⑥ — ปุ่ม "ดูแบบประเมิน" (EDDetailModal ตัวจริง)
กดปุ่มบนการ์ด followup (มี customerId เสมอ)→getCustomer → intakePerf (pickKioskAssessmentFields) listenToAssessments(customerId) — universal
→deriveRounds(intakePerf, assessments)→<EDDetailModal type={typeแรกที่มีข้อมูล} round={hero ล่าสุด} rounds hero isDark onClose /> — portal to body อยู่แล้ว (AV98)
Edge: Cloud Function เพิ่ง materialize รอบ (status pending ชั่วครู่)→listener ยิงซ้ำ → modal อัปเดตเองในไม่กี่วินาที

Architecture

Feature ① — Recall reason ทุกจุด

  • ไม่มี data change. reason preserved อยู่แล้ว
  • RecallRow.jsx: แทนที่ prominent-note box (บรรทัด ~126-129 + ~339-350) ด้วย Timeline block — โหนดเหตุผล (ทอง) เสมอเมื่อ reason ไม่ว่าง; โหนดผลติดต่อ (ฟ้า) เมื่อ outcome ถูกบันทึก (แสดง outcome chip inline; outcomeNote เป็นเนื้อความ ถ้าไม่มี note แสดง label ผลลัพธ์อย่างเดียว); ยังไม่ติดต่อ → โหนดเดียว; compact mode ใช้โครงเดียวกันแบบแน่นขึ้น (ครอบ RecallTab / RecallFrontendView / CDV RecallCard อัตโนมัติ)
  • RecallOutcomeModal.jsx + RecallSnoozeMenu.jsx + RecallLineTemplateModal.jsx: เพิ่มแถบ "🏷 นัดเพราะ: {reason}" ใต้ header (LINE message ที่ส่งจริงไม่แตะ)
  • RecallEditModal มีช่อง reason อยู่แล้ว — ไม่แก้; pill นับจำนวน — ไม่เกี่ยว

Feature ② — VIP

  • Schema: be_customers.vip: boolean + vipAt (serverTimestamp เมื่อ transition) + vipBy (uid) — mirror V41 isHidden + toggle notifyOptOut precedent (CDV:509-514) — ไม่ต้องแก้ firestore.rules (staff เขียน be_customers ได้อยู่แล้ว :139-141)
  • Toggle: ปุ่มในหัวการ์ด CDV ใกล้ชื่อ — staff ทุกคนเห็น/กดได้ (Q3=A); optimistic + live ผ่าน listenToCustomer ที่ CDV มีอยู่แล้ว
  • VipProvider (ไฟล์ใหม่ เช่น src/lib/VipContext.jsx): listener เดียว onSnapshot(query(be_customers, where('vip','==',true))) ผ่าน scopedDataLayer (universal collection; single-field where → ไม่ต้องมี composite index) → Set<customerId>; mount เฉพาะ BackendDashboard + AdminDashboard root (ห้าม App root — anon บนลิ้ง public จะโดน permission-denied)
  • ตัวแสดงผลกลาง (ไฟล์ใหม่ src/components/VipBadge.jsx): useIsVip(customerId) + <VipName> (ชื่อทอง dark #fbbf24 / light #b45309 ตาม theme) + <VipBadge> (👑 VIP chip ทอง) — นอก provider → คืน false เงียบๆ (ปลอดภัยโดยโครงสร้าง)
  • จุดเสียบ: CustomerOption.jsx (คุม 8 ไฟล์: CDV header, TFP header, AdminDashboard, DepositPanel, AppointmentCalendarView, AppointmentFormModal, CustomerCard, AppointmentLineBadge) + รายจุด inline: CustomerCard grid, SaleTab rows, AppointmentHubRowCard, AppointmentDetailBody (peek+modal), AdminDashboard queue (key ด้วย session.brokerProClinicId), recall rows, StaffChatSystemCard, Quotation/OnlineSales/InsuranceClaims tabs, reports rows (ทุก tab ที่มี customerId), MovementLogPanel, TreatmentTimelineModal, LinkRequestsTab, Membership/WalletPanel — ไล่ให้ครบใน plan (สแกน + enumerate เป็น task)
  • Exclusions (ห้ามมี vip render): PatientForm / PatientDashboard / ClinicSchedule / SalePrintView / QuotationPrintView / documentPrintEngine + templates / appointmentHubPrintTemplate / lineBotResponder / lineReminderTemplate / api/webhook / api/cron / ED-scoring ลิ้งลูกค้า — ล็อกด้วย source-grep guard test (ไฟล์เหล่านี้ต้องไม่ import VipBadge/VipContext/useIsVip)
  • Rule file update: .claude/rules/04-thai-ui.md — สีทองใช้ได้แล้ว (user 2026-07-04) ห้ามเฉพาะแดงบนชื่อ/HN

Feature ③④ — Staff-chat TFP cards

  • Kind ใหม่ 2 ตัว ใน system.kind: tfp-vitals ("📋 บันทึกซักประวัติเสร็จแล้ว") + tfp-doctor ("🩺 แพทย์ลงบันทึกเสร็จแล้ว" + "โดยแพทย์: {detail.doctorName}")
  • Writer: helper ใหม่ฝั่ง client (คู่กับ staffChatClient.js) เรียกจาก TFP handleSubmit หลัง save สำเร็จเมื่อ saveMode==='vitals'|'doctor' — try/catch non-fatal; doc id ตายตัว CHAT-SYS-TFP-{treatmentId}-{kind} + rules update:false → save ซ้ำไม่สร้างการ์ดซ้ำ (already-exists → swallow); snapshot ชื่อ/HN/customerId/doctorName มีครบใน TFP ตอน save
  • firestore.rules: แก้ clause !('system' in ...) ของ be_staff_chat_messages → อนุญาต system map เมื่อ system.kind ∈ {'tfp-vitals','tfp-doctor'} + validators (treatmentId/customerId string ไม่ว่าง) — kind อื่น (intake/followup) ยัง client-forge ไม่ได้; ต้อง deploy rules + Probe-Deploy-Probe + เพิ่ม probe ใหม่ (anon เขียน tfp card → 403; staff เขียน kind intake → denied)
  • Deep link ใหม่: buildTreatmentEditUrl(customerId, treatmentId) ใน customerNavigation.js → /?backend=1&customer={id}&treatment={BT-id}; ขยาย deep-link useEffect ใน BackendDashboard.jsx:232-282 ให้ ?treatment= → getCustomer → setTreatmentFormMode({mode:'edit', treatmentId, customerId, customerHN, patientName, patientData}) (mirror opener เดิม :482-502)
  • Card UI: ขยาย StaffChatSystemCard.jsx — kind ใหม่ 2 ตัว + ปุ่ม violet gradient (target _blank) + บรรทัดแพทย์; ชื่อลูกค้าลิงก์แบบเดิม; ไม่ต้อง live-resolve (ลูกค้าลงทะเบียนแล้ว ณ จุด TFP)
  • ไม่ทำ (YAGNI): การ์ดตอน full staff-save (finalize) — user ขอเฉพาะซักประวัติ + แพทย์; Cloud Function trigger — Q2=A client-write

Feature ⑤⑥ — ปุ่มบนการ์ดเดิม (รับเข้า + แบบประเมินติดตาม)

  • ⑤ ดูข้อมูลรับเข้า: สกัดส่วนแสดง patientData ของ renderViewingSessionModal (AdminDashboard.jsx:~5040-5230 — ข้อมูลส่วนตัว/ฉุกเฉิน/สุขภาพ/clinical summary) เป็น shared component ใหม่ OpdIntakeDetailBody (V127 shared-body pattern) → AdminDashboard modal เดิมใช้ (behavior เดิมเป๊ะ) + การ์ดแชทเปิด portal modal ใหม่ (precedent StaffChatImageLightbox) host ตัวเดียวกัน — ใช้ได้ทั้ง 2 dashboard mount
  • ⑤ data source: getDoc(opd_sessions/{system.sessionId}) ก่อน → ไม่มี (booking flow ถูก hard-delete, AV131) → getCustomer(customerId ที่ resolve แล้ว) + synthesizeSessionFromCustomer() (opdSessionState.js:85, __synthetic — precedent AppointmentHubView:327) → เปิดได้เสมอ; ทั้งคู่ยังไม่มี (การ์ด pending + session หาย = case ผิดปกติ) → ข้อความ "ไม่พบข้อมูลรับเข้า"
  • ⑥ ดูแบบประเมิน: reuse EDDetailModal ตรงๆ (pure display + portal to body + AV78/AV98 อยู่แล้ว — ไม่ fork): กดปุ่ม → getCustomer → intakePerf = pickKioskAssessmentFields(patientData) + listenToAssessments(customerId) (universal, staff-read) → deriveRounds → เปิดที่ hero (รอบล่าสุด) + type แรกที่มีข้อมูล; ปิด → unsubscribe; listener ทำให้รอบที่ Cloud Function เพิ่ง materialize เด้งเข้า modal เอง
  • ปุ่มทั้งคู่: สไตล์ violet gradient เดียวกับ "เปิดบันทึกการรักษา" (consistency ทุกการ์ดระบบ); isDark จาก useTheme (source เดียวกับ CDV)
  • ไม่แตะ: ตัว writer ฝั่ง Cloud Function ของการ์ดรับเข้า/followup (การ์ดเดิมสมบูรณ์ — เพิ่มเฉพาะฝั่ง render)

Decisions (locked via Q&A with user)

  • ซักประวัติ = A: ปุ่ม teal "บันทึกข้อมูลซักประวัติ" ใน TFP (ไม่ใช่ฟอร์มรับเข้าที่มีการ์ดอยู่แล้ว)
  • กลไกการ์ด = A: client เขียนตรง + แก้ rules แบบแคบ (deploy rules + Probe-Deploy-Probe)
  • สิทธิ์ VIP = A: staff ทุกคน toggle ได้
  • สีชื่อ VIP = Gold: dark #fbbf24 / light #b45309 + badge 👑 VIP — user ยืนยัน "ชื่อสีทองได้นะห้ามแดงเฉยๆ" (supersede กฎห้ามทองเดิม → อัปเดต 04-thai-ui.md)
  • Recall layout = C: Timeline เล่าเรื่อง (เลือกจาก 5 แบบ A-E)
  • Recall scope: เหตุผลแสดงทุกจุดที่แสดง recall (ลิสต์ ×3 ผ่าน RecallRow + modal ×3)
  • ปุ่มการ์ด = A: Violet gradient เรืองแสง + chevron, label "เปิดบันทึกการรักษา"
  • Theme: ทุกอย่างต้องเข้ากับ light theme ด้วย (AA ทั้งสองธีม — aaAccent V125 pattern)

Acceptance Criteria

  1. Recall ที่มีผลการติดต่อแล้ว → เห็นทั้งเหตุผลนัดและผลการติดต่อ (Timeline) ในลิสต์ backend / Recall วันนี้ / หน้าลูกค้า; recall ที่ยังไม่ติดต่อ → เห็นเหตุผลเหมือนเดิม
  2. Modal บันทึกผล / เลื่อน / ส่ง LINE แสดง "นัดเพราะ: {reason}"; ข้อความ LINE ที่ส่งจริงไม่เปลี่ยน
  3. กด toggle 👑 ใน CDV → ชื่อลูกค้าเป็นสีทอง + badge 👑 VIP ปรากฏทันที (ไม่ต้อง refresh) ในทุก surface ภายในที่เปิดอยู่ (ลิสต์ลูกค้า, ขาย, นัดหมาย, คิว, recall, staff chat, รายงาน, ฯลฯ); กดปิด → หายทันทีทุกจอ
  4. หน้า/เอกสาร/ข้อความที่ลูกค้าเห็น (patient links, schedule link, ใบเสร็จ/ใบเสนอราคา/เอกสาร print, LINE) ไม่มีร่องรอย VIP ใดๆ — มี test ล็อก
  5. field vip รอดหลังบันทึกฟอร์มแก้ไขลูกค้า (updateCustomerFromForm) — มี test ล็อก
  6. กด "บันทึกข้อมูลซักประวัติ" ใน TFP → การ์ด 📋 โผล่ใน staff chat สาขานั้น (ชื่อกดได้ + HN + ปุ่มเปิดบันทึกการรักษา); กด "บันทึกสำหรับแพทย์" → การ์ด 🩺 + "โดยแพทย์: {ชื่อ}"
  7. ปุ่ม "เปิดบันทึกการรักษา" เปิดแท็บใหม่เข้า TFP ใบนั้นในโหมดแก้ไขโดยตรง
  8. กด save ซ้ำใบเดิมโหมดเดิม → ไม่มีการ์ดซ้ำ; การ์ดเขียนไม่สำเร็จ → การ save ไม่พัง
  9. Client ปลอมการ์ด kind intake/followup ไม่ได้ (rules ยังบล็อก); anon เขียนการ์ดใดๆ ไม่ได้ (403)
  10. ทุก UI ใหม่ผ่าน AA ทั้ง dark + light theme
  11. การ์ดรับเข้า: ปุ่ม "ดูข้อมูลรับเข้า" เปิด modal ข้อมูลรับเข้า (เนื้อหาเดียวกับกด "ดูข้อมูล" ในคิว) ได้จากแชททั้งฝั่ง frontend และ backend — รวมถึงลูกค้า booking flow ที่ session ถูกลบแล้ว (synthetic จาก be_customers)
  12. การ์ดแบบประเมินติดตาม: ปุ่ม "ดูแบบประเมิน" เปิด EDDetailModal (สลับ type / เปรียบเทียบรอบ / swap ได้ครบเหมือนใน CDV) โดยแสดงรอบล่าสุด; ถ้าคะแนนเพิ่ง materialize → เด้งเข้า modal เองไม่ต้องปิดเปิดใหม่
  13. Modal ในคิว AdminDashboard เดิมแสดงผลเหมือนเดิมทุกประการหลังสกัด shared body (anti-regression test)

Risks / Open Questions

  • updateCustomerFromForm rebuild (V145-class): ฟอร์มแก้ไขลูกค้า rebuild doc — ต้อง verify ตอน implement ว่า top-level vip ไม่ถูก strip (ถ้า strip → แก้ให้ carry + test ล็อก)
  • VIP sweep completeness (V12-class): ชื่อลูกค้าโผล่ ~74 ไฟล์ — plan ต้องมี task สแกน enumerate + classifier test (internal = มี VipBadge / customer-facing = ห้ามมี) กันตกหล่น
  • Rules narrowing: การเปิดให้ staff เขียน system card kind tfp-* = staff ปลอมการ์ด tfp ได้ในทางทฤษฎี — ยอมรับ (staff auth แล้ว + พิมพ์ข้อความหลอกได้อยู่แล้ว); intake/followup ยัง server-only
  • AdminDashboard queue: คิวใช้ opd_sessions (denorm) — VIP ต้อง key ด้วย session.brokerProClinicId; ลูกค้า walk-in ที่ยังไม่ลงทะเบียน (ไม่มี customerId) จะยังไม่โชว์ VIP จนกว่าจะ link — ถูกต้องตาม semantic
  • Deploy: feature นี้ต้อง deploy ทั้ง vercel + firestore.rules (V15 combined + Rule B Probe-Deploy-Probe + probe ใหม่) — รอคำสั่ง "deploy" explicit (V18)
  • ⑤ AdminDashboard extraction (V21-class): การสกัด OpdIntakeDetailBody ออกจากไฟล์ 8000+ บรรทัด ต้อง byte-behavior-identical (destructive buttons gated __synthetic ตามเดิม) + เช็ค field-parity ของ synthetic session (be_customers.patientData เก็บ kiosk-shape ud_*/hasAllergies ครบไหม — V118 precedent บอกว่า render ได้ แต่ต้อง diff ตอน implement)
  • ⑥ be_assessments rules: ยืนยัน read rule ผ่านสำหรับ staff จาก context แชท (CDV ใช้ listener เดียวกันอยู่แล้ว — คาดผ่าน; เช็คตอน implement)
