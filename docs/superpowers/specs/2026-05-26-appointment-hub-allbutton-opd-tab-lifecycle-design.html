Appointment Hub — All-types button + OPD-pending tab + link auto-cleanup

Date: 2026-05-26 · Status: Approved by user (design) · Surface: Frontend AdminDashboard → "นัดหมาย" (V64 AppointmentHub) + OPD-session lifecycle (cron + save hook) · Implementation plan: docs/superpowers/plans/2026-05-26-appointment-hub-allbutton-opd-tab-lifecycle.html (pending)

Purpose / Problem

Four connected asks on the Frontend "นัดหมาย" tab + the patient-fill-link (opd_sessions) lifecycle:

  1. ① ปุ่มเดียวจองได้ทุกประเภท — วันนี้ปุ่ม "เพิ่มคิว Walk-in" เปิด kiosk session modal (flow แบบบันทึกลูกค้าก่อน→ค่อยสร้างนัด). เปลี่ยนเป็น "เพิ่มนัดหมาย" ที่เปิด AppointmentFormModal ตัวเดียวกันเป๊ะที่ AppointmentHubView render อยู่แล้ว (วันนี้ใช้ในโหมด edit ที่บรรทัด 609) — โหมด create, เลือกได้ทั้ง 5 type. ไม่สร้าง modal ใหม่ ไม่สร้าง flow ใหม่; เป็น component + save-path เดียวกับปุ่ม "เพิ่มนัดหมาย" ใน appointment-all/ปฏิทิน.
  2. ② แท็บ "รอ/ยังไม่ลง OPD" — รวบรวมนัด (ปัจจุบัน+อนาคต) ที่ลูกค้ายังรอกรอกข้อมูล หรือแอดมินยังไม่บันทึกลง OPD ไว้ในที่เดียว วางหลังแท็บ "ย้อนหลัง 30 วัน". วันที่เลยมาแล้วไม่ต้องโชว์.
  3. ③ ลบลิงก์อัตโนมัติเมื่อเลยวันนัด — opd_session ที่เป็นลิงก์ให้ลูกค้ากรอกใน PatientForm ถูกลบเองเมื่อเลยวันนัดของลูกค้าคนนั้น.
  4. ④ ลบ session+ลิงก์ทันทีเมื่อ save OPD สำเร็จ — ลูกค้าไม่ต้องกรอกอะไรอีก; ลบเลยเพื่อกันลิงก์ตกค้างจนระบบรวน (อ้างอิงอาการ perf 2026-05-24: opd_sessions/chat_history listener ดึง doc มากเกินไป).

Root motivation ของ ③+④: ป้องกัน opd_sessions สะสมจนกระทบ performance (listener payload). ③ = no-show/abandoned path; ④ = happy path.

Decisions (locked via Q&A with user, 2026-05-26)

  • Q1 = B+C+D: tab ② shows every not-yet-in-OPD appointment incl. "ยังไม่ส่งลิงก์" (B), present+future, past hidden.
  • Q2 = hard-delete: obsolete opd_session is hard-deleted (no archive) — collection stays lean (perf goal).
  • Q3 = delete all date-passed: once apptDate < today, delete the link even for linked sessions (override V116 preserve-on-expire) and even filled-but-unsaved (D). Tab ② warns admin to save before the date.
  • Q4 = single all-types button: replace Walk-in button; Walk-in remains 1 of 5 types in the modal; kiosk inverted flow retires.

Acceptance Criteria

  1. Frontend "นัดหมาย" tab shows "เพิ่มนัดหมาย" (not "เพิ่มคิว Walk-in"); clicking opens AppointmentFormModal with all 5 types selectable; creating each type produces the same artifacts as the calendar's "เพิ่มนัดหมาย" (appt; deposit-pair for มัดจำ; fill-link/card for booking-types) — parity verified (R1).
  2. New 5th primary pill "รอ/ยังไม่ลง OPD" appears after "ย้อนหลัง 30 วัน" with a count badge; list = state {B,C,D} appointments dated today-or-future; past + state A/E excluded.
  3. An opd_session linked to an appointment whose date has passed is hard-deleted by the cron (incl. filled-but-unsaved D); a fresh reschedule mints a new link.
  4. On successful OPD save, the originating opd_session is hard-deleted immediately (best-effort; save never rolls back on delete failure).
  5. No regression in V116 (queue-delete still preserves link before the date passes), V124/V125 bubble+cascade, V126 mark-complete gate.

Verification (Rule Q + Q-honest)

  • Helpers (isAppointmentOpdPending, decideCleanupAction new branch, filter): unit tests + Rule I flow-simulate (state matrix B/C/D/A/E × date past/today/future; cleanup decision order; idempotency).
  • Cron-delete + save-delete (touch real Firestore): Rule Q L2 admin-SDK e2e on real prod with TEST-/E2E- prefixed fixtures — date-passed session → deleted; save-attach → session deleted; re-run idempotent; zero orphans + audit doc.
  • Tab + button UI: Rule Q L1 Chrome MCP at verify stage (Rule S) — relabeled button opens all-types modal; new pill renders + filters; + user hands-on post-deploy.
  • Q-honest: disclose the test-vs-claim gap explicitly — L2 doc-level is acceptable here (no new composite index / rules); L1 real-browser for the click-flow is the user's gold standard.

Risks / Open Questions

  • R1 (parity — mostly resolved): ① reuses the SAME AppointmentFormModal that AppointmentHubView already renders (edit) and ปฏิทิน uses (create) → save-path/artifact parity is automatic as long as the create-open passes the same props ปฏิทิน's openCreate passes (lockedAppointmentType:null, initialDate, existingAppointments). Residual to confirm in the plan: the modal's no-deposit-booking create produces the fill-link/card the same way it does today (existing behavior, not new code).
  • R2 (data loss, accepted): Q3=A deletes a filled-but-unsaved (D) link when its date passes → the customer's filled data is lost. Tab ② is the mitigation (prompts admin to save before the date).
  • R3 ("อนาคต" range): tab ② = today + all future (no 30-day cap). Change if a cap is wanted.
  • R4 (state-B noise): state B may include non-booking appts (treatment-in/follow-up/walk-in with no customerId + no link). May add UI noise; predicate can be narrowed to booking-types in the plan if needed.
  • R5 (apptDate source): cron ③ needs the appointment date — confirm the opd_session stores it (appointmentDate/appointmentDateTime) or join be_appointments via linkedAppointmentId in the sweep.
  • R6 (showSessionModal): confirm the kiosk session modal isn't used by another button before removing its wiring (retire only if Walk-in was its sole caller).
