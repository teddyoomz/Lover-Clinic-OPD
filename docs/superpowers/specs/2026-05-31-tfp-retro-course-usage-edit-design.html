V136 — TFP retroactive course-usage edit

Date: 2026-05-31 · File: src/components/TreatmentFormPage.jsx · Trigger: user — "ในหน้า TFP ทำให้ edit ข้อมูลการใช้คอร์สย้อนหลังได้ ถ้ายังไม่มีการใช้คอร์ส/ตัดคอร์สใดๆในใบนั้น แต่ถ้ามีการใช้อะไรไปแล้ว จะ edit ไม่ได้เหมือนเดิม".

Problem / current behaviour

Opening a finalized treatment opens TFP in edit mode, but the ข้อมูลการใช้คอร์ส section renders a locked read-only table (the user's screenshot: "รายการรักษา → ไม่พบรายการรักษา"). The gate:

const canAddNewItems = (mode === 'create')
  || (loadedTreatmentStatus === 'doctor-recorded')
  || (loadedTreatmentStatus === 'vitalsigns-recorded');

A normal finalized treatment → canAddNewItems === false → locked. This lock is a deliberate guard: editing a treatment that deducted courses requires reverse-then-reapply (the V43–V49 / V104 course-balance danger zone).

Locked decisions (brainstorm 2026-05-31)

Q1What counts as "no usage" (unlock condition)?No course deducted — courseItems & treatmentItems empty at load. note/vitals/images don't count; consumables/meds present don't block. (matches screenshot)
Q2What becomes editable when unlocked?Only the ข้อมูลการใช้คอร์ส section. consumables / take-home meds / OPD notes stay as-is.

Design

Render gate (NEW, distinct from canAddNewItems)

// captured once at edit-load:
loadedHasNoCourseUsage = !(t.courseItems?.length) && !(t.treatmentItems?.length)

const canEditCourseUsageRetro = isEdit && !canAddNewItems && loadedHasNoCourseUsage;
const courseUsageInteractive  = canAddNewItems || canEditCourseUsageRetro;
  • The course-section locked-vs-grid branch keys on courseUsageInteractive.
  • The ซื้อ buttons stay on canAddNewItems ONLY → hidden in retro mode (Q3).
  • consumables (สินค้าสิ้นเปลือง) + take-home meds (ยากลับบ้าน) sections keep their own canAddNewItems gate → stay locked (Q2).

Save: dedicated saveMode = 'course' = staff-save MINUS auto-sale

The interactive grid (คอร์ส / โปรโมชัน / รายการรักษา columns) is entirely "use existing courses" (checkbox → toggleCourseItem → selectedCourseItems). The 3 ซื้อ buttons (hidden) are the only buy path. So the only remaining risk is the sale path firing when a pre-existing sale exists (consumables present → hasSale=true). saveMode='course' closes that:

Save step (handleSubmit)'staff''course' (retro)
Course over-deduction validation (2352)✓✓ (runs)
Reverse old course deduction (2566)✓✓ no-op (empty)
deductCourseItems — the new selection (2674)✓✓ deducts

Why a dedicated mode, not reuse 'staff': the 2944 edit-sale path creates a new sale when !linkedSale and runs the full deposit/wallet/points reverse-reapply saga. For a consumables-only treatment (hasSale=true, no course) that would fire on a course-only retro-edit — violating Q3's "zero money impact". Skipping only the sale path keeps course-balance + stock correct (matching the existing finalize path) while guaranteeing no money/INV side-effects.
Scope (Rule P): the locked course table exists ONLY in TFP. TreatmentReadOnlyPanel (timeline/split-screen) is intentionally read-only and out of scope. Single surface, single gate. No class-of-bug expansion.
