Mobile-Load Reliability Design Spec

Date: 2026-06-16 · Status: Approved by user · Implementation plan: docs/superpowers/plans/2026-06-16-mobile-load-reliability.html

Purpose / Problem

On mobile, pages SOMETIMES fail to load on the first try — (a) hang on a black screen, (b) hang on a "กำลังโหลด" spinner forever, or (c) load but show only an empty skeleton with NO data. The only recovery today is a manual refresh / close-reopen / re-scan. This hits BOTH the staff app (AdminDashboard Frontend + BackendDashboard) AND every customer-facing shareable link (PatientForm, ED Scoring, view-data, view-appointment). For customer links this is worse — the customer can't easily recover, and may act on incomplete/missing data (clinical risk).

Mockup Design

The target is a 3-state load machine at every entry point: loading (existing skeleton/spinner) → silently auto-retries → ready (data) OR, only if all retries fail, error+retry (never a dead/stuck screen).

useResilientLoad API (contract)

const { loadStatus, retryKey, markReady, markError, retry } = useResilientLoad({
  softTimeoutMs = 8000,   // no data in this window → auto-retry (silent)
  maxAutoRetries = 1,     // silent retries before showing error
});
// loadStatus: 'loading' | 'ready' | 'error'
// retryKey:   number — include in the load effect's deps; bumping re-runs the load
// markReady(): call inside the onSnapshot/fetch SUCCESS callback (doc-found OR not-found both count as "loaded")
// markError(err?): optional — call inside onError; funnels to the same auto-retry/error path
// retry(): bound to the LoadErrorRetry button — resets attempts → loading → bump retryKey
// On the FIRST auto-retry the hook performs ONE disableNetwork→enableNetwork (heals a half-dead connection that a bare re-subscribe can't).

Decisions (locked via Q&A with user)

  • Q1 = No offline cache (fresh-always): customers must never see stale course balance / appointment time. Fix via connection layer + timeout/retry only. No IndexedDB persistence.
  • Q2 = Auto-retry → error + "ลองใหม่" button: ~8s soft timeout → 1 silent auto-retry → if still failing, the LoadErrorRetry card with a retry button. No silent dead screen, no full page reload.
  • Q3 = All three test layers + L2: Playwright mobile-viewport e2e (throttle / offline→online / tab-suspend→resume) + unit (state machine, fake timers) + RTL flow-sim (stuck-snapshot → error+retry) + L2 client-SDK cold-start (real prod, proves long-polling fallback).
  • Scope: connection layer = global (all pages incl. all backend tabs). Resilient-load UI escape = customer links + AdminDashboard queue. Backend tabs = silent auto-heal via useBranchAwareListener + Suspense retry. (Per-tab error buttons beyond the shared hook = out of scope unless requested.)
  • KEEP V16 + V17 as-is; auto-retry reuses the same network-toggle path.

Acceptance Criteria

  1. On a real mobile viewport, every customer link (PatientForm/?session= · PatientDashboard/?patient= · ClinicSchedule/?schedule= · ED Scoring) loads data on first try under normal network.
  2. offline→online mid-load → data appears WITHOUT a manual refresh.
  3. tab background→foreground → data refreshes (V17) WITHOUT a refresh.
  4. Blocked/half-dead network → after ~8s auto-retry then a clear "โหลดข้อมูลไม่สำเร็จ — ลองใหม่" card (NOT a stuck spinner / black screen / empty skeleton). Tapping ลองใหม่ recovers once the network returns.
  5. No page can reach a permanent stuck-skeleton state: every load path ends in ready OR error+retry.
  6. Staff queue (AdminDashboard) + backend tabs auto-heal a half-dead connection (silent re-subscribe) without a manual refresh.
  7. Customers never see stale cached data (no persistence).
  8. L2 client-SDK cold-start against real prod returns data via long-polling fallback.
  9. Full vitest green + build clean + Playwright mobile e2e green.

Risks / Open Questions

  • autoDetectLongPolling vs forceLongPolling: autoDetect chosen (keeps WebSocket speed where it works; falls back only when broken). If real-device testing shows the auto-detection itself is unreliable on a target carrier, escalate to experimentalForceLongPolling (slower but bulletproof) — name the ceiling in a ponytail: comment.
  • 8s timeout tuning: too short = false auto-retries on slow-but-fine networks; too long = user stares longer. 8s + 1 silent retry ≈ ~16s worst case before the error card. Constant, tunable.
  • useBranchAwareListener retry + branch-switch interplay: the internal nonce must not fight the existing effectiveBranchId re-subscribe (BSA). Reset attempts on branch change.
  • doc-not-found vs load-failure: markReady() must fire for a legit "doc doesn't exist" snapshot (that's a successful load → the page shows its own notfound/expired), NOT be treated as a timeout. Only a snapshot that NEVER fires (or onError) triggers retry.
  • Test honesty (Rule Q): mock/unit tests prove the state machine; only Playwright mobile (L1) + L2 client-SDK cold-start prove the real network behavior. Both required before "no bug" claim.
