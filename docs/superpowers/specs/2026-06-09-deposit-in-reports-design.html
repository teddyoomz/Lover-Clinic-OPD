มัดจำในรายงาน — reports-payment + reports-sale

Date: 2026-06-09 · Status: Approved by user (Q1–Q4 locked) · Plan: docs/superpowers/plans/2026-06-09-deposit-in-reports.html · Mockup (live): .superpowers/brainstorm/.../deposit-in-reports.html

Purpose / Problem

reports-payment ("สรุปบัญชีรับชำระ") = เงินที่บัญชีได้รับจริง แต่ปัจจุบันรวมเฉพาะช่องทางชำระจาก be_sales. เงิน มัดจำ (be_deposits) ที่ลูกค้าจ่ายเข้ามา ไม่ปรากฏที่ไหนเลย → ยอดบัญชีต่ำกว่าจริง. ต้องเพิ่มมัดจำที่รับเข้าต่อวัน-ต่อช่องทาง, แยกคอลัมน์ ยอดขาย / มัดจำ / ยอดรวม, ทำเลขใบเสร็จกดดูรายการได้, และที่ reports-sale แสดง "มัดจำที่รับเข้า" (ไม่รวมยอดขาย) + "ยอดมัดจำคงเหลือในระบบ".

✅ ไม่นับซ้ำ (พิสูจน์จากโค้ดจริง — SaleTab.jsx:510-515): ในการขาย มัดจำถูกหักออก ก่อน สร้างช่องทางชำระ (afterDeposit = afterMembership − depositApplied → netTotal → channels[0].amount = netTotal) → payment.channels[] ไม่เคยมีส่วนมัดจำ. การเพิ่ม "มัดจำที่รับเข้า" จึงเป็นเงินคนละก้อน 100%. มัดจำนับตาม paymentDate + ช่องทาง paymentChannel.

Decisions (locked via Q&A)

  • Q1 = A: คอลัมน์ "มัดจำ" = gross รับเข้าจริง — Σ deposit.amount ที่ paymentDate ในช่วง, จัดตาม paymentChannel; ตัด status==='cancelled' ออก; ไม่หัก refund. ยอดคืนมัดจำ (refund ในช่วง) แสดงเป็น footnote เล็กแยก.
  • Q2 = A: เลขใบเสร็จกดได้ → modal รายการเอกสารของช่องทางนั้น = ใบขาย + ใบมัดจำ. ใบขาย→ SaleDetailModal เดิม; ใบมัดจำ→ เด้งหน้ามัดจำ.
  • Q3 = A: reports-sale แสดง ส่วน "มัดจำที่รับเข้า" แยกต่างหาก (ไม่รวม footer ยอดขาย).
  • Q4 = A: เด้งหน้ามัดจำ = window.open(?backend=1&tab=finance&subtab=deposit&deposit=DEP-ID, '_blank') → BackendDashboard เปิด finance·มัดจำ + DepositPanel เปิด DetailModal ของใบนั้น.
  • Auto-locked: ยอดมัดจำคงเหลือในระบบ = Σ remainingAmount ของ status ∈ {active, partial} (V154), branch-scoped, แสดงเป็นชิปบน header reports-sale. ดึง bar "สัดส่วน" ออกจาก reports-payment เพื่อให้ตารางโล่ง. นับมัดจำทุกสถานะที่เงินเข้าจริง (active/partial/used/refunded/expired) ยกเว้น cancelled.

  createdAt, updatedAt }

Acceptance Criteria

  1. reports-payment shows per-channel ยอดขาย / มัดจำ / ยอดรวม; grand total = Σ(sale channels) + Σ(deposit received, non-cancelled) for branch+range.
  2. มัดจำ column = gross deposit.amount by paymentChannel/paymentDate; cancelled excluded; refunds NOT subtracted; refund footnote present.
  3. ใบเสร็จ number clickable → lists both sales + deposits of that channel; sale→SaleDetailModal; deposit→new-tab deposit page.
  4. reports-sale shows "มัดจำที่รับเข้า" section (date-range), excluded from sale footer; each row → new-tab deposit page.
  5. reports-sale shows "มัดจำคงเหลือในระบบ" chip = Σ remainingAmount of active/partial deposits (branch).
  6. Jumping to a deposit opens finance·deposit tab with that deposit's DetailModal open.
  7. Branch switch re-fetches both tabs (BS-11); CSV export includes new columns; footer reconciles (audit-reports-accuracy); cancelled excluded.
  8. No double-count: a deposit applied to a sale appears once (as deposit-received), never inflates sale channels — locked by an AV invariant + flow-simulate test.

Verification (Rule Q — must pass before "done")

  • Unit/flow: aggregator no-double-count + reconcile + gross-no-refund + cancelled-excluded + union-methods + drill-down docs; loader; depositReportUtils; reports-sale received+remaining; source-grep + Rule I flow-simulate.
  • Rule Q L2: real-prod admin-SDK diag — pull a real branch's sales+deposits in a range, run the aggregator, assert grand total = sale-channels + deposit-received and that no มัดจำ channel exists inside sale.payment.channels (double-count guard).
  • Rule Q L1: preview both tabs on the running dev server (real Firestore) — see the new columns/section, click ใบเสร็จ → modal, click a deposit → deep-link opens DetailModal.
