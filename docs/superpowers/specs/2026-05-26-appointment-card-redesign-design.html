Appointment Card Redesign — Design Spec

Date: 2026-05-26 EOD+6 · Status: ✅ Approved by user — live mockup verified in Browser 1 (Dark + Light), Rule Q-vis · Scope: Q1=C full layout redesign · Q2=A theme-matched pills · Q3=A refined Editorial Ember · Q4=A stepper untouched (re-position only) · Constraint: COSMETIC-SHELL — zero changes to any button wiring / flow / logic / data-testid / conditional.
Theme: ← flip to see BOTH themes (this is the whole point)

Purpose / Problem

The appointment-hub card list (AppointmentHubRowCard.jsx) looks unrefined — especially in Light theme — and the OPD LIFECYCLE strip is visibly broken: its badges (OpdLifecycleRow.jsx) use unconditional dark-only Tailwind classes (bg-emerald-900/30 text-emerald-300, bg-blue-900/30 text-blue-300, bg-slate-800/50 text-slate-500) with no light base and no [data-theme="light"] override. The app theme is driven by the data-theme attribute (NOT Tailwind dark: / OS), so in Light theme those badges render light-green text on a translucent dark-green background → unreadable green-on-green (the screenshot). Goal: a genuinely beautiful, theme-correct card in both Dark + Light — without touching any button's behavior.

Flow

Band architecture — the card reads top-to-bottom instead of three cramped columns.

① HEADER BAND — HN · membership/ready chips · name (sky-blue, never red) · phone  |  right: LINE badge + status pill
② FINANCE STRIP — wallet · มัดจำ · ค่างชำระ · ยอดสั่งซื้อ (renders only if any present — conditional preserved)
③ DETAIL ZONE (2-col → 1-col mobile) — date + time chip + type/deposit/missed chips  |  ที่ปรึกษา/แพทย์/ผู้ช่วย/ห้อง + นัดมาเพื่อ
④ OPD FOOTER ZONE — OPD lifecycle pills (state A–E) + สถานะ OPD stepper, in one delineated warm-tinted footer
⑤ ACTION BAR — confirm / record-treatment / mark-complete (ember) · edit (sky) · cancel (rose) — every existing button + handler + data-testid, just relocated to a bottom bar

Architecture

  • src/components/admin/_apptHubStyles.js — add a new OPD_PILL token family (blue / emerald / slate-wait / rose-save) using the proven dual-tone pattern, mirroring STATUS_CHIP_CLS. Rule of 3: the OPD pills are a badge family → centralize here (same home as the other shared card tokens).
  • src/components/admin/OpdLifecycleRow.jsx — swap the 4 hardcoded dark-only class strings for the new theme-aware tokens. All button TEXT (labels unchanged), predicates, props, onClick handlers, data-testid, disabled state, and the State A–E logic stay byte-for-byte. Note: the new tokens mirror the card's existing STATUS_CHIP_CLS / TYPE_CHIP_CLS pattern (light base + dark: override) so the pills render identically to their neighbor chips. (Tailwind dark: here is OS-coupled — no darkMode config — but matching the neighbors is the correct, consistent fix; it resolves the reported Light-theme bug and a systemic dark:→data-theme migration is explicitly out of scope.)
  • src/components/admin/AppointmentHubRowCard.jsx — re-lay-out the JSX into the 5 bands above. Every interactive element, every conditional (hasTreatmentForDay, rawStatus, isPastDate, showMarkCompleteBtn, …), every handler prop, and every data-testid is preserved verbatim — only the wrapping markup + classNames move. AppointmentOpdStepperRow + TreatmentLifecycleStepper (the round-circle สถานะ OPD stepper: ซักประวัติ→แพทย์ลงบันทึก→เสร็จ) are OFF-LIMITS — NOT edited, NOT restyled (เด็ดขาด, user directive 2026-05-26 EOD+6); only re-parented verbatim into the new footer band. AppointmentLineBadge likewise re-parented unchanged.
  • No changes to opdSessionState.js, appointmentHubFilters.js, AppointmentHubView.jsx data flow, or any Firestore / endpoint / cron. Pure presentation.

Decisions (locked via Q&A)

  • Q1 = C — full layout redesign (band architecture), not a polish of the 3-column layout.
  • Q2 = A — OPD lifecycle states render as theme-matched dual-tone pills (blue / emerald / slate-wait / rose-save CTA).
  • Q3 = A — Refined Editorial Ember: warm amber/orange accent unified across Dark + Light; Light = crisp white card + hairline slate border + soft shadow + warm hover.
  • Q4 = A (locked 2026-05-26 EOD+6) — the round-circle สถานะ OPD stepper is left completely untouched — ONLY re-positioned into the new footer band. NOT even a color change. Rationale: TreatmentLifecycleStepper is a shared Phase 28 component also rendered by the Backend treatment-history view (CustomerDetailView); any color tweak would propagate there too. It already uses theme-aware CSS vars (var(--bg-base) / var(--bd-strong) / var(--tx-faint)) for its muted state + gradient+white for done states, so it renders correctly in both themes as-is. Verbatim re-parent only.
  • Amendment 2026-05-26 EOD+6: redesign EVERY part of the card EXCEPT the round-circle OPD-tracking stepper — the สถานะ OPD / TreatmentLifecycleStepper dots (ซักประวัติ → แพทย์ลงบันทึก → เสร็จ) on the วันนี้/Today tab are OFF-LIMITS (เด็ดขาด): re-positioned verbatim, never restyled or re-implemented. No text/wording changes anywhere — every button label, chip label, and field label stays byte-for-byte (this is a pure visual/layout redesign, not a content change).
  • Hard constraints: (a) cosmetic-shell — never touch wiring/flow/logic/handlers/conditionals; (b) every existing data-testid preserved (tests depend on them); (c) patient name stays sky-blue, never red (Thai-culture iron-clad); (d) theme via data-theme dual-tone, never dark-only classes.

Acceptance Criteria

  1. OPD lifecycle pills are AA-readable in both Dark and Light (no green-on-green / washed states).
  2. Card renders cleanly in both themes + mobile (≤620px collapses detail zone to 1 col, no overflow).
  3. Every button present in the old card is present in the new one, with the same data-testid, same handler, same show/hide condition.
  4. Patient name never red.
  5. Full vitest green (with V21 fixups for any layout-assert tests); build clean; /audit-ui-cultural-a11y + /audit-react-patterns pass.
  6. Rule Q-vis: rendered pixels verified by eye in a real browser (Dark + Light) — feasible here because the AdminDashboard is served by the local vite dev server (unlike the serverless patient-view).

Risks / Open Questions

  • Test churn (expected): the Scope-C layout change will break RTL/source-grep tests that assert the old 3-column structure (e.g. v118-*). These get V21 fixups — re-anchored on the preserved data-testids, not on layout. No real regression if every testid + handler is preserved.
  • Theme mechanism: the card already uses Tailwind dark: (OS-coupled) for its working chips; the new OPD pills follow the same proven dual-tone so they render identically to the neighbors. A systemic migration off dark: to isDark is out of scope (would touch the whole hub).
  • Mockup vs React: this HTML mockup hand-renders the design with plain CSS; the implementation maps it to the app's real Tailwind classes + CSS vars. Final sign-off is the real-browser pixel check (Rule Q-vis), not this mockup.
