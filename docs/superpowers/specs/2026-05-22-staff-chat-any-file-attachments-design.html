Staff Chat — Any-File Attachments (≤1 GB) Design Spec

Date: 2026-05-22 · Status: Approved by user (Q1–Q4 locked) · Base: V73 staff chat + 2026-05-22 multi-image · Implementation plan: docs/superpowers/plans/2026-05-22-staff-chat-any-file-attachments.html

Purpose / Problem

Staff chat today accepts images only (jpeg/png/webp/gif, ≤50 MB, thumbnail+original, ≤10/msg, 30-day prefix-sweep retention, grid+lightbox). Staff need to send every other kind of file too — PDFs, Office docs, video, audio, zip — up to 1 GB each (user: "ส่งไฟล์ได้ทุกประเภทขนาดไม่เกิน 1 GB … มีไฟล์อื่นๆต้องส่งอีกเยอะที่ไม่ใช่แค่รูปภาพ"). This spec extends the existing attachment pipeline (no reinvent) to mixed any-file attachments, with rich rendering (inline video/audio, PDF preview, download cards) while keeping the proven storage/retention/no-orphan design. Two pending image-polish issues (lightbox backdrop-close + grid overflow) are folded in.

Decisions (locked via Q&A with user)

  • Q1 = allow-all: accept ANY file type, no restriction (the staff-chat Storage path already does not gate content-type; only client validation changes).
  • Q2 = cards + inline media: non-image files render as cards (icon+name+size+download); video/audio play inline in the bubble; PDF gets a preview overlay; everything else = download card. Images keep grid+lightbox.
  • Q3 = progress + cancel + retry: resumable per-file upload with % progress, a cancel button, and retry-on-failure (essential for 1 GB on slow networks).
  • Q4 = split caps: image/* ≤ 50 MB (keep), all other files ≤ 1 GB. ≤10 attachments/message, 30-day retention — unchanged.

Flow

Flow 1 · Send (pick → validate → resumable upload → setDoc → render-by-kind)
1. pick / paste / drop N files→ 2. validate each
type = allow-all
image/* <50MB · else <1GB→ 3. preview chip (thumb|icon)
+ ✕ remove→ 4. Send: mint messageId
upload each (resumable)
5a. image → thumbnail + original
5b. other → original only→ 6. per-file %/cancel/retry→ 7. build attachments[]
(+ name, kind-derived)→ 8. setDoc be_staff_chat_messages→ 9. render by attachmentKindFor(mime)

Upload happens BEFORE setDoc (existing flow) → a cancel during step 6 just drops that file from the pending set; the doc is only written once the kept files finish. Files live under the per-message folder staff-chat-attachments/{branchId}/{messageId}/.

Flow 2 · Retention (daily cron — UNCHANGED, already file-type-agnostic)
PassConditionAction
A · age-outmessage doc createdAt > 30 daysdelete doc + prefix-sweep {branchId}/{messageId}/ (every file in the folder — image OR not) + legacy attachmentUrl via extractStoragePathFromUrl
B · orphanfolder exists, no message doc, age > 1-day gracedelete the orphan folder

Per-message folder ⇒ prefix-sweep deletes everything with no orphan ("ลบเกลี้ยง") regardless of file type. No retention change needed — the 1 GB video in a folder is swept exactly like a thumbnail. Admin-SDK-only delete (client update,delete: if false).

Flow 3 · Size-bracket + render-kind decision (mime → cap → render)
mimeTypesize cap (client + storage.rules)thumbnail?render kind
image/jpeg · png · webp · gif< 50 MByes (≤512px jpeg)image — grid + lightbox
video/*< 1 GBnovideo — inline <video>
audio/*< 1 GBnoaudio — inline <audio>
application/pdf< 1 GBnopdf — card + 👁 overlay + ⬇
image/heic · image/svg+xml · everything elseimage/* < 50 MB · else < 1 GBnofile — icon card + ⬇

Storage rule keys the bracket on request.resource.contentType.matches('image/.*'). Render-kind ("image" grid) is limited to the 4 browser-renderable image types so a HEIC/SVG never produces a broken <img> — it falls to a download card.

Architecture

Storage

Path unchanged: staff-chat-attachments/{branchId}/{messageId}/{fileId}-o.{ext} (images additionally -t.jpg). storage.rules raises the cap with a split:

match /staff-chat-attachments/{branchId}/{file=**} {
  allow read: if isClinicStaff();
  allow create: if isClinicStaff() && (
       (request.resource.contentType.matches('image/.*')  && request.resource.size < 50 * 1024 * 1024)
    || (!request.resource.contentType.matches('image/.*') && request.resource.size < 1024 * 1024 * 1024)
  );
  allow update, delete: if false;   // retention via admin SDK (bypasses rules)
}

Firestore

be_staff_chat_messages rule: attachments[] already allowed (≤10); extend the per-item validation to accept the new name (string) field. buildMessageDoc.normalizeStaffChatAttachment adds name (string, required for non-image) and keeps thumb/w/h optional (Firestore-undefined-safe, V14). Probe-Deploy-Probe (#9): anon create → 403.

Upload pipeline (staffChatImageResize.js — extend, do not fork)

  • attachmentKindFor(mime) — pure: image|video|audio|pdf|file (shared via staffChatRetentionCore.js so cron/CLI/components agree — Rule of 3).
  • validateStaffChatFile(file) — allow-all type; image/* < 50 MB else < 1 GB; Thai messages. (validateStaffChatImage kept for the image fast-path / back-compat.)
  • uploadStaffChatFile({file, branchId, messageId, onProgress, taskRef}) — renderable-image → thumbnail + resumable original; else → resumable original only (no thumb). Returns the attachment record incl. name = file.name. Exposes the uploadBytesResumable task via taskRef so the composer can .cancel() (Q3). preload="metadata" is a render concern (component), not upload.

Components

  • StaffChatComposer — accept removed (any file); preview strip: image thumb OR icon-chip (filename + progress + ✕ cancel); retry button on a failed chip; per-file task cancel.
  • StaffChatMessage — split attachments[] by attachmentKindFor: images → existing AttachmentGrid (now w-full max-w, polish #2) → image lightbox; video/audio → inline players; pdf/file → StaffChatAttachmentCard.
  • NEW StaffChatAttachmentCard.jsx — icon-by-category + name (truncate) + human size + ⬇ download (anchor with download={name}); pdf adds 👁 → opens the PDF overlay viewer.
  • StaffChatImageLightbox — (polish #1) remove backdrop onClick={onClose}; close via ✕ + Esc only. Add a sibling PDF overlay with the same close discipline + an <iframe> body + ⬇.

Folded-in polish (the 2 pending image issues)

New PDF overlay follows the same rule from day one.
  2. Grid fits the bubble: AttachmentGrid + cards use width:100%; max-width:240px so attachments never overflow the bubble's max-w-[80%].

Acceptance Criteria

  1. One message can mix images + a PDF + a video + a zip; each renders by its kind; order preserved.
  2. Video & audio play inline in the bubble; preload="metadata" means a 1 GB video does NOT download until play.
  3. PDF 👁 opens a preview overlay (iframe) with ⬇ + ✕; backdrop click does NOT close it.
  4. Non-image/non-media files show an icon card; ⬇ downloads with the original filename.
  5. A 1 GB file uploads with visible % progress; cancel aborts it; a failed upload offers retry.
  6. Size caps: a 60 MB image is rejected; a 60 MB PDF/video is accepted; a >1 GB file is rejected — enforced client-side AND by storage.rules.
  7. Retention sweep deletes the whole message + every file in its folder (no orphan), verified with a non-image file present.
  8. Image lightbox ignores backdrop click (✕/Esc only); image grid + cards never overflow the chat bubble.
  9. Legacy image messages (attachmentUrl scalar + old image attachments[]) still render unchanged.

Risks / Open Questions

  • 1 GB Firestore doc — N/A: only the Storage URL + metadata are stored on the doc, never the bytes.
  • Huge-image thumbnail decode — a very large/exotic image may fail makeStaffChatThumbnail; fall back to a file card (and/or raw <img>) rather than throwing.
  • firestore.rules name field — must be accepted by the per-item validation; covered by Probe-Deploy-Probe on the rules deploy.
  • PDF overlay vs new tab — chosen: in-app <iframe> overlay (consistent with the lightbox + AV78 close discipline) with a download fallback if the browser can't render it inline.
  • Cancel mid-upload cleanup — task.cancel() aborts before the object commits; if a thumbnail uploaded but the original was cancelled, the partial thumbnail is an orphan within the message folder and is swept by the same retention prefix-sweep (or never written since setDoc only happens after success).
  • Verification (Rule Q) — done at build/test time only (real client SDK L2 + Chrome MCP screenshots for the rendered bubble/players/overlay), NOT during this spec/plan stage.
