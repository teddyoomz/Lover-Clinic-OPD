Purpose / Problem

หน้าการขาย (tab=sales, SaleTab.jsx) ตารางขายตอนนี้แสดงเฉพาะ ยอดสุทธิ (column label เดิม “ยอดรวม”, ค่าจริง = billing.netTotal) — มองไม่ออกทันทีว่าแต่ละใบ เก็บเงินได้จริงเท่าไหร่ (จ่ายครบ / บางส่วน / ยังไม่จ่าย). และตารางเลื่อนยาวทั้งหมดไม่มีแบ่งหน้า (48+ รายการในภาพ) อ่านยาก. เพิ่ม:

  1. คอลัมน์ “ยอดชำระจริง” ข้าง ๆ ยอดสุทธิ — เงินที่รับจริง พร้อมสีบ่งสถานะการเก็บเงิน.
  2. แบ่งหน้า 30 รายการ/หน้า + ปุ่มเปลี่ยนหน้าด้านล่าง.
  3. สวย + เห็นชัดทั้ง 2 ตีม (Dark / Light).

Flow

Render flow (ไม่มี fetch/write ใหม่ — paginate + 1 คอลัมน์ จากข้อมูลที่โหลดอยู่แล้ว)

listener/loadSales → sales[]→ filtered = subTab + search + status→ usePagination(filtered,{pageSize:30, key})→ visibleItems (≤30)

└→ ต่อแถว: netTotal = sale.billing.netTotal paid = resolveSalePaidAmount(sale)→ render 2 cells (net + paid·สีตาม paid-vs-net)

<Pagination page totalPages onPageChange totalCount /> → auto-hide เมื่อ ≤1 หน้า · filter เปลี่ยน → key เปลี่ยน → page reset = 1

Helper flow — resolveSalePaidAmount(sale)

sale.payment.channels[] มี? → ใช่ → Σ parseFloat(c.amount) (round 2dp)

→ ไม่/ว่าง → sale.totalPaidAmount (legacy fallback) → ไม่มี → 0

Decisions (locked via Q&A with user)

  • Q1 = A: “ยอดชำระจริง” = เงินที่รับจริง = Σ payment.channels[].amount (= totalPaidAmount). มัดจำ/wallet ที่หักก่อน netTotal ไม่นับซ้ำ (ไม่สร้าง gap ปลอม).
  • Q2 = B: ตัวเลขยอดชำระจริงทำสีตามสถานะ (เขียว/เหลืองอำพัน/เทา) อย่างเดียว — ไม่มีบรรทัด “ค้าง ฿X”.
  • Q3 = rename: หัวคอลัมน์ 'ยอดรวม' → 'ยอดสุทธิ' (ตรงค่า netTotal + ตรงที่ user เรียก).
  • Pagination: 30/หน้า, reuse canonical usePagination+Pagination.

Acceptance Criteria

  1. Header ตารางอ่านว่า … ยอดสุทธิ · ยอดชำระจริง · สถานะ … (rename + คอลัมน์ใหม่หลังยอดสุทธิ).
  2. cell ยอดชำระจริง = fmtMoney(resolveSalePaidAmount(sale)) + “ ฿”.
  3. สี: paid+0.01 ≥ net → เขียว (รวม 0฿ ที่จ่ายครบ) · 0 < paid < net → เหลืองอำพัน · paid == 0 && net > 0 → เทา. legible AA ทั้ง 2 ตีม; ไม่ใช้แดง.
  4. >30 รายการ → แสดง 30 + pager “หน้า X / Y · N รายการ”; ≤30 → ไม่มี pager.
  5. เปลี่ยน search / status / sub-tab / branch → reset page = 1.
  6. resolveSalePaidAmount ทน adversarial: null/undefined sale, channels ว่าง→totalPaidAmount, ทั้งคู่ไม่มี→0, amount เป็น string/ติดลบ, over-paid (paid>net → ยังเขียว).
  7. ไม่เปลี่ยน flow ขาย/แก้/รับชำระ/ยกเลิก (cosmetic-shell); ไม่มี fetch/write ใหม่; ทุก handler/state/testid เดิมคงอยู่.

Risks / Open Questions

  • Data-shape (Rule R diag ตอน plan): ยืนยันบน prod ว่าใบที่ ชำระแล้ว มี payment.channels จริง (หรือใช้ totalPaidAmount) → lock ลำดับ fallback ให้ตรง. OPD auto-sale / exchange / share shape ต้องเช็คด้วย.
  • ใบยกเลิก: ยอดชำระจริง = ยอดที่เคยรับ (refund บันทึกแยก) → ยอมรับได้; cancelled sub-tab แยกอยู่แล้ว.
  • 0฿ OPD sale: net 0 / paid 0 → เขียว “ชำระแล้ว” (ถูกต้อง).
  • Rule Q gap: assembled L1 บน SaleTab จริง (admin-gated) = USER post-deploy; logic+pixels คุมด้วย unit/RTL/source-grep/flow-simulate + mockup นี้.
