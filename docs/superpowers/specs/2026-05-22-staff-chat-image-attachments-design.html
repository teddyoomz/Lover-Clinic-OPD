Staff Chat · Multi-Image Attachments — Design Spec

Date: 2026-05-22 · Status: Approved by user (design + visual layout) · Builds on: V73 staff chat (be_staff_chat_messages) · Implementation plan: (TBD — writing-plans next)

Purpose / Problem

แชทภายในของทีม (V73 staff chat) ส่งรูปได้แค่ รูปเดียวต่อข้อความ และ resize เหลือ ~1024px เสมอ (cap 1MB). ทีมอยากส่ง หลายรูปพร้อมกัน, ดูรูปย่อในแชทแล้วกดเปิดรูปใหญ่เลื่อนซ้าย/ขวาดูทั้งชุด, รับรูปได้ ≤50MB/รูป. และเพราะ staff chat ปัจจุบัน ไม่มีระบบลบประวัติเลย (Firestore + Storage append-only ถาวร) — ฟีเจอร์นี้ต้องสร้าง auto-retention ที่ลบทั้งข้อความ + ไฟล์รูปใน Storage ให้ เกลี้ยง ไม่เหลือ orphan (กันค่า storage บาน).

Decisions (locked via Q&A with user)

  • Q1 = Auto-retention only: ไม่มีปุ่ม manual; cron ลบอัตโนมัติ (เป็นตัวคุม storage หลัก)
  • Q2 = Hybrid: thumbnail (โชว์ในแชท/grid) + original ≤50MB (กดดูใน lightbox) — 2 ไฟล์ต่อรูป
  • Q3 = ลบทั้งข้อความ + รูป: retention ลบทั้ง doc (text) + thumb + original (ไม่ใช่ลบเฉพาะรูป)
  • Q4 = 30 วัน: retention period
  • Q5 = ≤10 รูป/ข้อความ
  • Layout approved: adaptive grid (สไตล์ LINE) + swipe lightbox (mockup ด้านบน)

Acceptance Criteria

  1. ส่ง 1–10 รูปใน 1 ข้อความได้ (pick / paste / drag); >10 หรือ >50MB หรือ type ไม่รองรับ → reject + ข้อความไทย
  2. แชทโชว์ adaptive grid (1/2/3/4/5+→"+N") จาก thumbnail (โหลดเร็ว)
  3. กดรูป → lightbox เปิดที่ใบนั้น, เลื่อน ‹ › / ปัด / ←→ ดูทั้งชุดได้, counter ถูก, ⤓บันทึกได้, ✕/Esc/แตะนอกปิด
  4. ข้อความเก่า (legacy attachmentUrl รูปเดียว) ยัง render ได้ปกติ (backward-compat)
  5. retention: ข้อความ > 30 วัน → cron ลบ doc + thumb + original; bucket.getFiles({prefix}) = 0 ไฟล์ บน prod จริง (Rule Q L2)
  6. orphan: upload-แล้ว-ส่งไม่สำเร็จ (folder ไร้ doc) เก่ากว่า ~1 วัน → Pass B ลบทิ้ง; ยืนยันด้วยการ list prod = 0
  7. Probe-Deploy-Probe: Probe #9 (firestore) + #10 (storage 50MB) ผ่านทั้ง pre/post
  8. vitest เขียว + build clean; AV108 + Rule I flow-simulate ครบ

Risks / Open Questions

  • HEIC (iPhone): reject + ข้อความ "แปลงเป็น JPEG ก่อน" (กัน lightbox โชว์ไม่ขึ้น). future: แปลง HEIC→JPEG ฝั่ง client. ยืนยัน: reject ก่อน (ไม่ทำ convert ในรอบนี้)
  • 500MB/ข้อความ: 10×50MB = สูงสุด ~500MB upload ต่อข้อความ — resumable รับได้แต่ช้าบนเน็ตช้า; progress + ยกเลิกรายใบช่วยได้
  • canvas memory: สร้าง thumb จากรูปใหญ่มาก (เช่น 100MP) อาจกิน RAM — ลด canvas ทันทีหลังได้ blob; ทดสอบ adversarial รูปใหญ่
  • original เก็บ as-is: รักษาคุณภาพ/ความคม/โปร่งใส (PNG) — แต่ไม่ re-encode → ขึ้นกับ type ที่รองรับ (jpeg/png/webp/gif)
  • cron pagination: ถ้าค้างนานแล้วข้อความเก่าเยอะ batch แรกอาจใหญ่ — paginate + limit ต่อ run
  • legacy scalar cleanup: รูปเดิม (path {branchId}/{ts}-{hex}.jpg ไม่อยู่ใต้ {messageId}/) → Pass A ถอด path จาก attachmentUrl มาลบ; ทดสอบเคสนี้แยก
  • retention เริ่มนับ: หลัง deploy cron จะลบข้อความ >30 วันที่มีอยู่แล้วทันทีรอบแรก — แจ้ง user (เป็นพฤติกรรมที่ตั้งใจ)
