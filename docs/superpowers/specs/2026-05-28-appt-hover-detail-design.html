Appointment Hover-Detail Peek Design Spec

Date: 2026-05-28 (EOD+2) · Status: Approved by user (Q1=A, Q2=A, Q3=A) · Plan: docs/superpowers/plans/2026-05-28-appt-hover-detail.html (to follow)

Purpose / Problem

In the backend appointment calendars/tables, seeing an appointment's full details today requires clicking the card (opens the centered AppointmentDetailPopover modal). The user wants hovering a card to show ALL details too — "ไม่จำเป็นต้องคลิ๊กเข้าไปอย่างเดียว" — across every appointment table in every tab, and it must be clearly readable + on-brand in both light and dark themes.

An AppointmentDetailPopover (click-opened, centered modal) already exists and is theme-aware (CSS vars + appointmentDisplay status SSOT). This feature adds an anchored hover "peek" card that reuses the same detail fields, leaving the click→modal flow untouched.

Flow

Hover (desktop mouse) → peek · Click/tap → existing modal (unchanged) · Touch → no hover
TriggerWhat happens
pointerenter with pointerType === 'mouse' on a cardstart ~150ms hover-intent timer
timer fires (still hovering)read card getBoundingClientRect() → portal-render <AppointmentHoverPeek> anchored beside the card (flip to stay on-screen); NO backdrop dim
pointerleave the cardclear timer + dismiss peek (short ~80ms grace to avoid flicker)
scroll / window resize while opendismiss peek (re-hover to re-open — simplest, no reposition churn)
click the card (any device)existing onClick → opens AppointmentDetailPopover modal (with แก้ไข). UNCHANGED.
touch tap (iPad — no hover)pointerType !== 'mouse' → peek never fires → tap = click → modal (today's behavior)

Architecture

DRY via a shared detail body (avoids modal/peek field drift — the V12 multi-reader lesson):

  • NEW src/components/backend/AppointmentDetailBody.jsx — extract the read-only field block currently inline in AppointmentDetailPopover (name, HN·time·room meta, phone, service/appointmentTo, doctor+assistants, status) AND ADD the Q3 extras when present: appointment type chip, deposit (มัดจำ), recurring (นัดซ้ำ). Takes a variant: 'modal' | 'peek' prop — modal renders phone as a tappable PhoneLink; peek renders phone as plain bold text (read-only) + a faint "คลิกเพื่อแก้ไข" hint. Same fields, same theme classes (CSS vars + appointmentDisplay status SSOT + V124 AA text tokens).
  • AppointmentDetailPopover.jsx — refactored to render <AppointmentDetailBody variant="modal" /> inside its existing modal chrome (backdrop + close + แก้ไข). Behavior identical; now also shows the extras (consistent with the peek).
  • NEW src/components/backend/AppointmentHoverPeek.jsx — anchored, createPortal(..., document.body) (AV98 — escape transformed ancestors + calendar overflow), positioned from the anchor rect (prefer right/below; flip if off-screen), NO backdrop, renders <AppointmentDetailBody variant="peek" />. Themed via bg-[var(--bg-card)] border-[var(--bd)] → both themes automatically.
  • NEW src/hooks/useApptHoverPeek.js — shared hover-intent controller (open delay ~150ms, close grace ~80ms, desktop-only pointerType === 'mouse' guard, tracks {appt, rect}). Returns getHoverProps(appt) → { onPointerEnter, onPointerLeave } + the peek state. One controller per surface so behavior is identical (Rule of 3).

Decisions (locked via Q&A with user)

  • Q1 = A: Anchored peek-card on hover (not hover-opens-the-centered-modal, not enriched native title).
  • Q2 = A: Keep the click→modal flow exactly; peek is a desktop-hover-only enhancement. Touch (iPad) has no hover → tap opens the modal as today (graceful — peek guarded to pointerType === 'mouse').
  • Q3 = A: Peek shows ALL present fields — the modal's set + appointment type / deposit / recurring WHEN present (omit when empty, no noise). The shared body means the modal gains the same extras (consistent).
  • Theme (user emphasis): reuse the proven CSS-var + appointmentDisplay status SSOT + V124 AA text-token styling → readable + on-brand in BOTH themes. Verified by a live light+dark scan.
  • Peek size = XL: ~345px wide · heading 17px · field 14px · sub 13px · status/type chips 12px · padding 20px · radius 14px (user chose XL from a live M/L/XL real-size comparison). Near a viewport edge the peek flips/clamps to stay fully on-screen.

Acceptance Criteria

  1. Hovering an appointment card with a MOUSE (desktop) on each surface (Calendar grid, Agenda, Hub rows) shows the anchored peek with all present fields after ~150ms; moving the mouse away dismisses it; no flicker between adjacent cards.
  2. Click/tap behavior is UNCHANGED on every device — still opens AppointmentDetailPopover with the แก้ไข action; touch never triggers the peek (no double-surface).
  3. The peek renders correctly + readably in BOTH light and dark themes (live scan: text ≥ AA on the card bg; status/type chips on-brand; no invisible text) — and the modal still does too after the body extraction.
  4. The peek is portal-rendered (not clipped by the calendar's overflow or a transformed ancestor) and flips to stay on-screen near a viewport edge.
  5. Tests: source-grep (peek uses shared body + portal + pointerType=mouse guard; all 3 surfaces wired) + an RTL/flow test (mouse pointerenter → peek appears with fields; pointerleave → dismiss; touch pointerenter → no peek; click → modal) + NEW AVxx invariant. npm run build clean + full vitest green.

Risks / Open Questions

  • Flicker between adjacent cells: the ~80ms close-grace + open-delay tuning addresses it; verify live on the dense day-grid.
  • Body extraction touches the proven modal: refactor must be behavior-preserving (same fields/testids) + the existing modal tests must stay green; the extras are additive (render only when present).
  • Tablet (touch) gets no peek: intended (no hover on touch) — tap→modal remains the path. Not a regression.
  • Existing modal uses dark: variants (e.g. text-red-300) which are OS-coupled, but V124's [data-theme] !important overrides win → AA in both themes. The shared body inherits this; the live scan confirms.
