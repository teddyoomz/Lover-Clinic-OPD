Staff Chat — Word/Excel/PowerPoint Preview (server-side PDF conversion)

Date: 2026-05-22 (EOD+2) · Status: Approved by user (Q1-Q4 locked) · Implementation plan: [docs/superpowers/plans/2026-05-22-staff-chat-office-preview.html — pending writing-plans]

Purpose / Problem

The prior in-chat Office preview attempt (commit e6a6f9d9, reverted 2026-05-22) used mammoth.js (DOCX→HTML) and xlsx (XLSX→HTML) client-side renderers, plus a fallback to Microsoft Office Online Viewer iframe. User rejection (verbatim, this turn): "Render Word ออกมาแย่มาก อย่างกับเอา notepad มาเปิด". The iframe path was blocked separately because (a) Firebase signed URLs are not publicly fetchable by 3rd-party viewers and (b) transmitting PHI-bearing files off-site to MS / Google viewers violates clinic privacy guarantees. AV108 was locked at that revert: "no 3rd-party doc viewer in staff-chat". The current production state: any Office file in chat shows a download-only card (📝 icon + ⬇ button) — no preview affordance.

User mandate for this round (verbatim): "Research หา หรือ Research สร้าง Tools ที่ใช้สำหรับ Render ไฟล์ Word และ Excel เพื่อใข้ Preview ไฟล์ต่างๆ ในโปรเจ็คของเราได้ โดยเฉพาะในช่องแชท ... หวังว่ารอบนี้จะไม่ขี้เกียจและเข้าข้างตัวเองไปเอาของแย่ๆแบบนั้นมาใส่อีก". The fidelity bar is pixel-perfect to the source document — fonts, page layout, tables, embedded images, charts, page breaks all preserved. Client-side semantic-HTML renderers cannot meet this bar by design. The only architecture that does, while preserving PHI privacy, is server-side conversion to PDF inside our own Firebase project, then reusing the existing PDF iframe preview.

Decisions (locked via Q&A with user)

  • Q1 = A — Pixel-perfect: server-side Office→PDF conversion; client reuses existing PDF iframe overlay. Output IS the document (LibreOffice rendering, not a parsed-and-restyled approximation).
  • Q2 = C — Hybrid (background convert on upload + 30d cleanup via V73 retention): Cloud Function triggers on Storage object create, conversion runs in parallel with the message send (no sender wait). Receiver shows a spinner if they open the file before conversion finishes; the listener flips it to 👁 the moment the doc patches. PDF lives next to the original under the same prefix and is auto-cleaned by the existing V73 30-day staff-chat retention sweep — zero new cron.
  • Q3 = C — Word + Excel + PowerPoint + CSV: MIME whitelist gate covers .doc / .docx / .xls / .xlsx / .ppt / .pptx / .csv. LibreOffice supports more (.odt/.ods/.rtf/.pages/.numbers) but they fall through to the regular download card with pdfPreviewStatus: 'unsupported'.
  • Q4 = B — Failure UX: card + ⚠ + tooltip: when LibreOffice cannot convert (corrupt file, password-protected, weird legacy format), the attachment card renders normally with the ⬇ button working, but the 👁 button is replaced by a ⚠ icon with a Thai-language tooltip explaining the reason from the server. The user can still download and open locally.

End-to-end data flow from upload to preview-ready. The Cloud Function is fire-and-forget — the message send path never waits on conversion.

Architecture

Layer 1 — Cloud Function 2nd Gen (Docker, server-side)

  • Location: functions/officeToPdf/ (new directory inside existing functions/ root)
  • Base image: gotenberg/gotenberg:8 — pre-built, Apache-2.0, current as of 2025-09. Bundles LibreOffice 24.x + Chromium. Stateless HTTP server on port 3000 with endpoint /forms/libreoffice/convert.
  • Trigger: Storage onFinalize event on the bucket, scoped to objects under staff-chat-attachments/. Fire-and-forget (no client wait).
  • MIME whitelist: pure JS constant OFFICE_CONVERTIBLE_MIMES (Set) reused from a new src/lib/staffChatOfficePreviewCore.js module — same source-of-truth used by client (to set initial pending) and server (to filter). Rule of 3 alignment.
  • Conversion call: HTTP POST multipart to the bundled Gotenberg → PDF bytes back → upload to Storage at {originalPath}.pdf.
  • Firestore patch: admin SDK update() on be_staff_chat_messages/{messageId}, mutating ONLY attachments[i].pdfPreview* fields (rules allow admin-SDK writes; client write rule unchanged).
  • Retry: Cloud Run built-in retry on transient (cold-start timeout, network); LibreOffice non-zero exit = permanent failure, no retry, status='failed'.
  • Resource caps: 1 GB memory, 540s timeout (Cloud Run max) — conversion of a 50MB file completes in <30s typical. Concurrency = 1 (LibreOffice not thread-safe inside the container).

Layer 2 — Client (display)

  • NEW src/lib/staffChatOfficePreviewCore.js: pure JS, no imports. Exports OFFICE_CONVERTIBLE_MIMES (Set), isOfficeConvertible(mime), OfficePreviewStatus (frozen object of state constants), pdfPreviewStateOf(att) (returns one of 'ready'|'pending'|'failed'|'unsupported'|'na' from an attachment record — pure derivation, no Firestore).
  • MOD src/lib/staffChatRetentionCore.js: attachmentKindFor returns 'office' when the MIME is in OFFICE_CONVERTIBLE_MIMES (else current behaviour). NEW branch sits BEFORE the generic 'file' fallback.
  • MOD src/components/staffchat/StaffChatAttachmentCard.jsx: new branch when attachmentKindFor === 'office' — render ⏳/👁/⚠ based on pdfPreviewStateOf(att). ⬇ unchanged. AV78-compliant (no backdrop click on the existing PDF overlay; the new branch only adds buttons inside the card).
  • MOD src/lib/staffChatClient.js buildMessageDoc: when an attachment's MIME is in the convertible set, stamp pdfPreviewStatus: 'pending' on the initial doc write. Receiver UI shows the ⏳ spinner immediately, then flips to 👁 when the Cloud Function patches.
  • StaffChatPdfOverlay reuse: zero changes. The existing iframe overlay accepts a fileUrl prop; the new card hands it pdfPreviewUrl.

Layer 3 — Rules + retention

  • storage.rules: existing match /staff-chat-attachments/{branchId}/{messageId}/{file=**} already allows admin-SDK writes (admin bypasses rules). No new path needed — the PDF lives in the same prefix as the original. The client-write rule remains as-is (only image/pdf MIME from authed clients).
  • firestore.rules: existing be_staff_chat_messages rule allows admin-SDK writes. Field-level whitelist not required for admin patches.
  • V73 30d retention sweep: existing api/cron/staff-chat-retention-sweep.js deletes the entire staff-chat-attachments/{branch}/{messageId}/ prefix on age-out. The PDF is in the same prefix → deleted alongside the original automatically. No new cron.
  • AV108 amendment: extend the existing invariant text to add ONE sanctioned exception — local server-side conversion via the in-project Gotenberg Cloud Function. The "no 3rd-party doc viewer" rule continues to forbid Google Docs Viewer / Office Online / Aspose Cloud / ConvertAPI / any external HTTP service that receives the patient file. Add closed-list of 1 sanctioned exception: functions/officeToPdf/.

Acceptance Criteria

  1. Pixel-perfect fidelity for a typical clinic .docx: uploading a Word doc that contains a Thai-script header, a 3-column table, an embedded signature image, and bullet points produces a PDF preview that is visually indistinguishable from opening the original in Microsoft Word. Verified by Chrome MCP screenshot vs reference (manual visual diff acceptable).
  2. Pending-to-ready transition completes in <10s typical: receiver opens chat right as sender uploads; sees ⏳ for at most 10s, then 👁 (Cloud Run cold start <6s + LibreOffice convert <4s for a 1MB file).
  3. Failure cases yield correct state: password-protected .docx → failed + Thai error tooltip; .odt → unsupported + plain card; corrupt .xlsx → failed; 49MB .pptx → ready with PDF (just under Cloud Function size limit).
  4. 30d retention sweeps both files: V73 retention sweep at day 31 deletes both the original .docx AND the cached .pdf in the same prefix. Verified via L2 admin-SDK e2e on real prod with a TEST-prefixed fixture.
  5. No PHI leak: AV108 source-grep regression test confirms no 3rd-party doc viewer URL/library appears in the client bundle; Cloud Function uses ONLY localhost:3000 (bundled Gotenberg) for conversion — no external HTTP calls during conversion.
  6. Rule Q V66 — L1 verification before any "ready" claim: Chrome MCP real-browser confirms (a) ⏳ spinner shows during pending, (b) the PDF iframe overlay opens on 👁 click with correct content, (c) ⚠ tooltip shows the Thai reason on failed conversions, (d) ⬇ download always works in every state. Screenshots captured per AV-Q-vis.
  7. vitest full suite stays green: existing 14078 tests + new tests (~30-50 added) all PASS. No V21 lock-in regressions from extending attachmentKindFor.
  8. Cost stays under free tier: 1500 Office uploads/month (clinic ceiling) consumes <10% of Cloud Run free-tier quota. Verified by Cloud Run metrics post-deploy.

Risks / Open Questions

  • First production deploy of a 2nd-gen Cloud Function with Docker base image in this project. Existing functions (chat retention, recall cron, etc.) are 2nd-gen but Node-native; this is the first Docker. Risk: Cloud Build wiring + IAM permissions for the function's service account to read/write Storage + Firestore. Mitigation: use the canonical firebase-tools CLI workflow + commit the functions/officeToPdf/Dockerfile alongside the source so deploy is reproducible.
  • Storage signed URL TTL (7d) shorter than V73 retention (30d): if a receiver opens a 10-day-old chat, the signed URL is expired. Mitigation: client falls back to a fresh signed-URL fetch via a small admin endpoint, OR (simpler) re-sign on each Firestore listener pickup at the Cloud Function level (only on read). Decide at plan-write time.
  • Cold-start latency on first conversion of the day: Gotenberg image is ~700MB → cold start can hit ~10-15s. Mitigation: min-instances=0 keeps cost low and accepts a one-time per-day cold start; the user sees the ⏳ spinner for that interval, then 👁. If unacceptable, bump min-instances=1 for ~$3-5/month.
  • CSV rendering quality: LibreOffice's auto-import of CSV may guess delimiters/encoding wrong for unusual files (e.g. TIS-620 Thai). Acceptable fallback: status='failed' with tooltip explaining the user should rename to .xlsx and re-upload, OR open in Excel locally.
  • Listener bandwidth — attachments[i] patches re-fire onSnapshot: V73's useStaffChat already handles per-message updates. The PDF patch is one extra update per Office message. Negligible at clinic scale, but worth a Rule I flow-simulate confirming no double-render.
  • iPad Safari iframe rendering of PDF: iOS Safari's native PDF iframe is sometimes blank without specific Content-Type headers. The existing StaffChatPdfOverlay already works for chat-shared PDFs (Q-vis verified prior session), so this should hold — but worth a Chrome MCP iPad-emulation pass during verify.
