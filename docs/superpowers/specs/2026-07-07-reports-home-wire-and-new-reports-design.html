Reports Home — Wire-Up + Data-Ready New Reports

Date: 2026-07-07 · Status: Approved by user (design) · Plan: docs/superpowers/plans/2026-07-07-reports-home-wire-and-new-reports.html

Purpose / Problem

The Reports home landing page (ReportsHomeTab.jsx) is a flat card grid replicating ProClinic /admin/report. Its card list is a hand-maintained mirror of "which reports exist" that has drifted from reality (a V52-class multi-reader / adoption gap):

  • 6 fully-built, registered, working report tabs (~2,445 lines of real report code) are shown on the landing page as "เร็วๆนี้" (coming-soon, disabled) OR are not shown at all — even though navigating to them via the left menu works fine. Mislabeled/hidden: reports-pnl, expense-report, reports-df-payout, reports-remaining-course, clinic-report, reports-payment.
  • ~20 more cards are genuinely un-built. Some are cheap to build from data + loaders that already exist; the rest are complex/low-value.

User goal (verbatim): "ทำให้หน้านี้ใช้ได้ทุก function และเอาที่แนะนำเพิ่มเติมด้วย ... ถ้าได้ก็จัดการเลย". Scope locked via Q&A: (A) wire every existing tab + build the data-ready new reports, and remove every card that isn't wired to a working tab this round → the page ends with zero dead cards.

Decisions (locked via Q&A with user)

  • Q1 = A: Wire every existing working tab and build the data-ready new reports (not wiring-only, not everything-including-complex).
  • Q2 = Remove all: Every card not wired to a working tab this round is deleted — the page has zero dead/"เร็วๆนี้"/disabled cards. Includes Smart Audience.
  • Consolidation: online + vendor share one tab (reports-alt-sales, 2 cards → 1 tab); the 3 stock-health cards share one tab (reports-stock-alert) — mirrors the existing "2 staff-sales cards → 1 tab" precedent; fewer files, less risk.
  • Expense collapse: "รายจ่ายทั้งหมด" → expense-report (which has an in-page category breakdown); a distinct "ค่ามือแพทย์ (DF)" → reports-df-payout. รายจ่ายพนักงาน/อื่นๆ removed (they are category filters inside expense-report).
  • Customer-sales dupe: ยอดขายลูกค้า / ยอดขายรายลูกค้า removed — reports-customer (ลูกค้าสาขา) already shows per-customer purchase summary.
  • Branch scope: all new tabs are branch-scoped (V52 BS-11) via useSelectedBranch + branchId; loaders are Firestore-only (Rule E).
  • Names on stock/sales reports may use red (product / invoice names) — the Thai-culture red rule forbids red only on patient names/HN.

Acceptance Criteria

  1. ReportsHomeTab renders zero cards with status:'soon' or disabled; every visible card navigates to a registered tab.
  2. The 6 wired cards open their existing tabs unchanged (pnl / expense-report / df-payout / remaining-course / clinic-report / payment).
  3. 4 new tabs render real data for the selected branch + re-fetch on branch switch + on date-range change + Export CSV works.
  4. Each new aggregator reconciles exactly to its source (audit-reports-accuracy: footer total === Σ row values); adversarial inputs (empty / null / cancelled / missing fields / Thai text) handled.
  5. All 4 new tab ids resolve as ?tab=… deep-links (present in ALL_ITEM_IDS).
  6. Drift-guard test: every active home card's tabId is a registered navConfig id (build fails otherwise).
  7. Full vitest green + build clean; Rule Q L2 (aggregators vs real prod) + L1 Playwright (click each new/wired card → correct tab + real data).

Risks / Open Questions

  • Outstanding definition: uses netTotal − totalPaidAmount with a 0.005 epsilon; must exclude cancelled/refunded and treat missing paid-amount as 0. Reconcile against a couple of known unpaid sales on prod (Rule Q L2) before trusting.
  • Stock-alert thresholds: per-product alertQtyBeforeOutOfStock / alertDayBeforeExpire may be unset for some products → fall back to sane defaults (e.g. near-expiry ≤ 90 days; skip low-stock rows with no threshold) and label the default clearly.
  • Deep-link-to-section (e.g. "ล็อตหมดอายุ" card scrolling stock-alert to the expired section) is deferred — v1 lands on the combined tab with all sections visible.
  • No new Firestore collections / rules → deploy is vercel-only (no Probe-Deploy-Probe). Confirmed all data + loaders already exist.
