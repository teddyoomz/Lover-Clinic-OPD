Stock-tab product edit — full-doc load + real-time table reflection

Date: 2026-06-02 · Status: Awaiting user review · Class: partial-object / frozen-denormalization at the stock↔product boundary (V12 multi-reader-sweep + Rule O display live-resolve) · Implementation plan: (to follow via writing-plans)

Purpose / Problem

The "แก้ไขสินค้า" modal opened from tab=stock (ยอดคงเหลือ → ปุ่ม แก้ไข behind a row) loads incomplete data — user screenshots: the same product (ถุงมือ Sterile ไซส์ M) shows ประเภท=ยา + blank หมวดหมู่/หน่วย/ราคา from stock, but ประเภท=สินค้าสิ้นเปลือง + หมวดหมู่=อุปกรณ์ + หน่วย=กล่อง from tab=products.

Root cause (confirmed by reading the code):

  1. StockBalancePanel.onEditProduct(p) passes the aggregated row ({productId, productName, unit, totalRemaining, batches, …thresholds}) — NOT the real be_products doc. ProductFormModal seeds form = {…emptyProductForm(), …row} → every field absent on the row falls to defaults (productType:'ยา', blank category/unit/price; the row's unit key ≠ form's mainUnitName so unit stays blank).
  2. CRITICAL data-integrity risk: saveProduct does setDoc(…, { merge:false }) (full overwrite) and normalizeProduct does return { …form, … } (spreads, no whitelist). So saving the stock-opened modal would (a) overwrite the real product with blanks/defaults, and (b) write junk (batches, totalRemaining, nextExpiry, valueCost, unit) onto the product doc. Stock holds millions of baht of inventory.
  3. The balance table's unit is frozen: it renders b.unit (the batch's denormalized unit), not the live be_products.mainUnitName. (Name is already canonical via the existing listenToProducts listener; unit/category/type are not.) → editing a unit anywhere never updates the table (user: "Matigen ยังเป็นครั้ง").

Goal: (1) the stock-tab edit modal loads the complete product (identical to tab=products); (2) editing unit / name / category / type / other fields — from tab=stock OR tab=products OR another device — reflects in the balance table in real time; (3) no save can ever corrupt a product doc; (4) comprehensive tests (stock is mission-critical).

Decisions (locked via Q&A with user)

  • Q1 = remove มูลค่าทุน + ความจุ columns; ADD หมวดหมู่ + ประเภท (both live-resolved from be_products). Name + unit live in all cases. (Header "มูลค่าต้นทุนรวม ฿X" summary kept — only the per-row column removed; flag if you want the header gone too.)
  • Q2 = whitelist only — add the whitelist to normalizeProduct (drop non-product junk). No extra "block save on partial object" guard; the full-doc-load fix is the primary wipe-prevention.
  • Real-time mechanism = live-resolve at display via the EXISTING listenToProducts listener (Rule O pattern) — NO mutation of existing batch denormalization.

Acceptance Criteria

  1. Edit-from-stock opens the modal with the COMPLETE product (type, category, unit, price, code, dosage, flags…) — byte-identical to opening from tab=products.
  2. Change unit ครั้ง→ชิ้น (from stock OR products) → balance table unit shows ชิ้น instantly, no page refresh. Same for ชื่อ, หมวดหมู่, ประเภท. Verified on a 2nd device/tab via the live listener.
  3. Saving the product (from either tab) NEVER wipes real fields and NEVER writes stock-aggregation junk (batches/totalRemaining/…) onto the doc — proven on a REAL prod product round-trip (Rule Q L2).
  4. Balance table renders with the new columns (สินค้า · Batches · คงเหลือ · หมวดหมู่ · ประเภท · หมดอายุถัดไป · ACTIONS); lot-expand sub-rows align; badges/filters (หมด/ใกล้หมด/เกินสต็อก/ติดลบ/ใกล้หมดอายุ) still work.
  5. Comprehensive tests: unit (map build + live-resolve + whitelist preserves stockConfig/forensic) + RTL (stock-edit opens full modal; columns) + Rule I flow-simulate (edit unit/name/category/type → table reflects) + corruption regression (partial object can't wipe/junk a doc) + Rule Q L2 real-prod round-trip + source-grep AV.

Risks / Open Questions

  • KEY RISK — whitelist drops a legit field. A be_products doc carries fields beyond emptyProductForm: stockConfig (trackStock — drives whether the product has batches!), mainProductId/mainProductName, isHidden (V41), forensic _v* stamps (V46). A careless whitelist would DROP these on save → break stock tracking / hide-flag / audit trail. Mitigation: Rule R diag on real prod docs to enumerate the FULL field set; the whitelist = that set; a test asserts a real doc round-trips through normalizeProduct with zero field loss.
  • Removing 2 columns + adding 2 changes the table's column count → the lot-expand sub-rows (which use fixed colSpan/cell positions) must be realigned, or they'll mis-render.
  • be_products is branch-scoped (BSA) → live-resolve is per-branch (consistent: the balance table is per-branch; editing a product reflects in that branch's table). No cross-branch surprise.
  • Header "มูลค่าต้นทุนรวม" — kept as the aggregate summary. Confirm you're OK keeping it (only the per-row column is removed).
