WS1 — Anonymous-Surface Lockdown

Date: 2026-06-10 · Status: Approved by user (Q1=แก้ทั้งหมดเรียงตามความเสี่ยง · Q2=C crypto-id + split get/list) · Implementation plan: docs/superpowers/plans/2026-06-10-ws1-anon-surface-lockdown.html

Purpose / Problem

LoverClinic stores patient PII (national ID idCard 13-digit, name, phone, medical/treatment data) and money records. A multi-agent adversarial security audit (2026-06-10) confirmed that several Firestore/Storage rules are over-permissive toward anonymous users. The app signs every public-link visitor in via signInAnonymously (src/App.jsx:101), so any rule written if isSignedIn() or if true is reachable by any internet user with zero credentials.

WS1 is the first (CRITICAL + HIGH) workstream of a 4-workstream hardening effort. It is a single firestore.rules + storage.rules deploy unit (one Probe-Deploy-Probe cycle — rule deploys are the riskiest, V1/V9), plus the minimal client changes needed so the patient flow does not break (V23 lesson: a prior over-tightening of opd_sessions broke patient form submission).

Architecture

Boundaries / units changed (small + isolated):

  • firestore.rules — the get/list splits + write tightenings (C1, H1, H2, M2). The single highest-leverage change; one file.
  • storage.rules — M2 documented (no functional change; Storage can't split get/list).
  • Client: opd_session id minting — audit the ~3-5 mint sites (AdminDashboard, backendClient, appointmentDepositBatch) and route them through one crypto-random helper if a shared one exists (Rule of 3); guarantee 128-bit id-as-secret.
  • Client: PatientDashboard ?patient= — replace the client-side query(opd_sessions, where patientLinkToken==token) with a fetch to /api/patient-view?token= (extend the endpoint payload only if the dashboard needs fields patient-view doesn't yet return).
  • Webhook chat writes — confirm/migrate every chat_conversations + messages write to admin SDK so the if true rule can be removed without breaking inbound messages.
  • Audit-anti-vibe-code invariant — add an AV invariant: no Firestore rule may grant list on a PII collection to anon (isSignedIn() without claim); source-grep the rules file. Locks the class permanently.

Decisions (locked via Q&A with user)

  • Q1 = A: Fix all findings in risk order; each workstream gets its own spec + plan; test + deploy per chunk. WS1 first.
  • Q2 = C: C1 depth = crypto-secret id + split get/list (not full server-mediation B, not enumeration-kill-only A). Keeps patient real-time + no endpoint rewrite of the submit flow; migrates session ids to crypto-random so get-by-id is safe.
  • Bundling: C1+H1+H2+M2 ship as one rules deploy (one Probe-Deploy-Probe) because rule deploys are the riskiest and batching = one carefully-probed cycle (safer than N cycles).
  • Legacy ids: rely on existing opd-session-cleanup-sweep cron age-out; optional Rule M migration as a non-blocking follow-up.

Acceptance Criteria

  1. An anonymous client-SDK session CANNOT list opd_sessions, clinic_schedules, or form_templates (PERMISSION_DENIED) — proven by scripts/diag-ws1-anon-lockdown.mjs (Rule Q L2).
  2. An anonymous client CANNOT write clinic_schedules or chat_conversations/messages (PERMISSION_DENIED).
  3. All 3 patient links (?session= submit, ?patient= dashboard, ?schedule= booking) work end-to-end on the LIVE deployed build with tightened rules (Rule Q L1, real browser).
  4. Every new opd_session id is crypto-random ≥128-bit (no Date.now()-only ids); source-grep audit + AV invariant added.
  5. Webhook inbound (FB/LINE) still lands chat messages after the if true rule is removed (admin SDK path).
  6. Probe-Deploy-Probe passes pre+post (existing probes 200; new anon-list probe denied); revert on any regression (Rule B).
  7. Full vitest green + build clean; no patient-flow RTL/flow-simulate regressions.

Risks / Open Questions

  • V23 re-break risk (highest): over-tightening opd_sessions previously broke patient submit. Mitigation: split get/list (NOT blanket staff-only), deploy client layer first, L1 before AND after rules, Probe-Deploy-Probe with a patient-get probe.
  • H1 hidden REST writer: if any chat write still goes through unauth REST (not admin SDK), tightening breaks inbound messages. Mitigation: grep/confirm every chat write path FIRST; migrate residual to admin SDK before the rule change; post-probe verifies a forged anon write is denied AND a real webhook message still lands.
  • PatientDashboard payload gap: /api/patient-view may not return every field the dashboard renders. Mitigation: diff the dashboard's needs vs the endpoint payload; extend the endpoint (still field-minimized, no national ID) if needed.
  • Legacy guessable ids: accepted residual; cron age-out + optional Rule M migration. Document, don't block.
  • clinic_settings public doc: open question — confirm via Rule R diag that the world-readable clinic_settings/{id} doc contains no secret (channel tokens / keys). If it does → move secret to the staff-only path.
