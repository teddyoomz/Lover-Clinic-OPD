Purpose / Problem

ปัจจุบันปุ่ม "สร้างลิงก์ดูข้อมูล" มีเฉพาะใน frontend queue (AdminDashboard) และผูกกับ opd_session + ข้อมูล latestCourses ที่ ProClinic เคย sync — แต่ V50 (2026-05-08) ถอด ProClinic ออกหมดแล้ว ทำให้ field นั้นไม่มีตัวเขียน. ผู้ใช้ต้องการปุ่มเดียวกันในหน้า CustomerDetailView (ดู be_customers) เพื่อสร้างลิงก์ให้ ลูกค้า (anon, ไม่ต้อง login) เปิดดูนัดหมาย + คอร์สคงเหลือของตัวเองได้ทุกเวลา. ลิงก์ต้องดึงข้อมูล สดจาก be_* (ไม่พึ่ง ProClinic).

Security correction จาก Q1: appointments อยู่คนละ collection (be_appointments by customerId) + courses บน customer doc. การเปิด anon-read be_customers/be_appointments ใน firestore.rules = เปิด PII ลูกค้าทั้งฐานให้ใครก็อ่านได้ (rules gate "การรู้ token" ไม่ได้สำหรับ collection query) → ห้ามทำ. ใช้ public serverless endpoint (admin SDK, token-gated) แทน — be_customers/be_appointments ยังล็อค clinic-staff เหมือนเดิม.

Flow

A · สร้างลิงก์ (clinic-staff)
กด 🔗 ในหน้าลูกค้า→ ยังไม่มี token → "สร้างลิงก์ดูข้อมูล"→ gen crypto token (128-bit)→ client write be_customers: patientLinkToken + patientLinkEnabled=true→ modal โชว์ลิงก์ + QR + คัดลอก/เปิด/ปิด/เพิกถอน
B · ผู้ป่วยเปิดลิงก์ (anon · NO login)
ลูกค้าเปิด ?patient=<token>→ PatientDashboard customer-mode: GET /api/patient-view?token=→ server (admin SDK) query be_customers WHERE patientLinkToken==token→ enabled? → อ่าน courses[] + query be_appointments by customerId→ return เฉพาะ field ปลอดภัย (ชื่อ·HN·courses·appointments)→ render หน้าผู้ป่วย

token ไม่พบ → 404 → "ไม่พบข้อมูล" · patientLinkEnabled=false → "ปิดใช้งาน". ถ้า endpoint 404 → PatientDashboard fallback ไป path เดิม (opd_session legacy, ไม่แตะ).

Architecture

ชั้นรายละเอียด
Token modelbe_customers.patientLinkToken (string, crypto 128-bit hex) + patientLinkEnabled (bool). สร้าง/เพิกถอนโดย clinic-staff (client SDK write — be_customers เขียนได้โดย clinic-staff อยู่แล้ว).
api/patient-view.js (NEW)Public GET, token-gated. admin SDK. resolve be_customers by token → ถ้า enabled → คืน { patientName, hn, courses[], appointments[] } (field-minimized; ไม่มี national ID). 404 ถ้าไม่พบ/disabled. อ่าน be_appointments by customerId ฝั่ง server. ไม่ต้อง Firebase auth (server validate token).
PatientDashboardcustomer-mode: ลอง fetch(/api/patient-view?token=); สำเร็จ → render customer view (shell/ธีม/TH-EN เดิม). 404 → fallback path เดิม (opd_session listener, unchanged).
CustomerDetailViewrestructure แถวปุ่ม → Layout A. ปุ่ม 🔗 ม่วง เปิด CustomerPatientLinkModal.
CustomerPatientLinkModal (NEW)เลียนแบบ image-1 modal, customer-scoped: empty→สร้าง; active→ลิงก์+คัดลอก+QR+เปิด/ปิด+เพิกถอน.
backendClient.jshelpers: generateCustomerPatientLink(customerId) · setCustomerPatientLinkEnabled(customerId, enabled) · revokeCustomerPatientLink(customerId).
firestore.rulesไม่แตะ ✅ — be_customers/be_appointments ยังล็อค clinic-staff. ไม่ต้อง Probe-Deploy-Probe.

Decisions (locked via Q&A with user)

  • Q1 = A: ลิงก์ระดับลูกค้า (customer-level) อ่าน be_* สด — ไม่ใช่ reuse opd_session legacy.
  • Q2 (layout) = A: action group [แก้ไข · ผูก LINE · 🔗 ลิงก์] + เส้นคั่น + [ลบลูกค้า] แยกแถวล่าง.
  • Data path = A2 (refinement): public token-gated serverless endpoint (admin SDK) — ไม่เปิด anon-read be_customers (security). ไม่มี snapshot doc (กัน stale).
  • Anon · NO login (ผู้ใช้ยืนยัน 2026-05-25): ลูกค้าเปิดลิงก์ดูข้อมูลตัวเองโดยไม่ต้อง login ใด ๆ. endpoint = public, validate ด้วย token เท่านั้น.
  • Modal scope = mirror image 1: สร้าง / คัดลอกลิงก์ / QR / เปิดลิงก์ / เปิด-ปิด / เพิกถอน.
  • Field-minimization: คืนเฉพาะ ชื่อ · HN · courses (ชื่อ+เหลือ) · appointments (วัน/เวลา/ประเภท/สถานะ/หมอ). ไม่มีบัตร ปชช.

Acceptance Criteria

  1. หน้าลูกค้าแสดงปุ่ม 🔗 ลิงก์ (ม่วง) ในกลุ่ม action; "ลบลูกค้า" แยกแถวล่าง (Layout A) — ทั้ง desktop + mobile.
  2. กด 🔗 (ยังไม่มี token) → "สร้างลิงก์ดูข้อมูล" → เขียน token ลง be_customers → modal โชว์ลิงก์ + QR.
  3. Modal: คัดลอก / เปิด / QR / เปิด-ปิด (toggle patientLinkEnabled) / เพิกถอน (clear token+enabled) ทำงานครบ.
  4. ลูกค้าเปิด ?patient=<token> โดยไม่ต้อง login → เห็น ชื่อ + HN + นัดหมาย + คอร์สคงเหลือ (สด).
  5. token ปิด → "ปิดใช้งาน"; token ผิด/เพิกถอนแล้ว → "ไม่พบข้อมูล".
  6. endpoint คืนเฉพาะ field ปลอดภัย (ไม่มี national ID); be_customers/be_appointments ยังล็อค clinic-staff (ไม่แก้ firestore.rules).
  7. path opd_session patient-link เดิมไม่พัง (fallback).
  8. Rule Q: L2 endpoint verified บน prod จริง (TEST customer + cleanup) + L1 เปิดลิงก์จริงใน browser เห็นข้อมูล TEST customer.

Risks / Open Questions

  • Public endpoint = attack surface — ลดด้วย token crypto 128-bit (brute-force ไม่ไหว) + read-only + field-minimized. พิจารณา rate-limit เบา ๆ (optional).
  • App.jsx render gate — route ?patient= ปัจจุบัน gate render จนกว่า anon-auth (user) จะ resolve (V16/V23). customer-mode ดึงข้อมูลจาก endpoint (ไม่ต้องใช้ Firebase auth) — anon-auth ที่ยังทำงานอยู่ไม่เป็นพิษ (สำเร็จเร็ว) แต่ทางเทคนิคไม่จำเป็นต่อ data fetch. Plan ต้องเช็คว่า gate ไม่บล็อก customer-mode โดยไม่จำเป็น (ปล่อย anon-auth ไว้ได้ ถ้าไม่ทำให้ช้า).
  • PatientDashboard ไฟล์ใหญ่ — เพิ่ม customer-mode branch ต้องระวังไม่พัง legacy path (Rule I flow-simulate ครอบทั้ง 2 path).
  • AVxx invariant — ล็อกว่า endpoint ต้อง field-minimize + be_customers ห้ามเปิด anon-read (กัน regression อนาคต).
