Flow

send.js — BEFORE (broken + latent weak) vs AFTER (proper clinic-staff gate)

BEFORE: POST /api/webhook/send→ import verifyAuth (V50-deleted)→ 💥 HTTP 500 (broken-closed)  ·  had the import worked: anon token → verifyAuth pass (no claim) → 🔓 clinic sends arbitrary message

AFTER: POST (no auth)→⛔ 401  ·  anon-auth token→⛔ 403 (no clinic-staff claim)  ·  staff token (isClinicStaff)→✓ send (feature restored)

Decisions

  • Restore with a CLINIC-STAFF gate, not admin-only nor any-auth: replying to a customer chat is a staff action; the gate mirrors firestore.rules isClinicStaff() (admin OR isClinicStaff claim). This fixes the 500 AND closes the latent weak-auth.
  • Schedule token → 128-bit for consistency with patientLinkToken.
  • patient-view rate-limiting = documented residual, NOT shipped (Rule Q-honest): the 128-bit token already kills enumeration; a per-instance in-memory limit on Vercel is security-theater (many instances, cold-start reset); a real shared-store limit adds a Firestore round-trip to every legit patient request. Given the strong token + Vercel's platform DDoS protection, application rate-limiting is low-value here. Recommend revisiting only if abuse is observed.

Acceptance Criteria

  1. Deployed POST /api/webhook/send with NO auth → 401 (was 500) — proves the import is fixed AND the gate rejects unauthenticated callers.
  2. No api file IMPORTS the V50-deleted proclinic/_lib/auth.js (regression guard).
  3. send.js + saved-replies.js gate on verifyClinicStaffToken; the gate checks isClinicStaff/admin claims.
  4. Schedule token is 128-bit; full vitest green; build clean.

Risks / Open Questions

  • Restoring send.js un-breaks FB-reply-via-app: it's been 500ing for a month (LINE replies already go via LINE OA Chat per the stack docs; FB reply via app was the broken path). Restoring it WITH proper auth is correct regardless; if the clinic doesn't want the FB-reply-via-app feature, it can be removed separately — the auth-gate fix stands either way.
  • verifyClinicStaffToken behavior: verified by source-grep + a post-deploy L2 (curl no-auth → 401). It mirrors the already-tested verifyAdminToken (same admin-SDK verifyIdToken + claim pattern), so the logic risk is low.
