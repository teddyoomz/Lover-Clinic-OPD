PatientForm Light-Theme WCAG-AA — B-i Selective

Date: 2026-05-28 (EOD+1 follow-up) · Status: Approved by user (Q1=B, Q2=B-i, Q3=Selective) · Plan: docs/superpowers/plans/2026-05-28-patientform-light-theme-aa.html (to follow)

Purpose / Problem

src/pages/PatientForm.jsx is the customer-facing intake form (anon-auth public route ?session= / ?patient=). It was the one major surface V125 deferred ("needs a DESIGN pass, NOT a mechanical aaAccent wrap") because its colors are a deliberate pink/rose brand expressed via bespoke isDark ? darkHex : lightHex inline styles.

V124's class-based [data-theme="light"] overrides already deepen every Tailwind text-{color}-{300..600} class to its AA-dark shade (index.css ~4093–4341), and the .pf-* CSS classes + V107 form-element safety net cover inputs/labels/submit. What still fails AA are the ~7 INLINE style={{color:'#hex'}} accents that no class can reach (the exact V125 class-of-bug, never wired into PatientForm) plus the dynamic clinic accent ac = cs.accentColor (no AA guard → if a clinic picks pink-500 the white-on-accent submit/toggle is 3.3:1).

Goal: zero sub-AA color in PatientForm light theme, in a way that harmonizes with the pink/rose hero (B-i) while keeping the existing per-section semantic zones (orange=emergency, blue=custom, red=medical-critical — all already AA via V124) (Selective). Dark theme: zero change (V124 lesson).

Architecture

Reuse, don't reinvent. V125 already shipped src/lib/themeAccent.js aaAccent(hex, isDark) whose LIGHT_AA_ACCENT map already contains every shade this fix needs (pink-500→#be185d, rose-500→#be123c, orange-500→#c2410c, red-500→#b91c1c, blue-500→#1d4ed8, etc.). No new helper, no new map entries.

  • Inline accent fixes — replace each broken inline hex with its deepened light value (wrapped through aaAccent(hex, isDark) where the value should stay vibrant in dark, or a flat light-only hex where the dark branch already differs).
  • Asterisks — introduce ONE shared class .pf-req in index.css (theme-aware: light #e11d48 rose-600 = 4.7:1; dark #fb7185 rose-400, AA on dark). Replace all <span> asterisks (both className="text-red-600" and inline #ef4444) → className="pf-req". Asterisks are a single uniform functional marker (NOT a semantic zone), so unifying them is in-scope for "Selective" + delivers the rose asterisk approved in Q2. Input accent-color classes (radio/checkbox text-red-600) are NOT touched.
  • Dynamic ac — add const acLight = aaAccent(ac, isDark) near the existing ac definition (PatientForm.jsx:20). Point the white-text-on-accent backgrounds (submit L1349 non-editing branch, lang-toggle L637/641) + the ac-as-foreground icon (L684) at acLight. The isEditing blue-600 #2563eb branch is already AA — untouched.
  • Decorative — hero gradient, top strip, card backgrounds, soft tints, faint borders all UNCHANGED (V124 lesson: only deepen text/foreground, leave decoration).

Decisions (locked via Q&A with user)

  • Q1 = B: Curated brand-cohesive light palette (not a mechanical aaAccent wrap, not failures-only minimal-touch).
  • Q2 = B-i: Pink/rose-harmonized flavor (rose-toned structural accents + pink gradient submit), over semantic-preserving (B-ii) or warm-clay (B-iii).
  • Q3 = Selective: Fix the ~7 broken inline-hex sites + dynamic ac; KEEP the orange-emergency / blue-custom / red-critical semantic zones (already AA via V124). NOT a full ~50-site monochrome-rose refactor.
  • Dynamic accent (locked, flagged for review): acLight = aaAccent(ac, isDark) — reuse V125 helper. Standard Tailwind clinic picks are all in the map; a truly custom non-standard hex would pass through unchanged (documented Risk below).
  • LINE green (locked): #06C755 button kept as the LINE brand standard (sanctioned exception, like V125 left teal/LINE-green).
  • Dark theme: zero change (V124/V125 lesson — deepen light only).

Acceptance Criteria

  1. Live-browser scan (Rule S / Q-vis) of the rendered PatientForm in light theme, across all 5 states (intake form, success, session-closed, session-expired, invalid-link) → 0 text/foreground AA fails on white/near-white backgrounds (the V125 gold-standard: scan the REAL build, not just inject-simulate).
  2. With cs.accentColor = '#ec4899' (pink-500 default) → submit button bg + active lang-toggle render #be185d (pink-700) in light; white text ≥ 4.5:1.
  3. All required asterisks render one consistent rose-600 #e11d48 (light) — no red/rose mismatch between inline and className sites.
  4. Per-section semantic zones still distinguishable in light: emergency=orange-700, custom-questions=blue-700, medical-critical=red-700, hero/submit=pink/rose.
  5. Dark theme pixel-identical to today (regression guard — every change is light-branch-only or aaAccent pass-through in dark).
  6. T7 (tests/light-theme-override-coverage.test.js) extended: source-grep that no raw sub-AA inline hex (#f97316/#ef4444/#3b82f6 as color:) remains in PatientForm.jsx light branches; AA-math asserts every mapped target ≥ 4.5 on white; .pf-req AA both themes; acLight wiring present.
  7. npm run build clean; full vitest green (no regression).

Risks / Open Questions

  • Non-standard clinic accent: aaAccent passes through a hex not in LIGHT_AA_ACCENT unchanged → a clinic that sets cs.accentColor to an odd non-Tailwind hex could still be sub-AA on the submit/toggle. LoverClinic's accent IS pink-500 (in the map). Mitigation if it ever matters: add a runtime contrast-clamp to aaAccent (HSL-darken until ≥4.5) — deferred, out of scope (YAGNI for a single-clinic accent today).
  • Asterisk unification touches ~16 spans: low-risk (single uniform marker), but it is the largest single edit. The .pf-req class isolates it; input accent-color classes are deliberately left alone.
  • Live-scan friction: the form needs a session token or simulation entry to render. Plan will use the dev server + a real prod patient-link (Rule R read-only) OR admin simulation mode for the verification scan; static analysis already enumerated the fixes above.
  • L555 LINE button stays 2.94:1 by brand decision — documented, not a defect.
