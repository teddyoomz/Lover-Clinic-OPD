Appointment Calendar — Density & Mobile Design Spec

Date: 2026-05-20 (EOD+3) · Status: Approved by user (A+B+C + Q1–Q3 locked) · Research: docs/superpowers/research/2026-05-20-appt-calendar-density-research.html · Plan: docs/superpowers/plans/2026-05-20-appt-calendar-density.html (next, via writing-plans)

Purpose / Problem

The appointment time-grid (AppointmentCalendarView.jsx) hard-couples block height to duration: height = span × SLOT_H − 4 (SLOT_H = 22px/15-min). A 15-min appointment = 18px tall → the customer-name line clips + overlaps the grid line; dense back-to-back short appointments become an illegible column (user screenshot). The grid already scales content by span (🎯 service@30min, 👨‍⚕️ doctor@45min, +assistant@60min) so span≥2 reads fine — span=1 is the failure. SLOT_H was already bumped 18→22 once for the same complaint, so bumping again is a symptom fix (architecture signal). On mobile the grid also horizontally scrolls (columns min-w-160px in overflow-x-auto) → 2D-scroll + tiny blocks.

Goal: let users see full details at ANY density, make the grid itself legible, and be mobile-friendly — by adding 3 layers (A popover + B adaptive cell + C mobile agenda) WITHOUT re-tuning the broken pixel-per-minute dimension.

Decisions (locked via Q&A with user)

  • Q1 = A: click/tap any block → read-only popover (full details + 📞 tap-to-call) with "✎ แก้ไข" → existing edit modal. (Popover-first, not edit-first.)
  • Q2 = A: below lg → auto-switch to Agenda list; ⊞ toggle to return to grid. (≥ lg defaults to grid.)
  • Q3 = A: Agenda is chronological (by time) with the room shown as a tag in each card.
  • Architecture: add 3 layers; do NOT bump SLOT_H again (symptom). Names/HN never red (Thai-culture); phone uses call-accent.

Acceptance Criteria

  1. Clicking/tapping ANY block (including a 15-min block) opens a popover with full details + a working 📞 tel: link + "แก้ไข" that opens the existing edit modal.
  2. 15-min (span=1) blocks render one clean legible line (status dot + name + service color) with NO clip/overlap of the grid line.
  3. Overlapping appointments (same room, intersecting time) collapse to a "+N" rollup; tapping it lists them.
  4. Below lg, the view defaults to a chronological Agenda list (room-tagged, full details); a toggle switches to the grid and back. ≥ lg defaults to grid.
  5. Both dark + light themes render cleanly; customer name/HN are never red.
  6. Verified per Rule Q (L1 real browser OR L2 real client SDK) before "done".

Risks / Open Questions

  • Popover positioning: near viewport edges must flip / use a portal (avoid clipping inside the overflow-x-auto grid).
  • Overlap detection: "+N" needs correct same-room + time-range intersection logic; build on the existing duplicate/+1 indicator rather than a parallel path.
  • View/state sync: Agenda + Grid share selected-date + data; the toggle + auto-switch must not fight each other (explicit user toggle wins until date/route change).
  • Don't regress sparse days: the grid must still look right when appointments are few (span≥2 tiers unchanged).
  • Verification: headless preview can't fully show theme/tap nuance — final L1 is the user on a real screen (per standing "user tests UI" preference).
