Duplicate-customer prevention + Recall display / delete / date fixes

Date: 2026-06-16 · Status: Approved by user (design) · Implementation plan: docs/superpowers/plans/2026-06-16-dup-customer-and-recall-fixes.html (pending)

A two-part batch. Part A closes the customer-save integrity gap (duplicate national-IDs + double-click double-creates). Part B fixes the recall subsystem (customer name shows "—", clickable confirmation, recall-case hard-delete + TEST junk, and surfacing the snooze/reschedule date). Both ship in one combined firestore.rules deploy (2 edits) + frontend, plus 2 local admin-SDK Rule-M scripts. Design hardened by a 4-agent read-only census + adversarial probe (workflow w89gycg12).

Purpose / Problem

Part A (items 1+2). Today addCustomer (the single create chokepoint at backendClient.js:735) does no pre-create duplicate check — findExistingCustomerByIdentity only fires on a notFound recovery (2 sites). HN minting is atomic, but the claim+counter+customer-write are 3 separate operations, so a double-click on "บันทึกลง OPD" (or two devices) creates two customers with the same national ID. The user has been manually deleting these duplicates.

Part B (items 3-7). The recall "bubble list" shows the customer name as "—" on every surface because recall.customerName was snapshotted empty at create time (the picker used customer.displayName || customer.name, which is empty for kiosk customers whose name lives in patientData). Separately: recall-case hard-delete is blocked by firestore.rules (be_recall_cases allow delete: if false — V41 soft-archive), so admins can't remove TEST-CASE junk; and the snooze/reschedule target date isn't surfaced in no-answer / reschedule rows.

Architecture

Part A — Customer-save integrity

  • Single chokepoint: all 6 production create paths (CustomerCreatePage, handleOpdClick ×2, handleResync, handleDepositSync ×2) route through addCustomer. Fixing addCustomer covers every caller. (The only bypass is an e2e script — test-only, out of scope.)
  • deriveClaimKey(citizen_id, passport_id) (new helper, colocated with normalizeCustomer): citizen → strip /[-\s]/, require ^\d{13}$, key CITIZEN:{digits}; passport → trim + UPPERCASE (≤30 chars), key PASSPORT:{X}; both present → citizen wins (passport searchable, not guarded); both empty → null (walk-in, no claim).
  • Atomic addCustomer: pre-check getDoc(claim) → uploads (with orphan-cleanup on later failure) → runTransaction{claim get/set, counter get/incr, customer set + _identityClaimKey}. Throws typed DUPLICATE_IDENTITY{existingCustomerId}. {overrideDuplicate:true} appends to linkedCustomerIds + stamps _duplicateOfCustomerId/_duplicateAckBy/_duplicateAckAt.
  • Edit: updateCustomerFromForm becomes a runTransaction that releases the old claim + claims the new one when the identity field changes (else unchanged). Throws DUPLICATE_IDENTITY if the new id collides.
  • Delete/cascade: add be_customer_identity to CUSTOMER_CASCADE_COLLECTIONS (client deleteCustomerCascade) + CUSTOMER_CASCADE_COLLECTIONS_FULL (server /api/admin/delete-customer-cascade + customerBackupCore). Delete the claim by the denormalized _identityClaimKey (O(1), no reverse query). V81 whole-system backup auto-includes it (V122 dynamic enumeration).
  • UI (warn + choose, Q1/Q2): CustomerCreatePage catches DUPLICATE_IDENTITY → warn modal (existing card + เปิดของเดิม / บันทึกซ้ำอยู่ดี). Phone match → inline soft hint (uses findCustomersByField('telephone_number'), no block). OPD/deposit paths catch it too → for intake, default to link to existing; the existing session idempotency (brokerProClinicId / depositSyncStatus / in-flight ref) already covers no-id walk-ins and same-session double-clicks. CDV shows a ⚠ บันทึกซ้ำกับ {existing} badge for flagged duplicates.
  • Backfill + dup-report (Rule M, local): backfill-customer-identity.mjs — two-phase dry-run → --apply: seed be_customer_identity for every existing customer with an id (claim → earliest createdAt; later ones → linkedCustomerIds), denormalize _identityClaimKey, and print a duplicate report (identities with >1 customer) for the user to merge/delete. Idempotent; audit doc.
  • firestore.rules (new block, before catch-all): be_customer_identity/{key} → get: if isClinicStaff(), list: if false (PII — no enumeration), create, update, delete: if isClinicStaff(). (Create-only-immutable was rejected — it breaks edit-reclaim, cascade-free, and override-append.)

Part B — Recall display / delete / dates

  • Name (item 3): NEW enrichRecallsWithCustomerNames helper applied at the recall-load boundary for the 3 list surfaces (RecallTab, RecallFrontendView) + CDV (customer already loaded). Overlays customerName via resolveCustomerDisplayName (live-resolve primary, denormalized fallback — V113; NO admin-SDK display backfill per feedback_no_admin_sdk_backfill_to_fix_display.md). Every downstream reader (RecallRow + 5 modal headers) gets it free. Write path (RecallCreateModal picker + createRecall) uses the same resolver for correct new snapshots.
  • Clickable (item 4): already implemented in RecallRow.jsx:148/204 (<a href="/?backend=1&customer=…">). Verify the ?customer= handler navigates on all 3 surfaces post-name-fix; keep new-tab (existing system pattern). No new work beyond item 3.
  • Snooze/reschedule date (items 6/7): add a chip in RecallRow: status==='no-answer' && snoozedUntil → 📞 โทรอีกครั้ง {dd/mm/yyyy พ.ศ.}; reschedule (outcome==='reschedule' or pending-with-snooze) → 📅 เลื่อนนัด {dd/mm/yyyy พ.ศ.}. Shared atom → shows in all 3 lists. (Bangkok date via the canonical helpers; พ.ศ. like the ED-followup dates.)
  • Delete (item 5): narrow firestore.rules be_recall_cases allow delete: if isClinicStaff() (safe — recalls snapshot the case name as a string; soft-archive isHidden coexists; UI already has both buttons). The "ลบเคส" button then truly deletes. nuke-test-recall-cases.mjs (Rule M) removes the TEST-CASE/TEST/E2E junk now. Appointment delete is the same recall (Recall lives under the นัดหมาย menu) — confirm real be_appointments deletes work (rules already allow write) and root-cause via systematic-debugging if any real appointment delete fails.

Decisions (locked via Q&A + adversarial hardening)

  • Q1 = warn + choose: national-ID dup → warn modal + open-existing (default) / override. Override is auditable (flagged + claim linkedCustomerIds), not a silent dup.
  • Q2 = ID/passport blocks, phone warns: citizen_id/passport_id = guarded identity (modal). phone = inline soft hint only.
  • Q3 = Bulletproof (Rule T): atomic identity-claim in one runTransaction (counter+claim+customer). Edit + delete + override all transactional re: the claim. Needs the new collection + rules + backfill.
  • Q4 = hard-delete (rules + deploy): be_recall_cases delete → isClinicStaff; TEST junk nuked via Rule M.
  • Hardening decisions: claim key type-prefixed + citizen=13-digit-validated · uploads pre-checked + orphan-cleanup · override appends to claim (not silent) · cascade/edit free/move the claim · rules get/create/update/delete if staff, list:false · name enriched at the load chokepoint (covers rows + 5 modals).

Acceptance Criteria (Rule Q)

  1. Saving a customer whose national ID exists → warn modal (open-existing / override); a phone-only match → inline hint, no block. Editing a customer's ID to a colliding one is blocked. Verify L1 (real browser, both surfaces) + L2 (real client-SDK concurrent double-create e2e on prod: two parallel addCustomer with the same id → exactly one created, the other gets DUPLICATE_IDENTITY).
  2. Deleting a customer frees its claim; backfill seeds existing claims and prints the duplicate report. Idempotent re-run = 0 writes. L2 on real prod (TEST-prefixed fixtures, cleaned up + audit doc).
  3. Every recall surface (3 lists + 5 modal headers) shows the real customer name, clickable to the customer tab; no "—" when a customerId exists. L1 real browser across backend recall tab + frontend recall + CDV.
  4. No-answer-snoozed rows show 📞 โทรอีกครั้ง dd/mm/yyyy พ.ศ.; reschedule rows show 📅 เลื่อนนัด dd/mm/yyyy พ.ศ.. L1.
  5. "ลบเคส" deletes for real (post rules deploy); TEST-CASE junk gone; appointment delete confirmed working. Probe-Deploy-Probe green both sides.
  6. Full vitest green + build clean; new AV invariants + flow-simulate + source-grep regression locks.

Risks / Open Questions

  • Deploy-coupled: Part A's atomic claim needs be_customer_identity live + backfilled BEFORE the new addCustomer enforces it — order: deploy rules → run backfill (seeds claims so existing dupes are detected) → the enforcement is live. Backfill must run with --apply right after the rules deploy.
  • Existing duplicates: the backfill reports them (claim → earliest, rest → linkedCustomerIds) but does NOT auto-merge — the user resolves them manually (consistent with "ผมไปนั่งลบมาหลายคนแล้ว"). A merge UI is out of scope (follow-up).
  • Walk-in no-id customers remain un-guarded by identity (only session idempotency). Acceptable — the user's pain is national-ID dupes. Documented, not "fixed".
  • Appointment-delete root cause is assumed = the recall-case rule (Recall under the นัดหมาย menu). If real be_appointments delete also fails, it's a separate bug to diagnose during implementation (id-resolution / handler) — will surface in the plan if real.
  • Scope: Part A is sizable (tx refactor of addCustomer + updateCustomerFromForm + cascade + backfill + UI + rules). The plan may split Part A and Part B into separate execution phases; both still ship under one combined deploy.
