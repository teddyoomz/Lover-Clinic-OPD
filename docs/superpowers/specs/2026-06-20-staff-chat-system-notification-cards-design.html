Staff-Chat System Notification Cards — Design Spec

Date: 2026-06-20 · Status: Approved by user · Implementation plan: (to be linked by writing-plans)

Purpose / Problem

Today, when a customer finishes the intake form ("ลูกค้ากรอกข้อมูลรับเข้าเสร็จ") or a follow-up assessment ("ลูกค้ากรอกแบบประเมินติดตามเสร็จ"), the system fires a Web-Push notification only (FCM, on devices). Staff who don't have push, or who live inside the app, miss it. We want the same event to ALSO drop a beautiful "System" notification card into the per-branch staff chat — with a cool icon, the customer's name + HN, and a clickable name that opens that customer's detail page in a new tab (the same affordance used elsewhere in the app). For walk-in intake (no customer record exists yet), the card shows the name only and auto-upgrades to a clickable name + HN the moment an admin registers that walk-in — consistent across every device that views it.

Mockup Design

Grounded in the real staff-chat bubble (src/components/staffchat/StaffChatMessage.jsx): dark panel, sender-tinted bubbles, name styled, timestamp muted. The new System card is style A · full card with the sparkles icon, a fire-red identity accent, and a sky-blue clickable name (never red on a patient name — Thai culture).

Flow

Two trigger paths, one write site (the existing sendPushOnSubmit Cloud Function, right after the FCM send), and a render-time live-resolve on the client.

Server write flow — submit → push → chat card
Client render flow — live-resolve the clickable name
#StageWhat happens
1ListenerThe branch's listenToStaffChatMessages snapshot includes the system card (it has branchId + createdAt). It flows through the V82 unread cursor + the chat chime → counts unread + plays sound (Q4).
2Render branchStaffChatMessage: if (message.system) return <StaffChatSystemCard/> — an early branch before the normal bubble (read-only: no reply/delete affordance).
3Resolve customerIduseSystemCardCustomer(card): if system.customerId (follow-up) → use it; else (intake) getDoc(opd_sessions/{system.sessionId}) → brokerProClinicId. Memoized per card id; unresolved intake re-checks on re-render.
4Live name + HNWhen a customerId is known → getCustomer(id) → resolveCustomerDisplayName + resolveCustomerHN (V113 live-resolve — always current).
5RenderResolved → sky-blue clickable name + ↗ + HN pill, <a href="/?backend=1&customer=<id>" target="_blank" rel="noopener noreferrer"> (RecallRow pattern). Unresolved intake → nameSnapshot + "(รอลงทะเบียน)", no link.
6Auto-upgradeAdmin registers the walk-in → handleOpdClick stamps opd_session.brokerProClinicId → next chat re-render, stage 3 resolves → the same card becomes clickable + shows HN ("เห็นพร้อมกันทุกที่").

Architecture

Write side — functions/

  • NEW functions/staffChatNotify.js — pure buildStaffChatNotification({kind, session, sessionId, customer, branchId}) returns the Firestore doc; + writeStaffChatNotification(db, BASE_PATH, doc) (admin-SDK setDoc at ${BASE_PATH}/be_staff_chat_messages/{id}). Mirrors the pure-builder pattern of notificationContent.js so it's unit-testable.
  • functions/index.js sendPushOnSubmit: right after the FCM sendEachForMulticast, determine kind, build + write the card. Wrapped in its own try/catch (non-fatal — a chat-write failure must never affect the push). Skip on edit (s.updatedAt).
// card doc (admin SDK bypasses the create validators)
{
  id: 'CHAT-<ts>-<hex>',           // crypto-secure id
  branchId,                          // session.branchId (fallback customer.branchId)
  deviceId: 'system',
  displayName: 'ระบบ',
  text: '🔔 <thai headline>',       // graceful fallback (chat-list lastMessage / non-card readers)
  createdAt: serverTimestamp(),
  system: {
    kind: 'intake' | 'followup',
    sessionId,                       // for intake live-resolve
    customerId: <id> | null,         // followup: linkedCustomerId; intake: null
    nameSnapshot,                    // followup: resolved name; intake: patientData name
    hnSnapshot: <hn> | null,         // followup: resolved HN; intake: null
  }
}

Render side — client

  • NEW src/components/staffchat/StaffChatSystemCard.jsx — style A. Theme-aware via clinic tokens (--tx-primary, --tx-muted, --bd), fire-red identity accent (icon circle + left border), sparkles lucide icon, sky-blue clickable name. Headline by system.kind.
  • NEW src/lib/staffChatNotifyResolve.js — pure resolveSystemCardCustomerId(card, getSession) + a small hook useSystemCardCustomer(card) with a module-level cache (resolved customerId/name/HN). Reads via scopedDataLayer getCustomer + a session getter.
  • src/components/staffchat/StaffChatMessage.jsx — add the early if (message.system) branch at the top of the render.

Alert, scope, rules

  • Alert (Q4): the card is a normal branch message → flows through the existing V82 unread cursor + chime → counts unread + plays sound. Only requirement: confirm nothing excludes system messages from the cursor/chime path.
  • Scope: fresh intake + fresh follow-up only (skip edits). Read-only card. The V161 retention cron cleans system cards like any message (acceptable — they're transient notifications). Per-branch via branchId.
  • firestore.rules: NO change. Admin-SDK write bypasses the create validators; staff read already allowed; the render reads opd_sessions + be_customers which staff can already read.

Decisions (locked via Q&A with user)

  • Q1 = A: full notification card (prominent, "noti แบบสวยงาม").
  • Q2 = sparkles: the system identity icon.
  • Q3 = name-only-until-registered, then live-resolve: intake shows name only (no link/HN) until the admin registers the walk-in; then the same card live-resolves to a clickable name + HN, consistent everywhere. Resolution key = opd_session.brokerProClinicId via the stored sessionId (no PII stored; no registration-flow change).
  • Q4 = counts unread + plays the chat sound (even though the push already alerts on devices).
  • Defaults (approved): skip edits (push only); read-only card; getDoc+cache resolution (flips on re-render, not a live per-card listener).

Acceptance Criteria

  1. A finished follow-up assessment writes a System card into the customer's branch chat: sparkles icon, headline "กรอกแบบประเมินติดตามเสร็จแล้ว", sky-blue clickable name + HN pill; clicking opens /?backend=1&customer=<id> in a new tab.
  2. A finished intake writes a System card into the session's branch chat: headline "กรอกข้อมูลรับเข้าเสร็จแล้ว", name shown, no link/HN, "(รอลงทะเบียน)".
  3. After an admin registers that intake (handleOpdClick → brokerProClinicId stamped), the SAME intake card becomes clickable + shows HN on the next chat render — for every device/viewer.
  4. The System card counts toward the chat unread badge AND plays the chat notification sound.
  5. An edit (re-submit of an existing session) does NOT create a card; the push still fires as today.
  6. The customer name renders sky-blue, never red (Thai culture); HN is a neutral pill.
  7. The push notification is unchanged; the chat write is non-fatal (a write failure never blocks the push).
  8. The card appears ONLY in its own branch's chat (per-branch isolation).

Risks / Open Questions

  • session.branchId presence: verify the opd_session doc the function reads carries branchId; fallback to customer.branchId (follow-up) — confirm during implementation.
  • Resolution cost: unresolved intake cards getDoc(opd_session) on re-render; bounded (few intake cards) + memoized once resolved. Live per-card listener is a possible future upgrade (not in scope).
  • Cloud Function deploy: the write side requires firebase deploy --only functions; the client render ships via vercel. No firestore.rules change → no Probe-Deploy-Probe.
  • Rule Q verification: L2 real-prod e2e (admin-SDK write at the canonical path → client-SDK branch query → renders + resolves) + L1 (clickable name opens the customer page). Functions builder unit + render RTL (both kinds, resolved/unresolved) + Rule I flow-sim.
  • Edge — intake with no eventual registration: the card stays name-only forever (correct — no customer exists). Retention cron eventually removes it.
