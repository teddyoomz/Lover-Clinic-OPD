ED Score Box + Follow-up Assessment (Link/QR) — Design Spec

Date: 2026-06-15 · Status: Pending user review of spec · Feature: A new "สมรรถภาพ · ED Score" box in CustomerDetailView (right column, below the 4-tab course box) that shows the customer's ED assessment scores (ADAM / IIEF-5 / MRS / PE) across rounds, with a button that opens a round-aware modal to generate a link + QR sending the customer a follow-up assessment to fill remotely. Implementation plan: TBD (writing-plans next).

Purpose / Problem

ED/hormone patients are assessed at intake (PatientForm: ADAM/IIEF-5/MRS/PE) and the doctor wants to re-assess at follow-up visits to track progress over time. Today there is no longitudinal view and no round tracking: assessment answers live as a singleton on be_customers.patientData, and any follow-up submit overwrites the prior answers — round-1 is lost (this is the same class as the ภูดิท LC-26000151 incident: the source opd_session was deleted and history was unrecoverable).

This feature adds (1) a display box showing every assessment round with the latest as the hero, and (2) a round-aware follow-up sender (link + QR) so the customer fills the next assessment remotely — with results stored durably so history survives session cleanup.

Must work immediately with CURRENT data (zero migration)

Every customer who already did the ED assessment at intake (perf fields present on be_customers.patientData — including the 28 AV194-backfilled + LC-26000082 + any future intake) MUST show as "ครั้งที่ 1" in the box instantly, with no backfill/migration. Achieved by deriving round-1 as a virtual record read live from patientData (no copy into the new collection required).

Mockup Design

App primary theme (dark fire/ember) shown; light theme maps automatically via the same CSS variables (--bg-surface / --bd / --tx-*) the existing cards already use. Card styling copied verbatim from the real 4-tab course box (CustomerDetailView.jsx:1187).

ED Score in the หมายเหตุ — CDV strips, TFP shows a clean latest-2 (de-duplication + doctor context)

The ED Score box owns the scores. The verbose baked ED text must leave the หมายเหตุ — but the doctor still wants ED context while writing in TFP. So:

  • CDV หมายเหตุ: strip the ED block entirely (the box right there owns it).
  • TFP "หมายเหตุทั่วไป": strip the verbose baked ED text AND show a clean "ED Score — 2 ครั้งล่าสุด + วันที่" read-only block, so the doctor sees the trend while writing the OPD card. Sourced from the SAME assessment data the box uses (assessmentRoundsCore) — not the old baked note text.

Mechanism (grounded)

ED scores are baked into a stored string, not live-rendered: kioskPatientToCanonical.js:95 → generateClinicalSummary(...) intake output includes "ผลการคัดกรองอาการ / Clinical Screening Results" (utils.js:407-422 TH / :469-483 EN) → stored at be_customers.patientData.note (:185). In TFP it renders as customerNote in the amber "หมายเหตุทั่วไป" card (TreatmentFormPage.jsx:3705-3722, a <pre>); CDV shows the same note. Existing customers already have the ED block baked in.

Approach (non-destructive, zero migration)

  1. Stop baking going forward: omit the screening section from the note-building path (includeScreening:false on generateClinicalSummary for the note, OR strip in kioskPatientToCanonical). New customers' note has no ED block. (OPD print + intake-view keep the screening section — generateClinicalSummary default unchanged for those callers.)
  2. Display-strip for existing data: a pure helper stripScreeningSection(note) removes the screening block (header → next separator) when customerNote renders in CDV + TFP. Zero migration; does NOT mutate the stored clinical record.
  3. TFP clean ED block (NEW): in the "หมายเหตุทั่วไป" card (TreatmentFormPage.jsx:3705), below the stripped note, render a compact read-only "ED Score" block = the latest 2 assessment rounds (date + ADAM/IIEF/MRS/PE per round) from assessmentRoundsCore. Doctor sees the trend inline while writing. (Empty if the customer has no ED data.)

Decisions (locked via Q&A with user)

  • Q1 = A: NEW be_assessments collection (durable; round-1 virtual from patientData; survives session cleanup — fixes the LC-151 data-loss class).
  • Q2 = A + refinement: latest round = hero (primary display); full history expandable below.
  • Q3 = A: button → modal shows "ครั้งที่ N" + doctor picks which assessment types → generate link + QR; customer fills remotely.
  • Q3 refinement (QR): QR has a full-screen mode — tap the modal QR (or "QR เต็มจอ" button) → QR fills the screen width, optimized for the customer to scan from the staff's phone on mobile. Generated high-res (reuse generateQrDataUrl @600px).
  • Q4 = A + refinements: per-round link, expires +1 day OR on submit; rounds are deletable; round# is derived (delete renumbers, no skip — for mis-fill recovery).
  • Refinement (current data): works immediately, zero migration for all customers with intake ED data (round-1 read live from patientData).
  • Default A (materialize): via extending the onPatientSubmit Cloud Function (deploy functions at ship).
  • Default B (chip): if the latest round didn't include a type, the chip shows that type's most-recent prior value + "(ครั้งที่ X)" tag — no data hidden.
  • ED in หมายเหตุ: CDV หมายเหตุ strips ED (box owns it). TFP "หมายเหตุทั่วไป" strips the verbose baked ED AND shows a clean "ED Score — 2 ครั้งล่าสุด + วันที่" (doctor sees while writing). Stop-baking forward + display-strip existing (non-destructive). Kept in OPD print + intake-view.
  • Multi-type link: a follow-up session carries types[] so one link/QR can present several assessments (e.g. ADAM+IIEF) — PatientForm gate extended to read types[].
  • TFP layout: DX/Tx/Plan textareas 2× taller (rows 3/3/2 → 6/6/4); "ข้อมูลสุขภาพลูกค้า" enlarged to fill the void; the two save buttons stay row-aligned (existing grid-stretch + mt-auto/flex-1 mechanism, tuned). Pixel-verified via preview (both themes).
  • Audit: run the audit design skill on the resulting TFP (a11y / theming both themes / responsive / alignment / contrast → fix P0-P3) per user request.

Acceptance Criteria

  1. A customer with intake perf fields shows "ครั้งที่ 1" in the box with correct ADAM/IIEF/MRS/PE scores — with NO migration run (read live from patientData). Verified against a real current customer (e.g. LC-26000082 → ADAM 4/10 Positive).
  2. Doctor clicks "ส่งแบบประเมินติดตาม (ครั้งที่ N)" → modal shows the correct derived N → picks types → a be_assessments pending round + opd_session + scannable QR + copyable link are produced.
  3. Customer opens the link (anon, real browser) → sees only the picked assessment sections → submits → within seconds the box shows the new round as the hero with computed scores (materialized into be_assessments, survives opd_session deletion).
  4. Deleting the latest follow-up round renumbers so the next send is the same N again (no skip). Deleting a middle round re-ranks. Round-1 (intake) cannot be deleted via the box.
  5. Link expires after 1 day or once the customer submits (a stale/used link shows expired, not a fillable form).
  6. Box renders correctly in BOTH dark (fire/ember) and light themes; customer name/HN never rendered red (Thai culture).
  7. On mobile, the QR can be enlarged to full screen-width and is verified scannable by a real phone camera held up to it (high-res, sufficient quiet-zone/contrast).
  8. be_assessments anon-write → 403 (Probe). Branch switch does not hide/leak assessments (universal collection).
  9. The verbose ED screening block no longer appears in the หมายเหตุ in CDV + TFP (NEW + existing customers, via display-strip); the rest of the note (chief complaint, history) is preserved intact. ED stays in OPD print + intake-view.
  10. TFP "หมายเหตุทั่วไป" shows a clean "ED Score — 2 ครั้งล่าสุด + วันที่" block for a customer with ED data (e.g. LC-26000082 → ครั้งที่ 1 · 20/05/2569 · ADAM 4/10); empty for a customer with no ED data.
  11. TFP DX/Tx/Plan textareas are visibly ~2× taller; "ข้อมูลสุขภาพลูกค้า" is enlarged; the teal "บันทึกข้อมูลซักประวัติ" and purple "บันทึกสำหรับแพทย์" buttons sit on the SAME row with no awkward empty gap — verified by a real preview screenshot in BOTH dark and light themes.
  12. The audit design skill run on the resulting TFP reports no P0/P1 (a11y, theming, responsive, alignment, contrast); P2/P3 triaged.

  • Assessment question text duplication (Rule C1): question labels are embedded in PatientForm + AdminDashboard render sections. If the box's history detail re-renders answers, consider extracting to lib/assessmentQuestions.js (flagged tech-debt; not required for v1 — box shows scores, not per-answer).
  • Chip "latest-per-type vs latest-round-only": default B (latest-per-type + tag) chosen; confirm it reads clearly for the clinician at spec review.
  • Verification (Rule Q): ship requires L1 real-browser (customer fills link → box updates) + L2 (be_assessments round-trip on real prod, materialize survives session delete). Not mock-only.
  • หมายเหตุ removal scope — RESOLVED: remove the ED screening section ONLY from the CDV + TFP หมายเหตุ note (display-strip); keep it in OPD print + intake-view for the doctor's records. Existing stored notes are NOT mutated (display-strip, not Rule M rewrite — confirmed).
