Whole-App Performance Audit & Optimization — Design Spec

Date: 2026-07-06 · Status: Approved by user (design conversation) · Implementation plan: 2026-07-06-performance-audit-optimization.html · Mode: Ultracode (audit/verify via Workflow fan-out; implement inline per V81/V86 lesson)

Purpose / Problem

User directive (verbatim): "ไล่ audit ความเร็ว ความลื่น การโหลดเว็ปแอปของเรา การแสดงผลของ UI ให้ลื่น ไม่กระตุก มีประสิทธิภาพ ใช้ทรัพยากรเครื่องน้อย … ทั้งเว็ป ทุกหน้า ทุกซอกมุมให้ลื่นไหลที่สุด … ปรับปรุงแบบเห็นผล และไม่ก่อให้เกิดความเสี่ยงหรือบั๊คใดๆใหม่ ต้องการโหลดเร็ว ลื่น และยังสวยงามเหมือนเดิม"

Known starting facts (explored 2026-07-06):

  • BackendDashboard chunk ≈ 933 KB, AdminDashboard ≈ 383 KB — every backend visit pays a ~1 MB parse.
  • Giant modules: AdminDashboard.jsx (~8k lines, single component state), TreatmentFormPage.jsx (~5k+), backendClient.js (~10k+ imported by nearly everything).
  • Heavy deps whose laziness must be PROVEN, not assumed: fabric, html2canvas, jspdf, qrcode, three (lazy ✓), signature_pad, cheerio/firebase-admin (server-only — must be absent from client bundle).
  • Dozens of concurrent onSnapshot listeners per page — re-render-storm risk on the monolithic dashboards.
  • Existing tooling ready: build:analyze (visualizer), knip (dead code), Playwright, Chrome MCP (Rule S), 17,245-test suite.

Architecture

New artifacts (tooling only — no product code in Phase 0):

  • scripts/perf-baseline.mjs — Playwright + CDP: enumerates every surface (frontend queue + all backend tabs + TFP + 4 customer links + chat + reports), collects Artifact-1 metrics, writes docs/perf/<runId>/*.json.
  • scripts/perf-compare.mjs — merges runs → Artifact-2 HTML report at docs/perf/report.html.
  • scripts/perf-visual-parity.mjs — screenshot set + pixel-diff gate, both themes.
  • Bundle manifest — parse ANALYZE=1 output → per-chunk gzip table committed alongside each run.

Execution split (Ultracode): audit + verification = Workflow fan-out (read-only agents, adversarial verify per finding); implementation = inline (V81/V86: subagents thrash on this baseline). Rule P applies to every real finding (class-of-bug grep before fixing one instance).

Hard constraints: zero visual change (pixel-diff gate) · zero behavior change (17,245 tests + Rule Q L1/L2) · no Firestore rules changes · no schema changes · business logic untouched · every fix carries a regression lock · iron-clad A: any fix that breaks anything → revert immediately, don't patch forward.

Out of scope: visual redesign, feature changes, service-worker/offline caching (stale-bundle risk — separate decision if ever wanted), server/API latency work beyond preconnect (Vercel/Firestore infra unchanged).

Decisions (locked via Q&A with user)

  • Q1 = C: Aggressiveness — everything measurable, including data-loading restructure (Phase 3), accepting multi-session duration. Every tier passes full verification.
  • Q2 = A: Metrics — full before/after measurement on EVERY surface (Lighthouse-class metrics + bundle + interaction profiling), numeric report proves "เห็นผล".
  • Q3 = A: Priority — no user-specified pain points; the Phase-0 audit ranks the punch-list by measured impact.
  • Approach = A: Measurement-first phased campaign (attribution + small blast radius per round) over big-bang or page-by-page.
  • Ultracode: user-invoked — Workflow fan-out for audit + adversarial verification; inline implementation.

Acceptance Criteria

  1. Baseline JSON exists for every enumerated surface (prod + local-preview) before any product-code change.
  2. Consolidated report shows measured improvement on the headline metrics (JS transferred, LCP, tab-switch, long-tasks) for the surfaces the audit targeted — improvements are claimed ONLY from re-measured numbers, never from reasoning (Rule Q-honest).
  3. Visual parity: pixel-diff of every surface × both themes shows no perceptible change after every phase.
  4. Full vitest suite green + build clean after every phase; every optimization has a regression lock (test or source-grep).
  5. No new bugs: Rule Q L1 real-browser pass on key flows after each phase; any breakage → iron-clad A revert.
  6. Final report committed at docs/perf/report.html; user L1 hands-on confirms perceived smoothness.

Risks / Open Questions

  • Chunk-split regressions: per-tab lazy chunks change load order — must keep Rolldown happy (Phase-29 recall-chunk panic precedent) and keep BS-4/listener wiring intact. Mitigation: one tab at a time, build + targeted tests per split.
  • Virtualization vs UI fidelity: virtualized lists can subtly change scroll behavior/selection. Mitigation: pixel-parity + interaction L1 per list; skip any list where parity can't be held.
  • Measurement noise: single-run metrics are noisy → each run = median of 3; prod measured off-peak.
  • Phase 3 blast radius: query-shape changes touch money-adjacent paths → each one gets Rule Q L2 with the exact UI query + Rule I flow-simulate; anything uncertain is dropped, not risked.
  • Obfuscator interaction: filler formula obfuscation must keep its narrow include (widening breaks 3D lazy chunk — locked lesson 2026-06-20).
