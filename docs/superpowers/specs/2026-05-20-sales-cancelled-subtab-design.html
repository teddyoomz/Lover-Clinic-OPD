Sales Cancelled Sub-Tab Design Spec

Date: 2026-05-20 · Status: Approved by user ("ok go") · Scope: single file src/components/backend/SaleTab.jsx · Implementation plan: (to be created by writing-plans skill)

Purpose / Problem

On the tab=sales page (ขาย / ใบเสร็จ) cancelled sales (status === 'cancelled', label "ยกเลิก") are interleaved with active sales in one list. In the user's branch the cancelled rows dominate the list (screenshot: 9 of 11 rows are "ยกเลิก"), burying the live receipts. The user wants cancelled sales moved out of the main list into a separate in-page sub-tab so the default view is clean.

User directive (verbatim): "ในหน้า tab=sales ฝากให้แยกการขายที่ยกเลิกแล้วไปใน Tab ย่อยต่างหาก ซึ่งเราจะสร้าง tab ย่อยนั้นมาในหน้านี้เลย ง่ายดี".

Architecture

Single-file change: src/components/backend/SaleTab.jsx. No backend, no new fetch, no BSA change. getAllSales({ branchId }) already returns all statuses for the selected branch; the split is purely client-side over the already-loaded sales array.

  1. State — add const [subTab, setSubTab] = useState('active') (values: 'active' | 'cancelled'; default 'active').
  2. Sub-tab definition — small SALE_SUB_TABS = [{id:'active',label:'การขาย'},{id:'cancelled',label:'ยกเลิกแล้ว'}]. Pill row mirrors the StockTab segmented pattern (rose-700 active / muted inactive), placed as the first child above the existing header card.
  3. Filter logic — extend the filtered useMemo: split first by subTab (active → s.status !== 'cancelled'; cancelled → s.status === 'cancelled'), THEN apply existing text search on both; apply the payment-status filterStatus dropdown only on the active tab. Add subTab to the deps array.
  4. Dropdown — render the status <select> only when subTab === 'active'; its options exclude the cancelled entry (e.g. PAYMENT_STATUSES.filter(s => s.value !== 'cancelled')). On switching to the cancelled tab, reset filterStatus = '' so a stale payment filter can't carry over.
  5. Empty states — active tab keeps the existing two empty states (onboarding when sales.length === 0; "ไม่พบรายการที่ตรงกับตัวกรอง" when filter empty). cancelled tab shows a new "ยังไม่มีรายการที่ยกเลิก" empty state.
  6. Untouched — cancel/edit/view/print handlers, the cancel modal, the + ขาย create flow, getAllSales, branch-scope, and the {filtered.length} รายการ count line (now reflects the selected sub-tab). The ✕ cancel action stays gated by sale.status !== 'cancelled' (unchanged), so it never appears on the cancelled tab.

Decisions (locked via Q&A with user)

  • Q1 = A: Two sub-tabs — "การขาย" (non-cancelled only) + "ยกเลิกแล้ว" (cancelled only). Cancelled rows move entirely out of the main list.
  • Q2 = A: Active tab keeps the status dropdown but drops the "ยกเลิก" option; cancelled tab hides the dropdown; search box stays on both tabs.
  • Q3 = B: No count badge on the sub-tab pills — labels only; the existing "{N} รายการ" line under the search shows the selected tab's count.
  • Assumption (confirmed at design review): default sub-tab = "การขาย"; + ขาย button kept on both tabs; cancelled rows keep 👁 view / 🖨 print / ✎ edit actions.

Acceptance Criteria

  1. On load the page opens on "การขาย" showing only non-cancelled sales; cancelled sales do NOT appear.
  2. Clicking "ยกเลิกแล้ว" shows only status === 'cancelled' sales for the current branch.
  3. The status dropdown is visible only on "การขาย" and contains no "ยกเลิก" option; it is hidden on "ยกเลิกแล้ว".
  4. Text search works on both tabs and filters within the active sub-tab only.
  5. Switching from "การขาย" (with a payment filter set) to "ยกเลิกแล้ว" resets the payment filter; switching back does not show a stale filter.
  6. Cancelling a sale from "การขาย" moves that row out of "การขาย" and into "ยกเลิกแล้ว" after the list reloads.
  7. "ยกเลิกแล้ว" with zero cancelled sales shows "ยังไม่มีรายการที่ยกเลิก"; "การขาย" with zero sales overall keeps the existing onboarding empty state.
  8. The "{N} รายการ" count reflects the selected sub-tab's filtered count.
  9. Switching the top-right branch reloads sales and both sub-tabs reflect the new branch (existing branch-scope behaviour preserved).
  10. Rule Q V66: verified via L1 (Playwright real browser) or L2 (real client SDK) before any "verified" claim — not mock-only.

Testing strategy

  • RTL/unit — pill row renders 2 tabs; default active; split correctness (active excludes cancelled, cancelled excludes non-cancelled); dropdown conditional render + cancelled option absent; search per tab; filterStatus reset on tab switch; both empty states.
  • Rule I flow-simulate — load → switch tab → search → cancel a sale → reload → row migrates active→cancelled; count line tracks tab.
  • Rule Q L1/L2 — real-browser or real client-SDK exercise on real prod (TEST-SALE-prefixed fixtures per V33.12) before claiming verified.
  • Source-grep regression — lock the split predicate + dropdown-conditional + reset-on-switch so future edits can't silently re-merge cancelled into the main list.

Risks / Open Questions

Low risk — client-side split over already-loaded data, single file, no wiring change. Watch items: (1) the filterStatus-reset-on-switch must fire so a stale payment filter never silently empties a tab; (2) the count line must read the post-split list (already handled since it reads filtered.length); (3) ensure the active tab's "no results" vs "no sales at all" empty states still discriminate correctly when all of a branch's sales happen to be cancelled (active tab should show "ไม่พบ…"/onboarding appropriately, not break). No open questions — all four decision points are locked.
