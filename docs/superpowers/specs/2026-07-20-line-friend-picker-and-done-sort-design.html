LINE Friend Picker (Hybrid + Real-time) + Done-tab Sort — Design Spec

Date: 2026-07-20 · Status: Approved by user ("โอเค") · Implementation plan: 2026-07-20-line-friend-picker-and-done-sort.html (writing-plans next)

Purpose / Problem

Feature ① — LINE Friend Picker. ปัจจุบันผูก LINE target สำหรับ noti สุขภาพระบบไม่ได้เพราะ admin หา LINE User ID ไม่ได้ (ต้องพิมพ์ Uxxxx… มือ) และการผูกลูกค้าเข้ากับ LINE ต้องให้ลูกค้าพิมพ์ ผูก <ID> + เลขบัตรประชาชนเอง ต้องการ: ดึงรายชื่อ + userId + รูปของคนที่แอด LINE OA แต่ละสาขามาแสดงเป็นลิสต์ให้ กดเลือก ได้ ทั้งสำหรับ (ก) ผูกเจ้าของรับ noti สุขภาพระบบ และ (ข) ผูกลูกค้าเข้าข้อมูลลูกค้าโดย admin โดยตรง — และลิสต์ต้อง real-time: ลูกค้าแอดหรือทักหน้างานปุ๊ป ชื่อโผล่ในลิสต์ปั๊บ แม้เปิด modal ค้างไว้ ไม่ต้อง refresh.

ข้อจำกัด LINE ที่ตรวจแล้ว: GET /v2/bot/followers/ids (รายชื่อผู้ติดตามทั้งหมด) ใช้ได้เฉพาะ OA แบบ verified / premium เท่านั้น (โล่น้ำเงิน/เขียว) — บัญชี unverified เรียกแล้วได้ 403. ส่วน GET /v2/bot/profile/{userId} (ชื่อ+รูปจาก userId) ใช้ได้ทุกบัญชี. OA ของคลินิกยังไม่แน่ใจว่าเป็นแบบไหน → สถาปัตยกรรมแบบ Hybrid ที่ตรวจเองอัตโนมัติ.

Feature ② — Done-tab sort. แท็บ "วันนี้" sub-pill "เสร็จแล้ว" ปัจจุบันเรียงตามเวลานัด (sortApptsByDateTimeAsc) — ต้องการเรียงตาม "คนที่เพิ่งกดรับบริการเรียบร้อยล่าสุดอยู่บนสุด" (serviceCompletedAt desc). กำลังรอ + แท็บอื่นเรียงแบบเดิม.

Flow

Flow A — Real-time roster: แอด/ทักปุ๊ป → โผล่ในลิสต์ปั๊บ (ไม่มี refresh)

ลูกค้าแอดเพื่อน LINE OA→ LINE ส่ง follow event → webhook→ resolver หาสาขา (V75 เดิม) + profile API (ชื่อ+รูป)→ admin SDK เขียน be_line_friends/{branchId}_{userId}→ client onSnapshot listener ยิง→ แถวใหม่โผล่ใน picker ทันที (badge "เพื่อนใหม่")

ลูกค้าทักแชท→ webhook สร้าง/อัพเดท chat_conversations (pipeline เดิม — ไม่แตะ)→ listener ตัวที่สองยิง→ โผล่ใน picker (badge "เคยทัก")

เปิด modal→ one-shot /api/admin/line-friends?action=list (Followers API backfill — verified OA เท่านั้น; 403 = ข้ามเงียบ)→ merge 3 แหล่ง dedupe ด้วย lineUserId

Flow B — ผูกลูกค้า (mirror link-requests approve — canonical V32-tris-quater)
#ขั้นรายละเอียด
1admin กด "เลือก" ใน picker (โหมดผูกลูกค้า)โชว์ confirm: ชื่อไลน์ + userId vs ชื่อลูกค้า + HN
2ยืนยัน → POST /api/admin/line-friends action=bind{customerId, lineUserId, branchId, displayName}
3server: dedup guardquery be_customers where lineUserId == X → ถ้ามีลูกค้าอื่นผูกอยู่ → Thai error "userId นี้ผูกกับลูกค้า … อยู่แล้ว" (ไม่เขียนอะไร)
4batch atomic (admin SDK)lineUserId + lineLinkedAt + lineUserId_byBranch.{branchId} (dotted-path) + audit doc be_admin_audit
5push แจ้งลูกค้าใน LINE (best-effort)ข้อความเดียวกับ approve flow เดิม; fail = ไม่ block
6client: live customer listener (V36-quinquies) สะท้อนผลปุ่ม "ผูก LINE" → "LINE ✓" เอง ไม่ต้อง refresh
Flow C — Done-tab sort (Feature ②)

กด "รับบริการเรียบร้อย" ใน กำลังรอ→ markAppointmentServiceCompleted stamp serviceCompletedAt (V71 เดิม + optimistic stamp ใน HubView)→ แถวย้ายเข้า sub-pill "เสร็จแล้ว"→ NEW sortApptsByServiceCompletedDesc → แถวนั้นอยู่บนสุดทันที

ใช้เฉพาะ done sub-pill ของแท็บวันนี้ — กำลังรอ / พรุ่งนี้ / ล่วงหน้า / ย้อนหลัง ใช้ comparator เดิม (V64-fix9 / 2026-06-14 desc-past) ไม่แตะ. Comparator ต้อง timestamp-safe: Firestore Timestamp (.toMillis()) + optimistic Date + {seconds} shape + null → 0 (ลงล่างสุด).

Architecture

1. Webhook follow/unfollow capture — api/webhook/line.js: เพิ่ม handler สำหรับ event follow/unfollow (ปัจจุบันถูก ignore). follow: resolve branchId ด้วย resolveChatBranchIdFromLineEvent (V75 เดิม) → ดึง profile (ชื่อ+รูป) → เขียน be_line_friends/{branchId}_{userId} ผ่าน admin SDK: {lineUserId, displayName, pictureUrl, branchId, branchIdSource, followedAt, unfollowedAt: null, updatedAt}. unfollow: merge {unfollowedAt: ts} (soft — ไม่ลบ doc). ทั้งหมด best-effort try/catch แยกจาก message pipeline (ห้ามกระทบ chat เดิมแม้ throw). ห้าม downgrade doc ที่มีอยู่ (re-follow → เคลียร์ unfollowedAt + refresh ชื่อ/รูป).

2. firestore.rules — NEW match: be_line_friends { allow read: if isClinicStaff(); allow write: if false; } (client อ่านเพื่อ real-time listener; เขียนได้เฉพาะ admin SDK จาก webhook). Deploy = combined vercel + firestore:rules พร้อม Probe-Deploy-Probe (Rule B) + probe ใหม่: anon read/write → denied.

3. Client listeners (BSA เต็มรูป) — NEW listenToLineFriendsByBranch(opts, onChange, onError) ใน backendClient.js (Layer 1 safe-by-default ตาม BS-13: branchId ว่าง + ไม่ allBranches → resolveSelectedBranchId → ยัง ว่าง → onChange([]) + noop) + Layer 2 wrapper ใน scopedDataLayer.js. reuse listenToChatConversationsByBranch (V75 เดิม) เป็น listener ตัวที่สอง. Merge client-side ใน modal: dedupe ด้วย lineUserId (คนเดียวกันทั้งแอด+ทัก = แถวเดียว, badge "เคยทัก" ชนะ, ชื่อ/รูปใช้ตัวที่ใหม่กว่า). be_line_friends ลงทะเบียน COLLECTION_MATRIX (BC1.1) scope=branch.

4. NEW /api/admin/line-friends — auth gate เดียวกับ api/admin/link-requests.js (แบบเดียวกัน byte-for-byte pattern):

  • action=list {branchId} — Followers API backfill: อ่าน token จาก be_line_configs (fallback legacy chat_config นครราชสีมา — pattern V75) → GET /v2/bot/followers/ids (paginate) → ถ้า 403/ไม่มีสิทธิ์ → คืน {followersApi: 'unavailable', rows: []} (ไม่ error) → resolve profile เฉพาะ id ที่ client ยังไม่รู้จัก (ส่ง knownIds มา) แบบ bounded-parallel (concurrency 10, cap 300/ครั้ง) → module-cache 60s ต่อ branch (pattern /api/tfp-options)
  • action=bind {customerId, lineUserId, branchId, displayName} — mirror handleApprove ใน link-requests.js: dedup guard → batch atomic customer update (dotted-path lineUserId + lineLinkedAt + lineUserId_byBranch.{branchId}) + audit doc + LINE push แจ้งลูกค้า (best-effort). แหล่งผูกระบุใน audit: source: 'friend-picker'

5. LineFriendPickerModal.jsx (NEW, shared — กัน V12 sibling-drift): props {open, branchId, mode: 'pick' | 'bind', customer?, onPick, onClose}. ข้างใน: 2 listeners + one-shot backfill fetch + search + source filter + AV78 explicit-close + useModalScrollLock (AV205). unfollowed rows = จางลง + badge "เลิกติดตาม" (ยังเลือกได้ — userId ยัง valid สำหรับ push ไม่ได้ แต่โชว์ตรงๆ ให้ admin รู้).

6. จุดใช้ 2 ที่ — (ก) InfraHealthSection.jsx: ปุ่ม "เลือกจากรายชื่อ" ต่อแถว lineTargets → picker (mode=pick, branchId จากแถว) → เติม lineUserId + label(=displayName) ลง draft — client-side ล้วน, save path เดิม. (ข) LinkLineQrModal.jsx (หน้าข้อมูลลูกค้า): เพิ่มแท็บ "เลือกจากรายชื่อเพื่อน" ข้าง QR เดิม → picker (mode=bind) → confirm → bind action → live listener สะท้อนผล. Flow เดิม (QR + ผูก <ID> + LinkRequestsTab) คงไว้ทั้งหมด.

7. Feature ② — NEW sortApptsByServiceCompletedDesc ใน appointmentHubFilters.js (pure, timestamp-safe helper svcCompletedMs) — ใช้ใน AppointmentHubView เฉพาะ (แท็บวันนี้ + sub-pill เสร็จแล้ว); จุดอื่นทั้งหมดใช้ comparator เดิม.

Decisions (locked via Q&A with user)

  • Q1 = C (Hybrid): Followers API ก่อน (verified OA = รายชื่อครบ) → 403 → fallback webhook roster อัตโนมัติ — ไม่ต้องรู้ประเภท OA ล่วงหน้า
  • Q2 = ไม่แน่ใจ (ประเภท OA): hybrid ตรวจเองตอนเรียก; UI ไม่ error เมื่อ unavailable
  • Q3 = ทั้ง 2 ที่: การ์ดสุขภาพระบบ (เติม target) + หน้าข้อมูลลูกค้า (ผูกตรงโดย admin, mirror approve semantics + audit); flow เดิมคงไว้เป็นช่องทางสำรอง
  • Q4 = เก็บ follow/unfollow: webhook เริ่มเก็บตั้งแต่ deploy — จำเป็นสำหรับ OA unverified + real-time "แอดปุ๊ปโผล่ปั๊บ"
  • Q5 (follow-up) = Real-time: ลิสต์เป็น onSnapshot listeners (ไม่ใช่ endpoint fetch) — แอด/ทักหน้างานโผล่ทันทีแม้เปิด modal ค้าง → ต้องเปิด read rule be_line_friends ให้ isClinicStaff → deploy รอบนี้ = combined + Probe-Deploy-Probe (user รับทราบ "โอเค")

Acceptance Criteria

  1. เปิด picker ค้างไว้ → ลูกค้าทัก LINE OA สาขานั้น → แถวใหม่โผล่ในลิสต์เอง ≤ ~3 วิ ไม่ต้อง refresh (badge "เคยทัก")
  2. เปิด picker ค้างไว้ → ลูกค้าแอดเพื่อน (follow) → แถวโผล่เอง (badge "เพื่อนใหม่") — ได้ผลกับคนที่แอดหลัง deploy
  3. OA verified → เปิด modal เห็นรายชื่อผู้ติดตามทั้งหมด (backfill); OA unverified → ไม่มี error ใดๆ ลิสต์ = เคยทัก + แอดใหม่
  4. การ์ดสุขภาพ: กดเลือก → lineUserId + ป้าย ถูกเติม → save → "ทดสอบแจ้งเตือน" ส่งถึงคนนั้นจริง
  5. ผูกลูกค้า: confirm → be_customers ได้ lineUserId + lineUserId_byBranch.{branchId} + audit doc; userId ที่ผูกกับลูกค้าอื่นอยู่แล้ว → Thai error, ศูนย์ write
  6. Flow เดิมทั้งหมดไม่แตก: QR/ผูก <ID>/LinkRequestsTab + chat pipeline (follow handler เป็น best-effort — throw ไม่กระทบข้อความ)
  7. แท็บวันนี้·เสร็จแล้ว: กด "รับบริการเรียบร้อย" → แถวขึ้นบนสุดของเสร็จแล้วทันที (optimistic); เรียง serviceCompletedAt ใหม่→เก่า; กำลังรอ + แท็บอื่นเรียงเดิมเป๊ะ
  8. Security: anon อ่าน/เขียน be_line_friends → denied; staff อ่านได้; write client → denied (Rule B probe ใหม่)

Risks / Open Questions

  • Followers API อาจคืนผู้ติดตามหลักพัน → profile-resolve ถูก cap (300/ครั้ง + concurrency 10 + cache 60s) — รายชื่อ backfill ที่เกิน cap จะทยอยครบเมื่อเปิดซ้ำ; รายชื่อ live (ทัก/แอด) ไม่ถูก cap
  • คนที่แอดไว้ก่อน deploy และไม่เคยทัก: มองไม่เห็นจนกว่าจะทัก หรือ OA เป็น verified (ข้อจำกัด LINE — แจ้ง user แล้ว)
  • LINE rate limit ต่อ profile API — bounded-parallel + cache ป้องกัน; follow event = 1 call/คน ไม่มีปัญหา
  • ชื่อ/รูปใน chat_conversations vs be_line_friends อาจต่างเวอร์ชัน (ลูกค้าเปลี่ยนชื่อ) → merge ใช้ตัวที่ updatedAt ใหม่กว่า — display เท่านั้น ไม่กระทบ identity (lineUserId คือ key เสมอ)
  • rules deploy = Probe-Deploy-Probe เต็มขั้น (Rule B) — probe list เดิม + probe ใหม่ #20 be_line_friends
