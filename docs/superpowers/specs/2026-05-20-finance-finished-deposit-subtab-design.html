Finance Finished-Deposit Sub-Tab Design Spec

Date: 2026-05-20 · Status: Approved by user (Q1-Q4 locked + verbal go) · Scope: src/components/backend/DepositPanel.jsx + new pure helper · Sibling: mirrors the sales cancelled sub-tab (2026-05-20-sales-cancelled-subtab-design.html)

Purpose / Problem

On tab=finance → "มัดจำ" (DepositPanel), finished deposits (used / cancelled / refunded / expired) are interleaved with still-usable ones, cluttering the list the user works in day-to-day. Mirroring the just-shipped sales cancelled sub-tab, split finished deposits into a separate in-page pill so the default "ใช้งานอยู่" view shows only deposits that still have a usable balance.

User directive: "ในหน้า tab=finance ฝากให้แยกมัดจำที่ใช้หมดแล้วออกไป โดยสร้าง Tab ย่อยใหม่ขึ้นมา ในลักษณะเดียวกับ tab=sales เมื่อกี้เลย" (then chose: move all finished statuses, not just used).

Architecture

One new pure helper + edits to DepositPanel.jsx. Client-side split over the already-loaded deposits (one-shot getAllDeposits); no backend, no new fetch, no BSA change. Mirrors saleSubTabFilter.js + the SaleTab pill pattern (emerald accent for deposits).

  1. NEW src/lib/depositSubTabFilter.js — ACTIVE_DEPOSIT_STATUSES = ['active','partial'], FINISHED_DEPOSIT_STATUSES = ['used','cancelled','refunded','expired'], isFinishedDeposit(dep), filterDepositsBySubTab(deposits, subTab).
  2. State — subTab ('active' default) + handleSubTabChange that resets filterStatus = '' (the dropdown option sets differ per tab, so a stale value must not carry over).
  3. filteredDeposits — split by subTab via the helper FIRST, then apply existing filterStatus + search + date-range; add subTab to deps.
  4. Pill row (emerald) above the toolbar: [ ใช้งานอยู่ ] [ สิ้นสุดแล้ว ].
  5. Scoped dropdown — options = (subTab === 'active' ? ACTIVE_DEPOSIT_STATUSES : FINISHED_DEPOSIT_STATUSES) mapped through STATUS_META, kept visible on BOTH tabs (Q4). Search + date-range + "สร้างมัดจำ" + count line kept on both.
  6. Empty states — finished empty → "ยังไม่มีมัดจำที่สิ้นสุด"; active: deposits.length===0 → existing EmptyState; filtered-empty with a filter → existing "ไม่พบ…"; filtered-empty with no filter → "ยังไม่มีมัดจำที่ใช้งานอยู่".
  7. Untouched — create/use/refund/cancel/edit/delete handlers, modals, branch-scope, getAllDeposits, the 4 finance entity sub-tabs.

Decisions (locked via Q&A with user)

  • Q1 = A: Pill row INSIDE DepositPanel (mirrors sales placement); the 4 finance entity sub-tabs are unchanged.
  • Q2 = B: Finished pill = used + cancelled + refunded + expired; active pill = active + partial.
  • Q3 = A: Labels "ใช้งานอยู่" / "สิ้นสุดแล้ว".
  • Q4 = A: Scoped status dropdown kept on BOTH pills (active → ใช้งาน/ใช้บางส่วน; finished → ใช้หมด/ยกเลิก/คืนเงิน/หมดอายุ); search + date-range on both.
  • Reactivity: verify-first (Rule Q L1) — list updates without F5 via reload-after-action + re-mount-on-navigation; add onSnapshot only if a real stale gap is found.

Acceptance Criteria

  1. Default opens on "ใช้งานอยู่" showing only active+partial deposits.
  2. "สิ้นสุดแล้ว" shows only used+cancelled+refunded+expired for the current branch.
  3. Status dropdown visible on both pills, scoped to the pill's status set.
  4. Search + date-range work on both pills, within the pill's split.
  5. Switching pills resets the status filter (no stale value).
  6. A deposit fully-consumed/cancelled/refunded/expired migrates from ใช้งานอยู่ → สิ้นสุดแล้ว after reload.
  7. Empty states: finished-empty → "ยังไม่มีมัดจำที่สิ้นสุด"; active no-deposits → existing onboarding.
  8. Count line reflects the selected pill's filtered count.
  9. Branch switch reloads deposits; both pills reflect the new branch.
  10. Rule Q V66: verified via L1 (real browser) before any "verified" claim.

Risks / Open Questions

Low risk — client-side split over loaded data, mirrors the shipped sales pattern. Watch: (1) filterStatus reset-on-switch (different option sets per tab); (2) the active-tab empty-state must discriminate "no deposits at all" (onboarding) vs "all finished" (no-filter message) vs "filter no match"; (3) date-range + search must compose with the split. No open questions.
