Real-time intake notification on appointment cards

Date: 2026-05-26 (EOD+7) · Status: Approved by user (direction "ไม่เพิ่มหน้าจอใหม่ — ใช้การ์ดนัดหมายเดิม" + "ok" on 3-part approach) · Origin: /systematic-debugging Phase 1 root cause · Plan: (writing-plans next)

Purpose / Problem

เมื่อลูกค้ากรอก patient form ผ่านลิงก์ที่ส่งจากการ์ดนัดหมาย (OPD link, hideFromQueue:true → card-flow), การ์ดในหน้า "นัดหมาย" ไม่อัปเดต real-time (ต้อง F5) และ ไม่มี blue bubble + เสียง. Push มือถือก็ไม่เข้า. ระบบนี้เคยใช้ได้ตอนยังมี tab "คิวหน้าคลินิก".

Root cause (verified, file:line):

  • ① Real-time การ์ด: hub re-fetch ผูกกับ be_appointments เท่านั้น (AdminDashboard.jsx:1165 bump appointmentDataVersion); form-fill เขียนแค่ opd_sessions (PatientForm.jsx:374) → ไม่ bump. และ resolveLinkedSession (:3793) สำหรับ card-flow ตกไปใช้ stale one-shot getDoc cache (:3797–3811) ที่ไม่เคย invalidate เมื่อ session เปลี่ยน.
  • ② Blue bubble + เสียง: card-flow sessions (createdFromBackendBooking && isHiddenFromQueue, set ที่ appointmentDepositBatch.js:1049,1058) ถูก exclude จาก data/ndData (:2305/2329/2354) → ไม่อยู่ใน allNotifData (:2392) → detection (:2398–2443) ไม่เห็น → toast+เสียงไม่ยิง. (หมายเหตุ: V124 แก้ "purple count badge" ให้ count ได้แล้ว แต่ blue toast+เสียงยังไม่ครอบคลุม card-flow.)
  • ③ Push: คนละเส้น — functions/index.js sendPushOnSubmit ถูกเรียกโดย PatientForm.jsx:380 เสมอ (server-side, ไม่ถูก exclude). Rule R diag: 7 tokens, globalPushMuted=false, แต่ token ทุกตัวเก่า ~2 เดือน (2026-03-21..26) → น่าจะ stale/หมดอายุ หรือ device ปัจจุบันไม่ได้ register token ใหม่.

Decisions (locked via Q&A with user)

  • Placement = ไม่เพิ่มหน้าจอใหม่ — surface บนการ์ดนัดหมายเดิม (user: "ใน tab นัดหมายก็มี Card list ของลูกค้าคนนั้นอยู่สักที่อยู่แล้ว ไม่อยู่ในคิววันนี้ ก็คิวล่วงหน้า"). ปฏิเสธ option เพิ่ม pill/tab/drawer.
  • 3-part fix = approved ("ok"): ① real-time linked-session live · ② blue bubble+เสียง สำหรับ card-flow · ③ push self-heal.
  • Scope = appointment-linked flow เท่านั้น (เคสที่ลูกค้ามีนัด). intake link แบบไม่ผูกนัด (ถ้ามี) bubble/เสียง global เด้งอยู่แล้ว แต่ไม่มีการ์ด — นอกขอบเขต.
  • Bubble clickable → jump to การ์ด = optional (เพิ่มได้ถ้าถูก ไม่เพิ่ม scope มาก; default: bubble แสดงชื่อ).

Acceptance Criteria

  1. ลูกค้ากรอกฟอร์ม (ลิงก์จากการ์ด) → การ์ดเป็น "📥 ลูกค้ากรอกแล้ว · รอบันทึก" ภายใน ~1–2 วินาที โดยไม่ต้อง F5 + purple pending-OPD count เด้ง +1.
  2. Blue bubble + เสียง เด้งตอนกรอก (เมื่อ isNotifEnabled เปิด) + ไม่เด้งรัว (dedup) + ไม่เด้งซ้ำตอนเปิดหน้า (first-load stamp).
  3. Push: app-load re-register token เมื่อ lc_push_enabled; user re-enable แล้วได้รับ push จริง (verify post-deploy).
  4. ไม่มี perf regression — listener ยัง read-only (ไม่เพิ่ม write/cascade; V34/V36 lesson).
  5. ไม่กระทบ flow อื่น (queue filters / deposit / V125 cancel-cascade / V124 purple count) — เป็น additive.

Risks / Open Questions

  • Perf: listener pull ทุก session ใน branch อยู่แล้ว — map เป็น read-only (ไม่เพิ่ม read/write). ต้องไม่ re-introduce cascade (V36).
  • resolveLinkedSession reactivity: ต้องมั่นใจ AppointmentHubView re-render เมื่อ linkedSessionsById เปลี่ยน (mirror lazyFetchedTick pattern; ระวัง React.memo บน rows).
  • Push validity: token 2-เดือน-เก่า — ยืนยัน valid ไม่ได้ถ้าไม่ส่ง push จริง (ไม่ทำ unprompted). user verify post-deploy.
  • Verification (Rule Q): real-time + bubble = ต้อง L1 real-browser (Chrome MCP, Rule S) ด้วย real opd_sessions write (TEST-prefixed fixture) ก่อน claim; push = user hands-on.
