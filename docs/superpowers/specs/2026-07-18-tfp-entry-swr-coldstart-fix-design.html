TFP Entry SWR Cold-Start Fix — Design Spec

Date: 2026-07-18 · Status: Approved direction (Q1=ก / Q2=A / Q3=A) — awaiting spec review · Implementation plan: 2026-07-18-tfp-entry-swr-coldstart-fix.html

Purpose / Problem

User report: เปิดหน้า TFP (TreatmentFormPage) บนเครื่องคลินิก (WiFi คลินิก) เจอ spinner "กำลังโหลดฟอร์มการรักษา..." หมุนค้างนานมาก — บน 5G เร็ว, หน้าอื่นเร็วหมด, ช้าเฉพาะ TFP, และเครื่องที่เปิด TFP บ่อยทั้งวันคือเครื่องที่ช้า.

Root cause (วัดจริงบน LIVE prod, Playwright + admin-SDK diag — 2026-07-18):
  1. Architectural: TFP เป็นหน้า staff หน้าเดียวที่ first paint ผูก network 100% — ทุกการเปิดดึง 6 collections (products 354 + courses 239 + doctors + staff + dfGroups + dfRates ≈ 600 docs / 520-630KB) จาก server ก่อนปิด spinner. TFP หลุดจาก AV206 SWR sweep (ไม่อยู่ทั้ง ADOPT/SANCTIONED — classifier ไม่ scan ไฟล์นี้).
  2. Cache eviction บนเครื่องใช้หนัก: working set ทั้งแอป ≈ 17.6MB raw ≈ ~44MB ใน IndexedDB ชนเพดาน default 40MB → LRU เขี่ย query targets ของ TFP ระหว่างวัน → เปิดซ้ำ = cold pull เต็ม.
  3. WiFi คลินิกคุณภาพต่ำเป็นตัวคูณ: วัดจริง — cold pull: 2.4s (เน็ทดี) / 7.2s (1.5Mbps/200ms) / 23.8s (0.4Mbps/500ms); congestion + loss จริงคูณต่อเป็น 30-60s+. Warm (resume-token delta ~4KB) ≤3.2s เสมอ.
ไม่ใช่: การโดน block (เครื่องเดียวกัน IP เดียวกันเร็ว; Firestore ไม่มี IP block) · ไม่ใช่ cookie (Firestore ใช้ IndexedDB; การล้าง cache/temp ยิ่งแย่ลงเพราะบังคับ cold pull).

Architecture

  1. ชั้น 1 — SWR-adopt TFP (ไฟล์หลัก src/components/TreatmentFormPage.jsx): แยก body ของ load-form effect (บรรทัด ~753-1068) เป็น fetchFormData(source) ภายใน effect เดิม แล้ววิ่งผ่านแพทเทิร์น swrRun ที่มีอยู่ (src/lib/swrRead.js): cache-paint → server-correct. getters ที่รองรับ {source:'cache'} แล้ว: listProducts / listCourses / listStaff / listDoctors (grep-verified); ต้องขยาย 4 ตัว: listDfGroups + listDfStaffRates (ผ่าน _getDocsBySource เดิม) และ getCustomer + getTreatment (เพิ่ม single-doc variant _getDocBySource แพทเทิร์นเดียวกัน). Apply → setState เท่านั้น (AV206.c). Edit-mode getTreatment ร่วม SWR ด้วย (server correction ทับก่อน user แตะฟอร์ม). แสดง sync chip มุมหัวฟอร์มระหว่างรอ server (reuse ภาษา/สไตล์ SyncIndicator ของ hub).
  2. Save-gate (Q2=A): effect เก็บ serverFreshRef.current = serverPromise; handleSubmit await ก่อนเข้าเส้นเขียน (bounded 15s — เกินนั้นปล่อยไป ให้ transaction ฝั่ง server เป็นผู้ตัดสิน/แจ้ง error จริง). ปุ่มบันทึกใช้สถานะ saving เดิม ไม่มี UI ใหม่.
  3. ชั้น 2 — cache cap (src/firebase.js): cacheSizeBytes: 200 * 1024 * 1024 (ไม่ใช้ UNLIMITED — คง GC เป็น backstop กัน IDB โตไม่จำกัดข้ามปี).
  4. ชั้น 3 — idle prefetch (Q3=A): module src/lib/tfpPrefetch.js — ยิง 4 listers (server source, Promise.allSettled, ครั้งเดียว/session ผ่าน module flag) หลัง shell mount + idle ~3-5s. Mount จุดเดียวต่อ shell: BackendDashboard + AdminDashboard (staff-authed เท่านั้น; ไม่แตะ customer-facing routes).
  5. ชั้น 4 — institutional lock: ขยาย AV206 classifier (tests/instant-coldstart-av206-classifier.test.js) ให้ scan mount-blocking one-shot loads ทั้ง src/components/** + src/pages/** (ไม่ใช่เฉพาะรายชื่อ ADOPT) — ไฟล์ที่มี pattern ต้องอยู่ใน inventory (ADOPT/SANCTIONED) ไม่งั้น fail. เพิ่ม TFP เข้า ADOPT table ใน docs/perf/swr-inventory.md.

Decisions (locked via Q&A with user)

  • Q1 = ก: Defense-in-depth 3 ชั้น (SWR-adopt TFP + cache cap 200MB + classifier expansion) — ไม่เอา minimal (ข) / listener-ize (ค).
  • Q2 = A: Save-gate แบบรอเงียบๆ อัตโนมัติ (await serverFreshPromise, bounded 15s, ใช้ saving state เดิม).
  • Q3 = A: Idle prefetch products+courses+DF หลัง shell paint (~3-5s, ครั้งเดียว/session) เพื่ออุ่นเครื่อง cache เย็นก่อนเปิด TFP แรกของวัน.

Acceptance Criteria

  1. Warm-cache TFP open (in-app, ไม่ reload): ฟอร์มโผล่จาก cache ≤1.5s แม้ throttle 400kbps/500ms — วัดด้วย probe เดิม (scripts/diag-tfp-cold-timeline.mjs WARM mode) ก่อน/หลัง.
  2. Cold-cache + prefetch: เปิด backend shell รอ ≥10s แล้วเข้า TFP ครั้งแรก → พฤติกรรมเท่า warm (prefetch อุ่นให้แล้ว).
  3. Rule Q L2: re-run diag probes บน LIVE prod — TFP reopen ดึง Firestore ≤ หลักสิบ KB (resume delta) ไม่ใช่ 630KB.
  4. Rule Q L1: Playwright บน LIVE prod + throttle เทียบ before/after + screenshot eyeball (Q-vis); ตามด้วย user L1 บนเครื่องคลินิกที่ช้าจริง.
  5. ไม่มี regression เส้นเขียน: full vitest เขียว; save produces docs เหมือนเดิม; AV206.c classifier เขียว (cache ไม่ถึง write โดยไม่ผ่าน save-gate); empty-cache ไม่ paint ฟอร์มเปล่า.
  6. Classifier ใหม่จับของจริง: ลบ TFP ออกจาก inventory → test ต้องแดง (พิสูจน์ scan ทำงาน).

Risks / Open Questions

  • Stale-price display window (วินาทีเดียว): ราคา/รายการจาก cache อาจเก่าชั่วครู่ก่อน server ทับ — semantics เดิมของฟอร์มคือ snapshot-at-pick อยู่แล้ว + เงิน/สต็อค validate ใน tx ฝั่ง server (Rule T) → ความเสี่ยงจริง ≈ 0 แต่บันทึกไว้ตรงนี้.
  • iOS IDB eviction หลังทิ้งแอปนาน: degrade กลับเป็นพฤติกรรม server-first วันนี้ (ครั้งเดียวแล้วอุ่น) — ไม่แย่กว่าเดิม (precedent AV206).
  • Prefetch bandwidth: ~630KB × 1 ครั้ง/session ทุกเครื่อง staff — ยอมรับ (เครื่องคลินิกเปิด TFP ตลอดวันอยู่แล้ว; mobile staff ก็ประหยัดกว่าที่ต้อง cold-pull ตอนเปิดจริง).
  • เพดาน 200MB: ถ้า working set โตเกินในอนาคต (หลายปี) LRU กลับมา churn — เฝ้าด้วย diag scripts/diag-staffapp-working-set.mjs (เก็บเป็น recurring health check).
  • Multi-tab primary-throttle stall (ยังไม่ได้พิสูจน์ — Q-honest): เป็นไปได้ว่าเครื่องคลินิกเปิดหลาย tab แล้ว primary tab โดน background-throttle ทำ query ค้างเพิ่ม — fix ชุดนี้ลดผลกระทบ (paint ไม่รอ network แล้ว) แต่ไม่ได้แก้กลไกนั้นตรงๆ; ถ้า user L1 หลัง deploy ยังเจอค้าง → รอบสอง investigate multi-tab โดยเฉพาะ.
