ProductFormModal — หมวดหมู่ Datalist Enrichment

Date: 2026-06-03 · Status: Approved by user (design) · Scope: 1 file (src/components/backend/ProductFormModal.jsx) — covers tab=products, tab=stock, tab=central-stock · Implementation plan: docs/superpowers/plans/2026-06-03-product-category-datalist.html

Purpose / Problem

ในโมดัล "แก้ไขสินค้า" / "เพิ่มสินค้า" ช่อง หมวดหมู่ เป็น <input list="product-group-list"> (datalist combobox) อยู่แล้ว แต่ตัวเลือกใน datalist ดึงมา เฉพาะ master be_product_groups (ตัวแปร groups) ซึ่งมักว่าง/บางตา → ผู้ใช้รู้สึกว่า "ไม่มี dropdown ให้เลือก". ในทางกลับกัน ช่อง หน่วย ทำงานดีเพราะ datalist ของมัน (unitDatalistOptions) merge 2 แหล่ง: master be_product_units + หน่วยที่มีอยู่จริงบนสินค้าใน be_products (harvest distinct mainUnitName).

เป้าหมาย: ให้หมวดหมู่ได้ "harvest + merge" เหมือนหน่วยเป๊ะ — ตัวเลือก = master groups + หมวดหมู่ที่มีอยู่จริงบนสินค้า (distinct categoryName) — โดยคง widget เดิม (<input list> = พิมพ์เลือก หรือ พิมพ์หมวดใหม่ได้, ไม่ทำของเดิมพัง). อ่านอย่างเดียว (read-time enrich) — ไม่ migrate/seed master.

Flow

Data-flow: modal mount → merge → datalist render (NO new Firestore read — listProducts already fetched)
1modal mount — existing useEffect Promise.all([listProductGroups, listProductUnitGroups, listProducts])
→
2harvest distinct categoryName from p (the already-fetched products) → productCategories state (mirror of productUnits, lines 39-49)
→
3categoryDatalistOptions useMemo: master groups first, then product-harvested not in master (dedup)
→
4render PLAIN <option value> (no source tag) in existing #product-group-list + data-testid="product-group-datalist"
→
5user picks existing or types new → form.categoryName → save (unchanged)

Dashed green box = reuses data already in memory. The whole change is additive — no read, no write-path, no validation change. Save/normalize (validateProduct → categoryName: trim(...)) untouched.

Decisions (locked via Q&A with user)

  • Q1 = A: Widget = datalist combobox (เหมือนหน่วยเป๊ะ) — เลือก/พิมพ์ใหม่ได้, non-breaking. (Rejected B: strict <select> — would block inline add-new + change workflow.)
  • Q1-bis (user revision 2026-06-03): dropdown แสดง ชื่อหมวดหมู่ล้วน เท่านั้น — เอา tag "master" / "มีในสินค้าแล้ว" ออก. การ merge ทั้ง 2 แหล่งยังทำเบื้องหลัง (master + product-harvest, dedup) แต่ render เป็น <option value> ล้วน (ไม่มี source field, ไม่มี data-source ที่หน่วยใส่ไว้). หมวดหมู่จึงสะอาดกว่าหน่วยเล็กน้อย — ความต่างนี้ผู้ใช้มองไม่เห็น (native datalist แสดงแค่ value อยู่แล้ว).
  • Scope: read-time enrich only — NO migration/seeding into be_product_groups master (mirror หน่วย, which is read-time only).
  • Rule C1 watch: unit + category = 2 near-identical "master + product-harvest merge" blocks. Keep as direct mirror now (proven หน่วย code, lowest risk; master shapes differ — u.units[].name vs groups[].name). Extract a shared helper on the 3rd occurrence.

Acceptance Criteria

  1. Opening "แก้ไขสินค้า" (and "เพิ่มสินค้า") from tab=products / tab=stock / tab=central-stock shows a หมวดหมู่ datalist containing both master be_product_groups names and every distinct non-empty categoryName currently on be_products (e.g. "อุปกรณ์ทั่วไป").
  2. Options are deduped (master wins) and a value appearing in both shows once.
  3. User can still type a brand-new category not in the list, and save it (no validation regression).
  4. An existing product whose category is off-master is preserved on edit/save (no clobber, no blank).
  5. No additional Firestore read vs. before (the existing listProducts() is reused).
  6. หน่วย behavior is unchanged (no regression to the reference field).

Risks / Open Questions

  • Low risk: additive UI over data already in memory; no read/write/rules/validation change. Single shared modal → no V12 multi-site drift.
  • Honest Rule Q scope: merge logic is pure JS → unit-tested + RTL (datalist renders harvested options) + source-grep regression. The "real harvest from real prod be_products" is exercised by the same already-shipped listProducts() path used by หน่วย; the L1 "open modal → click หมวดหมู่ → see real category in list" is user hands-on (or a quick live-browser check). No new compound query / index / rule → no L2 Firestore-query verification needed beyond the existing path.
  • Not a bug-class (feature): Rule P expansion not triggered; normal feature test bank suffices.
