Date: 2026-06-03 · Status: Approved by user (design + 2 revisions) · Scope: 1 file (src/components/backend/ProductFormModal.jsx) — covers tab=products, tab=stock, tab=central-stock · Implementation plan: docs/superpowers/plans/2026-06-03-product-category-datalist.html

Purpose / Problem

ในโมดัล "แก้ไขสินค้า" / "เพิ่มสินค้า" ช่อง หมวดหมู่ เป็น <input list="product-group-list"> (datalist combobox) อยู่แล้ว แต่ตัวเลือกใน datalist ดึงมา เฉพาะ master be_product_groups (ตัวแปร groups) ซึ่งมักว่าง/บางตา → ผู้ใช้รู้สึกว่า "ไม่มี dropdown ให้เลือก". ส่วนช่อง หน่วย ทำงานดีเพราะ datalist ของมัน harvest หน่วยที่มีอยู่จริงบนสินค้า (distinct mainUnitName จาก be_products).

เป้าหมาย: ให้หมวดหมู่ harvest "หมวดหมู่ที่มีอยู่จริงบนสินค้า" (distinct categoryName จาก be_products) มาเป็นตัวเลือก dropdown — เหมือนวิธีที่หน่วย harvest จากสินค้า แต่ ไม่ใช้ master be_product_groups เลย (ผู้ใช้สั่ง 2026-06-03: "ไม่ต้องมี master เหี้ยไรทั้งนั้น"). คง widget เดิม (<input list> = พิมพ์เลือก หรือ พิมพ์หมวดใหม่ได้, ไม่ทำของเดิมพัง). อ่านอย่างเดียว — ไม่ migrate/seed.

Flow

Data-flow: modal mount → harvest → datalist render (NO new Firestore read — and listProductGroups removed)
1modal mount — useEffect Promise.all([listProductUnitGroups, listProducts]) (listProductGroups REMOVED — master not used)
→
2harvest distinct categoryName from p (the already-fetched products) → productCategories state (mirror of the productUnits harvest, lines 39-49)
→
3categoryDatalistOptions useMemo: productCategories only → trim · dedup · Thai-locale sort (no master)
→
4render PLAIN <option value> (no tag) in existing #product-group-list + data-testid="product-group-datalist"
→
5user picks existing or types new → form.categoryName → save (unchanged)

Dashed green box = reuses data already in memory. The change is additive on the consumer side and removes one now-unused read (listProductGroups) + the dead groups state. No write-path, no validation change — save/normalize (validateProduct → categoryName: trim(...)) untouched.

Decisions (locked via Q&A with user)

  • Q1 = A: Widget = datalist combobox — เลือก/พิมพ์ใหม่ได้, non-breaking. (Rejected strict <select> — would block inline add-new + change workflow.)
  • Q1-bis (user revision 2026-06-03): dropdown แสดง ชื่อหมวดหมู่ล้วน — เอา tag "master" / "มีในสินค้าแล้ว" ออก (render plain <option value>, no data-source ที่หน่วยมี).
  • Q1-ter (user revision 2026-06-03 — "ไม่ต้องมี master เหี้ยไรทั้งนั้น"): ไม่ใช้ master be_product_groups เลย. ตัวเลือก = distinct categoryName บน be_products เท่านั้น. listProductGroups import + groups state ถูกลบออกจาก ProductFormModal (dead-code cleanup). หน่วยไม่เปลี่ยน (ยัง master + product).
  • Scope: read-time harvest only — NO migration/seeding.
  • Rule C1 watch: หน่วย และ หมวดหมู่ ต่าง harvest distinct values จาก be_products (mainUnitName / categoryName). 2 occurrences → keep inline now; extract a tiny shared "distinct trimmed values" helper on the 3rd. (หน่วย ยัง merge master; หมวดหมู่ ไม่ merge.)

Acceptance Criteria

  1. Opening "แก้ไขสินค้า" (and "เพิ่มสินค้า") from tab=products / tab=stock / tab=central-stock shows a หมวดหมู่ datalist containing every distinct non-empty categoryName currently on be_products (e.g. "อุปกรณ์ทั่วไป") — and NOTHING from master be_product_groups.
  2. Options are deduped (a category used by many products shows once), Thai-locale sorted, and rendered as plain <option value> (no tag).
  3. User can still type a brand-new category not in the list, and save it (no validation regression).
  4. An existing product whose category is off-list is preserved on edit/save (no clobber, no blank).
  5. No additional Firestore read — reuses the existing listProducts(); the now-unused listProductGroups() call + groups state are removed (one fewer read).
  6. หน่วย behavior is unchanged (no regression to the reference field).

Risks / Open Questions

  • Low risk: additive UI over data already in memory; no read/write/rules/validation change (net one fewer read). Single shared modal → no V12 multi-site drift.
  • Master-only categories don't appear (by user directive): a category that exists in be_product_groups but is not yet used on any product won't be in the list. Non-issue — the combobox lets the admin type it; it then saves on the product and appears thereafter (self-bootstrapping).
  • Honest Rule Q scope: harvest/dedup/sort logic is pure JS → unit-tested + RTL (datalist renders harvested options, none from master) + source-grep regression (incl. anti-regression that listProductGroups/groups are gone). The "real harvest from real prod be_products" runs through the same already-shipped listProducts() path หน่วย uses; L1 "open modal → click หมวดหมู่ → see real category, no master" = user hands-on (or a quick live-browser check). No new compound query / index / rule.
  • Not a bug-class (feature): Rule P expansion not triggered; normal feature test bank suffices.
