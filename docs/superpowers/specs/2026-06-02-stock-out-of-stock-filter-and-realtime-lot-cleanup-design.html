Stock — "หมด" filter + real-time lot auto-clear · Design Spec

Date: 2026-06-02 · Status: Approved by user (design Q&A locked) · Page: tab=stock → ยอดคงเหลือ (StockBalancePanel) + backendClient stock-mutation paths · Implementation plan: docs/superpowers/plans/2026-06-02-stock-out-of-stock-filter-and-realtime-lot-cleanup.html

Purpose / Problem

Two user-reported issues on tab=stock → ยอดคงเหลือ:

  1. No "หมด" filter. The balance table has 4 filter checkboxes (ใกล้หมดอายุ / ใกล้หมด / เกินสต็อก / ติดลบ) but none for fully out-of-stock products — even though a "หมด" badge already renders at totalRemaining === 0. User: "มี filter เยอะแยะ แต่ไม่มี filter สินค้าที่หมดแล้ว".
  2. Spent 0-lots linger until the nightly cron. A lot drained to 0 is dead weight and clutters the per-product lot list. The V143-quater planLotCleanup already removes redundant 0-lots, but only on the 03:45 BKK cron. User wants it real-time — "lot ที่เป็น 0 … เป็น 0 แล้วมันไม่ได้ใช้อะไรแล้ว … เมื่อใดมันกลายเป็น 0 ปุ๊บ ลบไปเลย", clarified: "มันเป็น 0 ได้ ถ้ามี lot เดียว แต่ถ้ามี lot อื่นเข้ามา lot ที่เป็น 0 จะต้องหายไป".

The clarification is the key design fact: the rule is EXACTLY the existing planLotCleanup — keep one 0-lot when a product is fully drained (placeholder so it still shows at "0 / หมด", honoring prior user directive V143/AV166 "ต้องแสดงจำนวนเสมอแม้เป็น 0"); delete every 0-lot the moment a live lot exists. So no panel rewrite is needed — the placeholder keeps drained products visible, which is exactly what makes the new "หมด" filter meaningful.

Architecture

Part 1 — "หมด" filter (StockBalancePanel.jsx, pure UI): new state showOutOfStockOnly, a 5th checkbox data-testid="filter-out-of-stock" after ติดลบ, predicate isOutOfStock(p) = Number(p.totalRemaining) === 0 (matches the line-414 badge logic; negative is excluded — covered by ติดลบ), wired into the displayed useMemo + its deps. ~8 LOC. No data/listener change.

Part 2 — real-time auto-clear (backendClient.js): NEW _clearRedundantZeroLotsForProducts(affectedKeys) — for each (productId × locationId): query be_stock_batches (status ∈ {active,depleted}) for that group, call the pure planLotCleanup (stockLotCleanupCore.js, unchanged), delete the resulting deleteIds via a chunked writeBatch. DELETE-only on remaining === 0; idempotent; never reads/writes a live lot. Each of the 8 entry points collects the (productId × location) keys it touched and calls the helper after its transaction commits, wrapped in try/catch (non-critical side-effect). The pure planLotCleanup + the 03:45 cron + the Rule-M CLI all stay as-is.

Part 3 — existing lingering 0-lots (Rule M): run the existing scripts/stock-lot-cleanup.mjs dry-run on prod → review count + affected products → --apply on user go (two-phase + audit doc). No new script.

Invariants honored: Rule O (productId identity; movements carry live productName, unaffected by lot delete) · V143/AV166 (drained product stays visible via placeholder) · V36 anti-multi-writer-sweep (AV169 source-grep covers all 8 entry points) · Rule M (Part 3 two-phase) · Rule Q (L2 real-client-SDK verify, not mock).

Decisions (locked via Q&A with user)

  • Placeholder rule (user clarification): keep 1 zero-lot when fully drained; delete all zero-lots when a live lot exists. = existing planLotCleanup. No panel rewrite, no products-first.
  • Q1 = A: real-time trigger = post-commit hook at all 8 stock-mutation entry points + keep the 03:45 cron as backstop + AV169 source-grep lock. (Inside-transaction rejected — higher V36 risk.)
  • Q2 = A: clean the current lingering 0-lots now via the existing Rule-M script — dry-run first, review, then --apply.

Acceptance Criteria

  1. "หมด" filter ON → shows ONLY products with totalRemaining === 0; negative (<0) products do NOT appear (they're under ติดลบ); positive products do NOT appear.
  2. Import a 2nd lot into a product that has a 0-lot → the 0-lot is deleted within the same operation (no reload) — verified on real prod via L2 client SDK.
  3. FIFO-drain an older lot to 0 while a newer lot is live → the drained lot is deleted; the product's Batches count drops.
  4. Drain the LAST lot to 0 → exactly ONE 0-lot remains (placeholder); the product still appears in the table at "0 / หมด".
  5. A negative (AUTO-NEG debt) lot is NEVER deleted by the helper.
  6. Deleting a 0-lot does NOT break the Movement Log (movements read be_stock_movements, carry their own productName per Rule O) — verified.
  7. AV169 source-grep passes: every exported stock-mutation entry point calls _clearRedundantZeroLotsForProducts (or is annotated exempt).
  8. Full vitest green + build clean; no regression in stock test banks (V34/V35/V36/V42–49/V143).

Risks / Open Questions

R1 — Movement Log reader. Must confirm no UI reader JOINs a deleted 0-lot's be_stock_batches doc for display (expected: movements are self-contained per Rule O). Verified during implementation; if a reader does join, fall back to keeping the lot + flag.

R2 — Multi-writer-sweep (V36 class). Mitigated by AV169 source-grep over all 8 entry points + the nightly cron backstop. A future new entry point that forgets the call → lingering lots until 03:45 (graceful), AND fails AV169.

R3 — Concurrency. Helper is read-then-delete-zeros; a concurrent mutation racing it can only mean a 0-lot is deleted twice (idempotent / already gone) or skipped (cron catches). It never deletes a live lot (re-reads remaining at delete time is unnecessary because planLotCleanup only targets remaining===0, and a lot at 0 that concurrently gains stock would just be skipped on the next pass).

R4 — Verification is L2, not L1. The auto-clear is a backend behavior best driven by the real client SDK (exact functions the UI calls). The "หมด" filter is a UI add verifiable via the panel. Full L1 real-browser of the filter is a nice-to-have; L2 of the cleanup is the gold standard here.
