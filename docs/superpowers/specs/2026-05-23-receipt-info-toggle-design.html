Purpose / Problem

User request (verbatim 2026-05-23 EOD+1 LATE+2):

"งั้นทำระบบ Toggle ในหน้า Preview ใบเสร็จ/ใบเสนอราคา ให้เลือกได้ว่าจะแสดงที่อยู่ หรือแสดงแค่ชื่อลูกค้า และ HN และ เบอร์โทร ซึ่ง Toggle ไว้แบบไหนพอกดพิมพ์ ก็ได้แบบนั้น ทำปุ่ม Toggle อยู่ด้านบน header ในหน้าพรีวิวแถวๆปุ่มพิมพ์เลยก็ได้"

Problem: Post-V113-C, every receipt with a customer that has populated patientData now shows the full receipt-info block (taxId + address + phone + name) via live-resolve from the customer master. This is the right default for tax-document customers, but many clinic receipts only need a minimal customer identifier (name + HN + phone) — full address on a flyaway receipt is over-disclosure and a PDPA concern.

Solution: Add an in-preview toggle that lets admin choose show full receipt-info block vs show compact name + HN + phone only. Toggle state persists per-device via localStorage so the admin's preference sticks across receipts. Print output matches the toggle state at the moment 🖨 is clicked.

Architecture

New file — src/hooks/useReceiptInfoToggle.js (~35 LOC)

import { useEffect, useState, useCallback } from 'react';

const STORAGE_KEY = 'lover_receipt_show_address';
const DEFAULT_SHOW = false; // Q3=B — PDPA-friendly default

function readStorage() {
  try {
    const raw = localStorage.getItem(STORAGE_KEY);
    if (raw === 'true') return true;
    if (raw === 'false') return false;
    return DEFAULT_SHOW;
  } catch { return DEFAULT_SHOW; }
}

export function useReceiptInfoToggle() {
  const [showAddress, setShowAddressState] = useState(readStorage);

  // Cross-tab sync (F5)
  useEffect(() => {
    function onStorage(e) {
      if (e.key === STORAGE_KEY) setShowAddressState(readStorage());
    }
    window.addEventListener('storage', onStorage);
    return () => window.removeEventListener('storage', onStorage);
  }, []);

  const setShowAddress = useCallback((next) => {
    const val = !!next;
    setShowAddressState(val);
    try { localStorage.setItem(STORAGE_KEY, val ? 'true' : 'false'); }
    catch { /* private-mode / quota — render still works, just not persisted */ }
  }, []);

  return { showAddress, setShowAddress };
}

Acceptance Criteria

  1. First open on a fresh browser/device → toggle is OFF → customer header shows นาย X Y + HN LC-xxx · โทร. xxxxxxxxxx (single line) → no receipt-info block.
  2. Click switch (or label "ที่อยู่") → toggle flips → full block appears (taxId + address + phone + name-if-different-from-customerName) via existing V113-C mergedReceiptInfo.
  3. Click 🖨 พิมพ์ in either state → printed PDF matches the current toggle state. The switch chrome itself NEVER appears on the printed page (verified via print:hidden Tailwind class on parent header bar).
  4. Close preview → open a different sale's receipt → toggle state matches what admin last chose (localStorage persistence).
  5. Open QuotationPrintView → same toggle state (shared key). Toggle in Quotation preview → SalePrintView opens with the same state.
  6. Edge case: customer has no phone → compact line shows HN LC-xxx alone (no trailing · dot, no empty โทร. label).
  7. Edge case: localStorage unavailable (private/incognito + quota error) → hook gracefully falls back to in-memory state; toggle still works during the session, just doesn't persist.
  8. Edge case: storage event from another tab → preview re-renders with the new state without manual refresh.

Out of Scope (explicitly excluded)

  • No backend writes — pure client UI state. No firestore.rules, no audit doc, no migration.
  • No per-customer or per-sale override — toggle is global (per-device) only.
  • Toggle does NOT affect other live-resolve fields: course names (V111), customer name (V113-A), receipt-info block CONTENT (V113-C) all keep their behavior — V114 just adds a SHOW/HIDE switch over the block.
  • No new printed-format option (e.g. "always show", "always hide", "ask each time") — toggle is binary persistent, period.
  • No backfill / one-time scripts — V112-B trap not repeated. Pure renderer-level change.
  • Toggle does NOT appear in any non-preview surface (e.g. SaleDetailModal, reports, exports). Just the 2 PrintViews.

Risks / Open Questions

  • Print CSS interaction: The existing header bar already has print:hidden on the sticky-top wrapper. The switch SHOULD inherit. Verification: Rule S print preview to confirm switch doesn't leak into printed page.
  • Switch styling consistency: Project doesn't have a canonical Switch component yet (DateField, ProductSelectField etc. exist; no switch). Should V114 introduce a reusable Switch.jsx, or inline the styles in the 2 PrintView headers? Recommendation: inline in the PrintView header for now (Rule C1 — 2 uses, not yet 3). Extract on the 3rd use.
  • Phone source resolution: Compact line reads mergedReceiptInfo.phone (V113-C output). This already walks the priority chain: snapshot wins, then liveCustomer.patientData.phone || liveCustomer.phone. No new helper needed.
  • Accessibility: Switch should have role="switch" + aria-checked + aria-label="ซ่อน/แสดงที่อยู่บนใบเสร็จ" for screen readers. Acceptance criterion #4 doesn't currently cover this — flag as a small a11y polish in the plan.
  • Quotation print signature line: QuotationPrintView has a "เรียน (Customer)" header (line ~221) with the same shape as SalePrintView. Verified during V113-C: same pattern, same hook works.
