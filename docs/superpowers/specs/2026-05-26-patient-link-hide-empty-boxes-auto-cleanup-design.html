Patient-Link: Hide Empty Boxes + Auto-Cleanup of Stale Links

Date: 2026-05-26 (EOD+5) · Status: Approved by user · Surface: Customer patient-link page (?patient=<token>, __customerMode) + new daily cron · Implementation plan: docs/superpowers/plans/2026-05-26-patient-link-hide-empty-boxes-auto-cleanup.html

Purpose / Problem

The customer patient-link page (the public ?patient=<token> view a customer opens without login) currently renders an ugly empty-state box "ไม่มีคอร์สคงเหลือ" when a customer has no remaining courses (see the user screenshot of 3cc66f7e…: 1 appointment + a big empty "no courses" box). The user wants only boxes that have data — hide the empty courses box, and likewise hide the appointments box when there are no appointments (the appointments box is already conditional — confirmed at PatientDashboard.jsx:882, no change needed there).

Second, links accumulate: staff generate a patientLinkToken per customer, and stale links (a customer with no upcoming appointment and no remaining courses) pile up. The user wants the system to auto-delete a link once it has shown nothing useful for over 1 month — if a customer later needs a link again, staff regenerate it from the customer-detail page. Goal: keep the active-link set from flooding the system.

Architecture

Shared core (Rule of 3 / C1): the "what does this link show" logic currently lives inline in api/patient-view.js:85-109. Extract it to a new pure module src/lib/customerLinkPayloadCore.js (no firebase imports) so the endpoint AND the cron compute "empty" identically:

  • computeUsableCourses(courses, todayISO) → { remaining, expired } — the deriveEffectiveStatus usable filter + expiry split.
  • isAppointmentUpcoming(appt, todayISO) — future-date + not-cancelled + not-serviced/attended (COMPLETED_APPT_STATUSES + serviceCompletedAt/wasServiceCompleted).
  • isCustomerLinkEmpty({ courses, appointments, todayISO }) → boolean — no usable non-expired course AND no upcoming appt. Expired courses do NOT count as "คอร์สคงเหลือ" and do NOT block deletion.

api/patient-view.js refactors to consume these (behavior-preserving; the existing e2e is the contract). The endpoint still owns the I/O (token resolve, branch-name resolve, Thai-date formatting) — only the filter logic moves to the core.

Cleanup decision core: decidePatientLinkCleanup(customerDoc, isEmpty, now) → { action: 'stamp'|'clear'|'delete'|'skip', patch } + constant PATIENT_LINK_EMPTY_GRACE_MS = 30 * 24 * 3600 * 1000. Lives in customerLinkPayloadCore.js (or a sibling cleanup-core file). Pure → fully unit-testable.

Cron: api/cron/patient-link-cleanup-sweep.js mirrors opd-session-cleanup-sweep.js exactly — CRON_SECRET gate, admin SDK, canonical artifacts/{APP_ID}/public/data paths, exported sweepPatientLinkCleanup({ db, now, apply }) (dry-run when apply=false), batch writes, audit doc. scripts/patient-link-cleanup-sweep.mjs is the Rule-M dry-run/apply mirror. vercel.json gets one daily cron entry ("30 21 * * *" = 04:30 BKK, off-peak after the other daily sweeps).

UI: PatientDashboard.jsx — derive isCustomerMode = !!sessionData?.__customerMode; gate the empty courses box (:914-919) with !isCustomerMode; add the subtle line + tx TH/EN strings. The admin/sync feedback boxes and the loading/noData fallbacks are untouched.

Decisions (locked via Q&A with user)

  • Q1 = A: Hide empty boxes in the customer link view only (__customerMode); keep the feedback boxes in the admin/sync view.
  • Q2 = B: When a customer has nothing at all → one subtle line "ยังไม่มีนัดหมายหรือคอร์สในขณะนี้" (not a bare page).
  • Q3 = A: Auto-delete trigger = empty for ≥ 30 days, tracked via patientLinkEmptySince (stamp on first-empty, clear on data-return).
  • Q4 = A: "Delete" = clear the token (true delete) + disable; customer sees the existing "ไม่พบข้อมูล"; staff regenerate.

Acceptance Criteria

  1. Customer link with appointment(s) but 0 remaining courses → NO "ไม่มีคอร์สคงเหลือ" box; appointment box renders. (the screenshot case)
  2. Customer link with remaining courses → courses box renders normally.
  3. Customer link with no appt + no remaining course + no expired course → exactly one subtle line "ยังไม่มีนัดหมายหรือคอร์สในขณะนี้"; no boxes.
  4. Admin/sync patient view with 0 courses → the "ไม่มีคอร์สคงเหลือ" feedback box STILL shows (regression guard).
  5. Cron: a customer empty for < 30 days → patientLinkEmptySince stamped, link still works.
  6. Cron: a customer empty for ≥ 30 days → patientLinkToken cleared + disabled; opening the old URL → 404; CustomerDetailView shows "สร้างลิงก์".
  7. Cron: a customer that regains data (new appt/course) before 30 days → patientLinkEmptySince cleared; link kept.
  8. Cron is idempotent (re-run yields 0 new writes once stable) and reconciles exactly with the endpoint's "empty" definition.
  9. Endpoint output unchanged after the helper extraction (existing e2e green).

Risks / Open Questions

  • Expired-only customers (flagged to user): a customer with only EXPIRED courses (no remaining, no appt) counts as "empty" → its link is cleaned after 30 days, even though the expired-courses box still renders until then. This matches the literal "ไม่มีคอร์สคงเหลือ" wording. If the user prefers keeping expired-only links alive, change isCustomerLinkEmpty to also require expired.length === 0.
  • No firestore.rules / index change → no Probe-Deploy-Probe. The be_appointments where customerId == query (single-field equality, JS-sorted) is already used by the endpoint; no new composite index.
  • Cron cost: appt query is bounded — only fires for customers with an enabled link AND empty courses. Enabled-link count is small (manual staff action). Trivial.
  • No notification on auto-delete (YAGNI) — audit doc records deletions; staff regenerate when a customer reports a dead link.
  • Verification scope (Rule Q-honest): UI is conditional-render only — RTL + build + L1 real-browser on the anon/public customer link (feasible for me at verify, Rule S). Cron is verified by pure-core unit + flow-simulate + an L2 real-prod dry-run (no mutation); --apply runs only post-deploy with user authorization (Rule M).
