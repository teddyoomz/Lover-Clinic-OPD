Frontend Tab Removal + Deposit-Aware Cancel Dialog

Date: 2026-05-26 · Status: Approved by user · Implementation plan: docs/superpowers/plans/2026-05-26-frontend-tab-removal-deposit-cancel-dialog.html (pending)

Purpose / Problem

Two connected asks on the Frontend admin page (src/pages/AdminDashboard.jsx) + the deposit-booking lifecycle:

  1. Remove 4 redundant tabs — คิวหน้า Clinic, จองไม่มัดจำ, จองมัดจำ, ประวัติ. Queue creation + viewing + history are all covered by the unified นัดหมาย (AppointmentHubView) surface + Backend now. Continues the V124–V126 strategic direction ("unify นัดหมาย, deprecate the queue tabs"); V125 cancel-cascade was the tactical first step.
  2. Deposit-aware cancel — today, cancelling a deposit-booking appointment in the นัดหมาย tab sets status='cancelled' + archives the linked opd_session but leaves the deposit money active in Finance·มัดจำ (orphan). The admin should be asked: "ลบตัวเงินมัดจำไปด้วยเลยไหม หรือลบแต่นัดหมาย".

Scope: single implementation plan. Frontend tab structure + a shared cancel-confirmation dialog wired into all cancel surfaces. No new Firestore collection, no rules/index change.

Decisions (locked via Q&A with user)

  • Q1 = นัดหมาย: default landing tab after removal is 'appointment'.
  • Q2 = นัดหมายครอบคลุมหมด: remove the live queue entirely; no dedicated arrival surface.
  • Q3 = ลบหายเลย (hard): "ลบมัดจำ" → deleteDepositBookingPair (row vanishes from Finance), matching the kiosk precedent.
  • Q4 = ทุกที่ที่ยกเลิกได้: the shared dialog appears on all 3 live cancel surfaces (นัดหมาย + AppointmentCalendarView + Finance·มัดจำ).
  • Approach: one shared dialog component + one pure decision helper (Rule C1), not inline-per-surface.
  • Removal depth: hard-delete the tab buttons + render branches + orphaned state (not hide-only).

Acceptance Criteria

  1. Frontend admin shows exactly 4 tabs (แชท · นัดหมาย · ตั้งค่า · หลังบ้าน) on desktop AND mobile; opening it lands on นัดหมาย.
  2. Any stale adminMode / deep-link to a removed mode redirects to นัดหมาย (no blank screen).
  3. Cancelling a นัดหมาย appt WITH a deposit → dialog with 3 choices; 🗑 removes both deposit + appt; 📌 cancels appt only, deposit stays usable.
  4. Cancelling a นัดหมาย appt WITHOUT a deposit → unchanged plain confirm.
  5. Deposit with usedAmount > 0 → "ลบมัดจำ" disabled + clear Thai reason; ลบแต่นัด still works.
  6. The same dialog fires on Backend AppointmentCalendarView appt-delete + Finance·มัดจำ deposit-cancel (deposit-oriented copy).
  7. No orphan deposit left in Finance after a 🗑 cancel; no orphan appt after a Finance 🗑.
  8. Full vitest green + build clean + Rule I flow-simulate + real-prod e2e (L2) for the cascade branches. No firestore.rules/index change.

Risks / Open Questions

  • Dead effects gated on adminMode==='dashboard' (walk-in arrival modal :3549, inline cleanup) — since dashboard is never active after removal, verify these were already retired by the appointment-hub work (checkpoint says "kiosk walk-in modal wiring retired") or relocate/delete them. Must not silently disable a needed flow.
  • Mobile dock parity — the 4 removals must land in the mobile menu/dock too, not just the desktop bar.
  • Finance·มัดจำ orientation — the dialog there is deposit-oriented ("ลบมัดจำ → ลบนัดด้วย/เก็บนัด"). Confirm at L1 the copy reads correctly.
  • "ลบแต่นัด" leaves deposit.linkedAppointmentId → a cancelled appt — acceptable (money preserved, customer rebooks); the deposit stays usable. Confirm no downstream reader breaks on a linked-but-cancelled appt.
  • Surviving kiosk-delete callsites — confirm in the plan which of :4227/:5519 die with the tabs vs survive elsewhere.
