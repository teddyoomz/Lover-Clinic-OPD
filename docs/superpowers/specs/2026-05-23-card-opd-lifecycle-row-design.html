Card-Level OPD Lifecycle Row — Design Spec

Date: 2026-05-23 EOD+1 LATE+5 · Status: Approved by user ("approve") · Owner area: AdminDashboard Frontend tab → นัดหมาย sub-tabs · Implementation plan: writing-plans (pending) · V-tag: V118 (next available)

Purpose / Problem

Admin running the Frontend นัดหมาย tab currently has to open the "แก้ไขนัด" modal for every customer-later booking before being able to send/view the OPD-fill QR link, AND has to switch to a different sub-tab (คิวหน้าคลินิก / จองมัดจำ / จองไม่มัดจำ) to click "บันทึกลง OPD" when the customer's form arrives. Both actions are one extra navigation step per appointment — multiplied by ~50 appts/day this becomes real friction.

This spec lifts both actions onto the appointment Card itself:

  • 🔗 Link button — send/view the OPD-fill QR (only visible when no HN, mirrors AppointmentFormModal:938-984)
  • 🩺 OPD button — always-visible 3-state button (⏳ รอกรอก disabled · 🔴 บันทึก OPD clickable · 🟢 ดูข้อมูล opens session modal)

Behavior mirrors the existing คิวหน้าคลินิก / จองมัดจำ / จองไม่มัดจำ tabs exactly — same handleOpdClick(session) action, same setViewingSession(session) modal target, same V116 auto-attach + un-hide cascade. This is a presentational lift, NOT new business logic.

Flow

User journey · customer-later booking from creation through OPD save

1
Admin creates appointment via Frontend "+ จอง" — picks "เลือกลูกค้าภายหลัง" → enters customerNameTemp + customerPhoneTemp. customerId is null. linkedOpdSessionId is null. → Card renders State B (no HN + no link sent).
↓
2
Admin clicks 🔵 ส่งลิ้งค์ลูกค้ากรอก OPD on the card. Calls provisionOpdLinkForBookingPair(...) (existing function from AppointmentFormModal:943). Creates opd_sessions/{id} with formType:'intake', stamps linkedOpdSessionId on both the appt + linked deposit. Opens <SendCustomerLinkModal> with URL + QR. → Card renders State C on modal close (link exists, no patientData yet).
↓
3
Customer opens the link (LINE/SMS) → fills the PatientForm via the unique URL → submits. Firestore opd_sessions/{id}.patientData becomes non-empty. Listener fires → Card auto-renders State D with red pulsing 🔴 button + 📥 ready-to-save chip.
↓
4
Admin clicks 🔴 บันทึกลง OPD. Calls handleOpdClick(linkedSession) (existing function from AdminDashboard:3470). NO confirm dialog (mirrors existing flow). Creates be_customers/{newId} via kioskPatientToCanonical → addCustomer; sets session.opdRecordedAt = serverTimestamp() + brokerStatus='done'; attachCustomerToOpdSessionLinks stamps customerId on the appt + deposit; V116 un-hides the queue entry if previously hidden. Toast: "บันทึก OPD สำเร็จ ✓".
↓
5
Card auto-renders State E → State A on next render cycle: HN populates from customer.proClinicHN (or local LC-XXX id), 🔗 link button disappears, 🩺 button stays as 🟢 ดูข้อมูล. Admin can now click 🟢 to re-open the OPD session modal anytime.

State machine (visibility predicates)

StatePredicate🔗 Link button🩺 OPD button📥 Chip
A appt.customerId truthy ❌ hidden 🟢 ดูข้อมูล OPD —
B !customerId && !linkedOpdSessionId 🔵 ส่งลิ้งค์ลูกค้ากรอก OPD ⏳ ยังไม่มีข้อมูล (disabled) —
C !customerId && linkedOpdSessionId && !session.patientData 🟢 ดูลิ้งค์ที่ส่งไป ⏳ รอลูกค้ากรอก (disabled) —
D !customerId && linkedOpdSessionId && session.patientData && !session.opdRecordedAt 🟢 ดูลิ้งค์ที่ส่งไป 🟢 ดูข้อมูล OPD (review) + 🔴 บันทึกลง OPD (save) 📥 ลูกค้ากรอกแล้ว · รอบันทึก
E session.opdRecordedAt && session.brokerStatus === 'done' (transient — next render hits A) ❌ hidden 🟢 ดูข้อมูล OPD —

Data wiring contract

// AdminDashboard.jsx (parent — owns state)
const sessionsById = useMemo(() => {
  const m = new Map();
  for (const s of [...sessions, ...archivedSessions, ...depositSessions, ...archivedDepositSessions, ...noDepositSessions]) {
    m.set(s.id, s);
  }
  return m;
}, [sessions, archivedSessions, depositSessions, archivedDepositSessions, noDepositSessions]);

const lazyFetchedSessionsRef = useRef(new Map());
const resolveLinkedSession = useCallback((appt) => {
  if (!appt?.linkedOpdSessionId) return null;
  if (sessionsById.has(appt.linkedOpdSessionId)) return sessionsById.get(appt.linkedOpdSessionId);
  if (lazyFetchedSessionsRef.current.has(appt.linkedOpdSessionId))
    return lazyFetchedSessionsRef.current.get(appt.linkedOpdSessionId);
  // Lazy fetch (fire-and-forget; sets state on resolve to trigger re-render)
  getDoc(opdSessionDoc(appt.linkedOpdSessionId)).then(snap => {
    if (snap.exists()) {
      lazyFetchedSessionsRef.current.set(snap.id, { id: snap.id, ...snap.data() });
      forceUpdate({});  // setLazyTick(t => t+1)
    }
  });
  return null;  // First render: not yet loaded; row shows "⏳" disabled until next paint
});

const handleViewOpdFromCard = useCallback((appt) => {
  const session = resolveLinkedSession(appt);
  if (session) {
    setViewingSession(session);
    return;
  }
  // State A with no linked session — synthesize from customer
  if (appt.customerId) {
    const customer = customersById.get(appt.customerId);
    if (customer?.patientData) {
      setViewingSession({
        id: `synth-${appt.customerId}-${appt.id}`,
        patientData: customer.patientData,
        opdRecordedAt: customer.createdAt || null,
        brokerStatus: 'done',
        brokerProClinicId: customer.proClinicId || null,
        brokerProClinicHN: customer.proClinicHN || customer.HN || '',
        __synthetic: true,
      });
    }
  }
}, [resolveLinkedSession, customersById]);

Sub-tab scope (per Q2 = A locked)

  • ✅ วันนี้, กำลังจะถึง, ก่อนหน้า — OPD lifecycle row renders.
  • ✅ เสร็จแล้ว sub-pill on "วันนี้" — renders (admin may want to view data post-service).
  • ❌ ยกเลิก — OPD lifecycle row hidden entirely (no follow-through needed for cancelled appts).

  • AV118 (NEW) — every site deriving "OPD save state" from (session.opdRecordedAt && session.brokerStatus === 'done') MUST use isOpdSessionSaved(session) from src/lib/opdSessionState.js. Grep target enforces. Sanctioned callsites: AdminDashboard.jsx, AppointmentHubRowCard.jsx, opdSessionState.js itself.
  • V87/AV84 preserved — this brainstorm does NOT introduce a new setPatientLinkModal trigger (different modal — SendCustomerLinkModal vs PatientLinkModal). AV84's closed list of 2 stays untouched.
  • V116/AV116 preserved — the new save button is a NEW caller of EXISTING handleOpdClick. provisionOpdLinkForBookingPair's existence-check + un-hide path remain unchanged.
  • V12 multi-reader-sweep guard — the synthesized session shape ({__synthetic: true, ...}) is opt-in. Existing readers that call setViewingSession continue with real-session shapes. Future modal-internal readers that check session.id for Firestore writes must check !session.__synthetic first (handled in spec — no current writers exist).

Decisions (locked via Q&A with user)

  • Q1 = B: OPD lifecycle row placement = between status pill and existing action row (Option B in Visual Companion mockup).
  • Q2 = A: Scope = all active sub-tabs (วันนี้ / กำลังจะถึง / ก่อนหน้า), excluding ยกเลิก.
  • Q3 (user clarification post-design): 🩺 OPD button is always present (5 states A-E above) — NOT conditional on HN. Admin uses it as a fast-glance OPD viewer for any customer, saved or unsaved.
  • Modal target: setViewingSession(session) — opens the existing "ประวัติผู้ป่วย OPD" modal (already wired in AdminDashboard, currently used by คิวหน้าคลินิก / ประวัติ tabs).
  • Behavior: mirrors handleOpdClick exactly — no confirm dialog, V116 cascade preserved.
  • Synth session shape for State A with no linked session: derived from be_customers.patientData + __synthetic:true marker.

Acceptance Criteria (testable)

  1. Card in State A (has HN, no linked session) → 🔗 hidden + 🩺 = 🟢 ดูข้อมูล OPD. Click → opens "ประวัติผู้ป่วย OPD" modal with synthesized session showing the customer's patientData. "ดูข้อมูลลูกค้า" button inside the modal still navigates to CustomerDetailView normally.
  2. Card in State B (no HN, no link) → 🔗 = 🔵 ส่งลิ้งค์ลูกค้ากรอก OPD + 🩺 = ⏳ ยังไม่มีข้อมูล (disabled with tooltip). Click 🔗 → provision → SendCustomerLinkModal opens → URL + QR display.
  3. After State B modal close → card transitions to State C: 🔗 = 🟢 ดูลิ้งค์ที่ส่งไป + 🩺 = ⏳ รอลูกค้ากรอก (disabled).
  4. Customer submits PatientForm via URL → session.patientData populates → card auto-transitions to State D: 🔗 = 🟢 + 🩺 = 🔴 บันทึกลง OPD (clickable, pulsing) + 📥 chip in top-left.
  5. Click 🔴 บันทึกลง OPD → handleOpdClick fires → toast "บันทึก OPD สำเร็จ" → card transitions to State E → next listener tick converts to State A (HN populated, 🔗 hidden, 🩺 = 🟢).
  6. ก่อนหน้า sub-tab card (past month, has linkedOpdSessionId but session not in current listener window) → first render: 🩺 = ⏳ briefly while lazy-fetch resolves → next render: correct state (A/D/etc.) based on fetched session.
  7. ยกเลิก sub-tab → OPD lifecycle row NOT rendered at all (no 🔗, no 🩺, no 📥 chip).
  8. V116 invariant: clicking 🔴 บันทึกลง OPD when the appt was previously hidden from queue (via queue-delete-with-link-survives) → admin's queue auto-restores the entry via existing un-hide cascade.
  9. Source-grep regression bank locks: (a) all derivations of opdRecordedAt && brokerStatus === 'done' in src/components/admin/** use isOpdSessionSaved; (b) setPatientLinkModal trigger count remains 2 (V87/AV84 unchanged); (c) at least 1 provisionOpdLinkForBookingPair call in AdminDashboard.jsx (the new card-level wrapper).

Risks / Open Questions

RiskMitigation
R1: ก่อนหน้า tab lazy-fetch race — admin scrolls fast through 50 past appts; 50 parallel getDoc calls overload. Throttle via in-flight Set; coalesce duplicate fetches; cap at 5 concurrent. Loaded sessions stay in useRef Map across re-renders.
R2: Synthesized session shape breaks an existing modal-internal feature that expects real Firestore doc (e.g. "ซิงค์ข้อมูลใหม่" button at the modal top performs a Firestore write). The modal's existing buttons gate on !session.__synthetic — disabled for synth sessions with tooltip "ใช้ได้กับ session จริงเท่านั้น". 4 buttons to inspect: "แก้ไขข้อมูล" / "ซิงค์ข้อมูลใหม่" / "พิมพ์สรุป A4" / "พิมพ์ฟอร์มมาตรฐาน" / "Resync OPD". Print buttons should work (read-only); edit + resync should gate.
R3: V12 multi-reader-sweep — adding a new caller of setViewingSession may surface existing callers that didn't handle some shape edge case. Pre-flight grep for all setViewingSession callsites; verify each handles null + missing patientData gracefully. AV118 enforces post-fact.
R4: User notices "ก่อนหน้า" tab cards flash ⏳ before resolving to correct state. Acceptable UX (~200ms per card). Add subtle skeleton-shimmer effect on the disabled state if it bothers anyone — can be follow-up polish.

Out of Scope

  • Modifying the existing "บันทึกการรักษา" button on cards with customerNameTemp (no HN). That's a separate UX gap (admin shouldn't see "บันทึกการรักษา" if customer has no Firestore record). Tracked as follow-up.
  • Modifying คิวหน้าคลินิก / จองมัดจำ / จองไม่มัดจำ tabs — those keep their current buttons as the canonical source of truth for OPD lifecycle. This spec only adds a SHORTCUT to the same actions on the appointment Card.
  • Adding new collections, endpoints, Firestore rules, or Storage paths. Pure UI lift.
