Tablet Chart Editor Design Spec

Date: 2026-05-20 · Status: Approved by user (Q1–Q5 + device-cache locked) · Implementation plan: docs/superpowers/plans/2026-05-20-tablet-chart-editor.html (created next via writing-plans)

Purpose / Problem

Doctors want to annotate medical chart templates (face / body / teeth / blank / uploaded) with a stylus on an iPad or Android tablet, and have that drawing land on the patient's treatment chart. Today the only editor is ChartCanvas.jsx (fabric.js) which runs on the same device that opened the TreatmentFormPage (TFP) and has no pressure / stylus support at all.

This feature adds a paired tablet editor: a clinician on a PC (or any device) opens the TFP chart modal, picks a template, and chooses to edit it on a standby tablet in the same branch. The tablet pops a full-screen, best-in-class pen editor (Apple Pencil / Android stylus — the hard requirement), the clinician draws, taps Save, and the finished image flows back into the PC's TFP form. The whole feature must live in new files — TFP itself is not touched.

Flow

State machine — session lifecycle
PC: pick template → "แก้ที่แท็บเล็ต" → pick ready tablet
        │  [transaction: re-check tablet idle → create session 'requested'
        │   + presence→busy + upload template image to Storage]
        ▼
  (session 'requested', targets tabletDeviceId)
        │  tablet onSnapshot sees a session for ITS deviceId
        ▼
📱 editor pops INSTANTLY → status 'active' → (tablet heartbeat starts) → draw ✏️
        │  Save → upload result PNG to Storage → status 'saved'
        ▼
🖥️ PC onSnapshot 'saved' → download result → merge into charts[] (dataURL)
        │  → delete session + Storage folder → presence→idle ✅ → save w/ treatment

CANCEL / TIMEOUT (any state before 'saved'):
  tablet Cancel → 'cancelled' (by:tablet)   → PC shows ❌ failed
  PC Cancel     → 'cancelled' (by:pc)        → tablet shows 🔌 → standby
  heartbeat>30s → 'cancelled' (by:timeout)   → both exit
  → on every terminal state: presence→idle, Storage folder cleaned
Sequence — who writes what (PC ⟷ Firestore ⟷ Storage ⟷ tablet)
#ActorAction
1TabletOn standby: upsert be_chart_tablet_presence/{deviceId} {idle, branchId, name}; heartbeat every ~10s.
2PCLists idle+fresh presences in branch (BSA listener). User picks one.
3PCTransaction: re-read target presence is idle → set busy + create be_chart_edit_sessions 'requested'. Upload template image to uploads/chart-edit-sessions/{sessionId}/template.png; write templateImageUrl.
4TabletonSnapshot: a 'requested' session for my deviceId → download template → open editor → set 'active' + start tablet heartbeat.
5TabletSave: upload result PNG → write resultImageUrl + status 'saved'.
6PConSnapshot 'saved': download result → push {dataUrl, templateId, source:'tablet'} into charts[] → delete session doc + Storage folder → presence→idle.
7BothHeartbeat watchdog: if the other side's heartbeat is >30s stale → set 'cancelled'/timeout → exit + cleanup.

Orphans are reaped by a scheduled sweep that cancels sessions with stale heartbeats & deletes their Storage folders (mirrors the existing cron pattern, e.g. stock-movement-retention).

Architecture

All new logic lives in new files plus the existing ChartSection.jsx (already a separate component that owns the chart flow; TFP itself: 0 lines changed) and a small route addition in App.jsx.

src/
├─ pages/TabletChartEditorPage.jsx # the ?tablet=chart link target: standby ↔ editor
├─ components/tablet-chart/ # NEW folder
│  ├─ TabletStandby.jsx # name + branch picker + waiting; both cached (localStorage)
│  ├─ PenCanvas.jsx # Pointer Events + perfect-freehand; pressure + palm rejection
│  ├─ EditorToolRail.jsx # pen / highlighter / eraser / color / size / undo-redo / clear
│  ├─ PcPairingModal.jsx # "edit here / on tablet" + ready-list + waiting/failed
│  └─ TabletReadyList.jsx # live idle-tablet picker (branch-scoped)
├─ lib/chartEditSession.js # create/listen/update/cancel/heartbeat/cleanup + Storage up/download
├─ hooks/useChartEditSession.js # PC-side session hook
├─ hooks/useTabletPresence.js # tablet heartbeat + idle/busy
├─ components/ChartSection.jsx # EDIT (small): insert choice + relay between template-pick and local canvas
└─ App.jsx # EDIT (small): route ?tablet=chart → TabletChartEditorPage
  • Boundaries. chartEditSession.js is the only module that talks to Firestore/Storage for this feature; hooks wrap it; components are presentational. PenCanvas.jsx knows nothing about sessions (pure draw → onSave(dataUrl)). The PC editor (ChartCanvas.jsx, fabric.js) is left exactly as-is for local edits.
  • Branch scoping (BSA). Both new collections are be_* and branch-scoped: listeners go through scopedDataLayer / useBranchAwareListener; writers stamp branchId via the standard resolver. Registered in the branch-coverage audit.
  • Final persistence unchanged. The PC merges the tablet result into charts[] as {dataUrl, templateId, fabricJson:null, source:'tablet'} — the same shape used today — so the treatment-timeline, prints, and treatment save all work without modification.

Decisions (locked via Q&A with user)

  • Q1 = A: Session-doc relay — the tablet never writes the treatment; the PC merges the result into its in-memory charts[] and persists on treatment save. Works in TFP create mode (no treatmentId yet).
  • Q2 = A: Tablet authenticates as real staff (existing email+password), any permission tier; branch via the normal selector; only ever writes the session/presence docs.
  • Q3 = A: Purpose-built pen canvas (Pointer Events API + perfect-freehand), isolated from the fabric.js PC editor. Tools: pen / highlighter / eraser / colors / size / undo-redo / clear.
  • Q4 = A: Named tablets + live ready-list (auto-select if one; "ไม่มีแท็บเล็ตพร้อม" if none). Editor pops instantly on the tablet — no accept tap (spec point 5).
  • Q5 = A: Firebase Storage as image transport (session doc carries only URLs); final chart still stored as a dataURL in charts[] = zero downstream change; session Storage folder cleaned on terminal state.
  • Device cache: tablet's device name + branch are saved on-device (localStorage); Firebase keeps the staff session logged in → no re-entry on reopen.

Acceptance Criteria

  1. A tablet opening the link, logging in (any staff tier), naming itself, and choosing a branch appears as a ready device to PCs in that same branch; name + branch survive a reload with no re-entry.
  2. From the TFP chart modal, picking a template offers "แก้ที่เครื่องนี้ / แก้ที่แท็บเล็ต"; "เครื่องนี้" behaves exactly as today (no regression).
  3. "แก้ที่แท็บเล็ต" lists only tablets currently idle & fresh in the same branch; sending opens the editor on the chosen tablet instantly with the exact template (including a just-uploaded one).
  4. The tablet editor supports Apple Pencil + Android stylus with pressure-variable strokes, palm rejection, low latency, and the listed tools.
  5. Tablet Save → the image appears in the PC's chart section and persists when the treatment is saved (create & edit mode).
  6. Cancel from either side, or a >30s disconnect on either side, exits both screens with the correct message per the error matrix; presence returns to idle and the Storage folder is cleaned.
  7. Two PCs cannot drive the same tablet (transaction guard); a busy tablet is not offered as ready.
  8. TFP source is unchanged (0 lines); all new behavior is in new files + ChartSection.jsx + a small App.jsx route.
  9. Tests: unit + RTL + Rule I flow-simulate + stress all green; Rule Q L2 real-client-SDK relay e2e passes on real Firestore + Storage.

Risks / Open Questions

  • Stylus feel can't be verified headless. Per Rule Q V66: pressure / palm-rejection / latency feel is an L1 hands-on on a real iPad + Apple Pencil by the user. Automated tests cover the relay contract, state machine, and tool logic — not the tactile feel.
  • Re-editing a tablet-drawn chart. Tablet charts have fabricJson:null (perfect-freehand, not fabric). Re-opening one for edit is out of scope for v1 (YAGNI — the flow is pick→draw→save). If needed later, store native stroke JSON for tablet re-edit. Flagged, not built.
  • Image size. Storage transport removes the 1 MB Firestore-doc risk; we still cap result PNG resolution sensibly so the final dataURL in charts[] stays reasonable (matching today's behavior).
  • Orphan cleanup. Heartbeat-driven cancel handles live disconnects; a scheduled sweep is the backstop for crashes. Sweep cadence + Storage-lifecycle policy to be finalized in the plan.
  • Deploy. Two new be_* collections + (possibly) a Storage match → a rules change requiring Probe-Deploy-Probe (Rule B) + explicit user "deploy". No data migration.
