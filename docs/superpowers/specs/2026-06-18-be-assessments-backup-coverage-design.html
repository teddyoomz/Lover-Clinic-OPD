be_assessments Backup Coverage — Design Spec

Date: 2026-06-18 · Status: Approved by user (Q1=A comprehensive+drift, Q2=A diag+clean) · Scope: backend constants + guard test + Rule-M orphan cleanup. Implementation plan: docs/superpowers/plans/2026-06-18-be-assessments-backup-coverage.html

Purpose / Problem

The be_assessments collection (ED Score / follow-up assessment rounds — a flat, UNIVERSAL collection keyed by a customerId field, added 2026-06-15) is omitted from every hardcoded backup / restore / cascade list. The whole-system FULL backup (the 03:00 daily cron + manual button) DOES capture it (V122 dynamic listCollections() + empty DENYLIST) — so no ED data is lost in the daily backup — but every customer-scoped and branch path misses it:

  • Per-customer backup/restore (V74) + whole-fleet (V75): ED rounds NOT backed up → a restored customer loses ED history.
  • Customer DELETE cascade (server and the active client path): ED rounds orphaned after delete.
  • Customer MERGE (dup-resolution): ED rounds NOT moved to the keeper.
  • Branch backup/restore (V40): the universal be_assessments not round-tripped.
  • Whole-system customer-only backup: omitted.

This is the V122/V196 class-of-bug (a new collection added without updating the hardcoded lists — same as the V196 "missed be_recalls" lesson). A related drift surfaced: the active client-side customer-delete (customerDeleteClient.js) uses an 11-collection list (Phase 24.0 baseline) while the server uses the 16-collection CUSTOMER_CASCADE_COLLECTIONS_FULL — so V74's +5 gap-closure collections (be_quotations / be_vendor_sales / be_online_sales / be_sale_insurance_claims / be_recalls) are ALSO orphaned on a client-side delete. All 6 new collections are read, write: if isClinicStaff() in firestore.rules → fully client-deletable → the reconcile is safe (no rules changes, no CLIENT_DELETE_BLOCKED additions).

Decisions (locked via Q&A with user)

  • Q1 = A (comprehensive + drift): add be_assessments to every list AND reconcile the client delete to the single-source FULL list (also closes V74's +5 client-orphan). All 6 collections are isClinicStaff() read+write → client-deletable → no rules changes.
  • Q2 = A (diag + clean): Rule R env-pull (read-only) to find orphaned be_assessments; if found, Rule-M two-phase cleanup (dry-run → user-authorized --apply + audit doc). Code fix ships regardless.
  • Branch buckets excluded: be_assessments is universal (no branchId) — it does NOT belong in the branchId-filtered make-fresh buckets; only the branch backup UNIVERSAL set round-trips it.
  • be_customer_identity + be_recall_cases NOT added: verified not customer-data (identity = a global uniqueness index freed/promoted by _freeCustomerIdentityClaim; recall_cases = clinic-wide presets). Correctly excluded.

Acceptance Criteria

  1. be_assessments ∈ CUSTOMER_CASCADE_COLLECTIONS_FULL, CUSTOMER_ONLY_UNIVERSAL, branchBackupCore.UNIVERSAL.
  2. Client CUSTOMER_CASCADE_COLLECTIONS equals CUSTOMER_CASCADE_COLLECTIONS_FULL (single source; 17 items); client + server COL_TO_RESPONSE_KEY map all 17.
  3. A per-customer backup of a customer with ED rounds includes them; restore re-creates them at the same docIds with customerId preserved.
  4. A customer delete (client + server path) removes that customer's be_assessments docs (no orphan); the auto-pre-backup captures them first.
  5. The dup-merge script moves a customer's be_assessments to the keeper.
  6. Guard test: any customer-scoped-universal collection missing from the 3 lists → red build (drift prevention).
  7. Rule R diag reports orphan count; if >0, a Rule-M cleanup script exists (two-phase, audit doc, idempotent) and is run only on explicit authorization.
  8. Full vitest green + build clean.

Testing

  • Unit/source-grep (tests/be-assessments-backup-coverage.test.js): be_assessments in each list; client≡full lockstep; COL_TO_RESPONSE_KEY completeness (client + server); guard test (classifier-driven drift catch); branch-buckets correctly EXCLUDE be_assessments.
  • Flow-simulate (Rule I): chain customerId → backup (collect by customerId) → file shape → restore (recreate) → assert be_assessments round-trips; delete-cascade collects + deletes be_assessments refs; merge moves them.
  • Rule Q L2 (real prod, read-only diag): the orphan diag IS a real-prod client/admin-SDK read; the backup/restore round-trip can be e2e-verified against a TEST- customer with seeded be_assessments (mirror the V74/V122 e2e templates) — backup → restore into an isolated namespace → assert ED rounds present.
  • Honest gap (Rule Q): destructive paths (real customer delete / make-fresh) verified via TEST-prefixed fixtures + isolated-namespace round-trip, not live customer data.

Risks / Open Questions

  • Client reconcile changes the delete surface (11→17): mitigated — all 6 added collections are isClinicStaff() read+write (client-deletable; verified in firestore.rules) → no PERMISSION_DENIED, no CLIENT_DELETE_BLOCKED additions. The audit-immutable trio stays blocked as before.
  • COL_TO_RESPONSE_KEY drift: client + server maps must both cover 17 — locked by the guard test (else cascadeCounts[undefined] collision).
  • Restore FK: be_assessments docs are keyed by their own assessmentId + carry customerId; overwrite-mode restore preserves both → re-links cleanly (same pattern as be_course_changes which already works). Clone-mode (re-mint customerId) is out of per-customer-restore scope (T1-only).
  • No open questions — scope locked via Q1/Q2.
