Date: 2026-05-28 · Status: Approved by user ("ok", 2026-05-28) · Follow-up to: V129 (reports-sale seller/creator resolve) · Implementation plan: docs/superpowers/plans/2026-05-28-reports-sale-createdby-and-responsive-table.html (next step)

Purpose / Problem

Two improvements on the tab=reports-sale surface (รายการขาย):

  1. Item 1 — true "ผู้ทำรายการ" (createdBy). be_sales has never written a createdBy field (empty on all 49 real sales). V129 made the report's ผู้ทำรายการ column fall back to the first seller's resolved name. The user wants the column to reflect who actually keyed the sale in — captured at write time from the logged-in staff.
  2. Item 2 — responsive table fit. The report table has 18 columns and a fixed min-w-[1400px] (SaleReportTab.jsx:408) inside an overflow-auto wrapper. On a display scaled via Windows (e.g. 1920px physical @ 150% → ~1280 CSS px) the table is wider than its panel, so columns are cut at the panel edge and the only way to reach them is a horizontal scrollbar that sits at the bottom of a tall 39-row table (below the fold) — it looks like "ล้นจอ" / unreachable columns.
⚠️ Diagnosis correction (real-browser verified, Chrome MCP, 2026-05-28). The original premise — "a backend shell flex column missing min-w-0 blows the table past the viewport" — is WRONG. Both shells already carry min-w-0 (BackendNav.jsx:94, BackendShellNew.jsx:97). A faithful repro of the exact class chain at an emulated 1200px viewport measured: pageOverflows=false, mainClips=false, wrapScrolls=true (a 1400px table inside a 922px wrapper scrolls in-panel). So there is NO page overflow — the table is contained and scrolls. The real pain is (a) the table is simply too wide for the panel and (b) the in-panel horizontal scrollbar is below the fold / hard to reach. The shell "fix" is therefore dropped (no-op); Item 2 = compact density + a height-capped table region so the horizontal scrollbar is always reachable (user-confirmed 2026-05-28).

Audit + tests

  • NEW AV148 (audit-anti-vibe-code): (a) regression-guard — both backend shells' <main> MUST retain min-w-0 (so wide tables keep scrolling in-panel, never off-screen); (b) reports-sale table region MUST be height-capped (max-h-* + overflow-auto) with the compact-density classes. Source-grep both shells + SaleReportTab.
  • NEW AV149: sale-write MUST capture createdById/createdByName/createdBySource at the createBackendSale chokepoint; report createdBy fallback chain locked.
  • Unit + source-grep tests: _resolveSaleCreatedBy 4 branches, aggregator fallback chain, compact-class source-grep, shell min-w-0 source-grep.
  • Rule S real-browser (Chrome MCP) at a scaled viewport: table contained + compact readable in both shells.
  • Rule Q L2 real-prod verify script for Item 1; backfill dry-run on real prod before --apply.

Decisions (locked via Q&A with user, 2026-05-28)

  • Q1 = C (compact density): shrink padding/font (Thai-readable floor) + relax min-width + truncate long text columns. Combined with Q2 the remaining overflow scrolls in-panel.
  • Q2 = A → DROPPED: the shell-level "fix" was premised on a flex blowout that real-browser measurement disproved (shells already carry min-w-0). No shell code change — only an AV148 regression-guard. (User confirmed 2026-05-28: "ตัดการแก้ shell ทิ้ง ไม่จำเป็น".)
  • Item-2 follow-up = A (Compact + reachable scroll): height-cap the table region (max-h-[~70vh] + overflow-auto) so the horizontal scrollbar is always in view; sticky header/footer stick to the capped box. (User confirmed 2026-05-28.)
  • Q3 = A (id + name snapshot): store createdById (live-resolve ref) + createdByName (snapshot) + createdBySource (honesty tag). Report prefers the snapshot, then live-resolve, then legacy/first-seller.
  • Q4 = B (backfill = first seller): Rule M script stamps legacy createdBy = resolved first seller, tagged createdBySource:'first-seller-backfill' (distinguishable from true capture).
  • Compact = always-on (no density toggle — user did not pick the toggle option; default per recommendation).

Acceptance Criteria

  1. A new sale created by a logged-in staff writes createdById = that staff's id, createdByName = that staff's name, createdBySource = 'staff' on the be_sales doc.
  2. A sale created by a non-staff admin (owner gmail) writes createdByName = displayName/email, createdBySource = 'auth' (no crash; no '-' leak of an id).
  3. The reports-sale ผู้ทำรายการ column shows the true creator for new sales; legacy/un-backfilled sales still show the first-seller fallback (never a raw id).
  4. Backfill --apply stamps legacy sales' createdBy = first seller with createdBySource:'first-seller-backfill'; re-run yields 0 writes; an audit doc is written.
  5. On a Windows-scaled desktop viewport, the reports-sale table region is height-capped so the horizontal scrollbar is reachable within the viewport (not below the fold), and compact density fits more columns before any scroll is needed; the page never overflows off-screen.
  6. Both backend shells (BackendNav, BackendShellNew) retain min-w-0 on <main> (AV148 regression-guard) — no shell code change in this work.
  7. Compact density keeps Thai text readable (≈11px, never below ~10px; no clipping of single Thai glyphs); long รายละเอียด/ช่องทาง columns truncate with a title tooltip.
  8. Mobile (<lg) card view is byte-unchanged.
  9. No firestore.rules/storage/index/cron changes → no Probe-Deploy-Probe; full vitest green + build clean.

Risks / Open Questions

  • 18 columns can't ALL fit readably on a heavily-scaled screen — compact only reduces scroll; B1 makes the residual scroll safe (in-panel). This is an accepted limit, disclosed.
  • Sticky header/footer inside a height-capped scroll box — once the table region gets max-h + overflow-auto, the existing sticky thead(top-0)/tfoot(bottom-0) stick to the capped box instead of the page. Must verify in a real browser they still pin correctly (Rule S) and the mobile card view (which has its own sticky footer) is untouched.
  • Shells already correct — no shell code change; the only risk is a future edit removing min-w-0 from a shell <main>, which AV148's regression-guard catches.
  • online-sale path — if any sale is created server-side (no auth.currentUser), createdBy is empty → first-seller fallback at read (branch 4). Acceptable; flagged.
  • Backfill honesty — the legacy createdBy is a best-effort guess (first seller), tagged first-seller-backfill so it's never mistaken for true audit capture. Display is unchanged either way.
  • Compact aggressiveness — exact min-width/padding/font values to be tuned against the real rendered table via Chrome MCP at a scaled viewport during implementation (Rule S).
  • Not in scope: updatedBy capture on sale edits; a comfortable/compact density toggle; redesigning the 18-column set (no columns removed).
