Light-Theme Audit & Fix Implementation Plan

For agentic workers: REQUIRED SUB-SKILL: use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement task-by-task. Steps use HTML checkboxes for tracking.
Spec: docs/superpowers/specs/2026-05-27-light-theme-audit-design.html

Goal: Make every App-UI page and modal clearly legible in light theme (WCAG AA), fixing only design CSS/colour — zero wiring/flow/logic.

Architecture: Discovery-first: a grep + audit + live-browser sweep produces a concrete bug inventory, then a fixed per-batch procedure applies central index.css overrides (FM-A/B/C) and component inline-colour swaps (FM-D), each batch live-verified at the pixel level in light AND auto modes.

Tech Stack: React 19 + Vite + Tailwind (arbitrary classes) + src/index.css [data-theme] override layer; Chrome MCP / Claude Preview for real-pixel verification; /audit-ui-cultural-a11y + /audit as automated backstops; Vitest.

Decisions (locked): Q1 AA 4.5:1 · Q2 Hybrid central-first · Q3 App UI excl. print · Q4 Full live-browser pass.

HARD CONSTRAINT (every task): diff = CSS rule / className string / inline colour-value ONLY. If a fix would require touching a handler / state / prop / hook / flow / data → STOP (violates cosmetic-shell, feedback_cosmetic_shell_redesign_constraint). Every light fix is scoped to [data-theme="light"],[data-theme="auto"] (or :root[data-theme="light"]) and must never regress dark theme.

Commits / deploy: the executor commits LOCALLY per task (frequent commits). push + vercel --prod remain gated on explicit user word (V18) — this work is frontend-only (no rules/storage/cron → no Probe-Deploy-Probe) and ships alongside the existing unpushed EOD+11/EOD+12 stack.

Tasks 2–6: Fix per batch — B1 app shell · B2 backend tabs · B3 modals · B4 patient-facing · B5 shared

Each batch task repeats the SAME procedure over its inventory rows. Files: src/index.css (FM-A/B/C) + the specific component files the inventory flags FM-D. One commit per batch.

Append into the matching labelled section of src/index.css (BACKGROUNDS / TEXT COLORS / BORDERS / SHADOWS…). Always pair both selectors, map to the correct token, target AA:

[data-theme="light"] .<dark-class>,
[data-theme="auto"]  .<dark-class>  { <prop>: var(--<token>) !important; }
/* FM-C colour: pick a shade with >= 4.5:1 on white, e.g. *-400 -> *-700 */

Token map: dark bg #0f0f0f→--bg-card, #141414→--bg-input, #1a1a1a→--bg-hover; text-white/#fff→--tx-heading; text-gray-300→--tx-secondary; border-[#222]→--bd; dark glow→--shadow-panel/none.

- style={{ color: '#fff', ... }}
+ style={{ color: 'var(--tx-heading)', ... }}
- style={{ background: '#0f0f0f', ... }}
+ style={{ background: 'var(--bg-card)', ... }}

Only the colour value changes. If a value is computed in JS / chosen by a handler → that's logic; leave it and instead theme it via a wrapping class + central override, or flag for user. Do NOT edit the handler.

For every surface touched in the batch: in the browser set data-theme="light", navigate/open it, screenshot, confirm AA visually + via computed style; repeat with data-theme="auto" + emulated prefers-color-scheme: light. Sample contrast at a point:

const el = document.querySelector('SELECTOR');
const s = getComputedStyle(el);
console.log(s.color, s.backgroundColor);   // eyeball + compute ratio >= 4.5:1

Task 7: Override-coverage regression test (Rule D guard against future drift)

Files: Create tests/light-theme-override-coverage.test.js

import { readFileSync, readdirSync, statSync } from 'fs';
import { join } from 'path';
import { describe, it, expect } from 'vitest';

const css = readFileSync('src/index.css', 'utf8');
// dark arbitrary classes that MUST have a light override if used in app code
const DARK_HEX = ['#050505','#0a0a0a','#0f0f0f','#111','#111111','#141414','#1a1a1a','#222','#222222','#262626','#333','#444'];
// classes intentionally NOT overridden (white text on solid accent bg, etc.)
const SANCTIONED = new Set([/* fill from inventory, e.g. 'bg-[#1a0000]' hero */]);

function walk(dir){ return readdirSync(dir).flatMap(f=>{const p=join(dir,f);return statSync(p).isDirectory()?walk(p):p;}); }
const srcFiles = walk('src/components').concat(walk('src/pages'))
  .filter(f=>/\.(jsx?|tsx?)$/.test(f));
const used = new Set();
for (const f of srcFiles) {
  const t = readFileSync(f,'utf8');
  for (const m of t.matchAll(/(bg|border|from|via|to)-\[(#[0-9a-fA-F]{3,8})\]/g)) used.add(m[0]);
}

describe('light-theme override coverage', () => {
  it('every used dark arbitrary class has a [data-theme=light] override (or is sanctioned)', () => {
    const missing = [...used].filter(cls => {
      if (SANCTIONED.has(cls)) return false;
      const hex = cls.match(/#[0-9a-fA-F]{3,8}/)[0];
      if (!DARK_HEX.includes(hex)) return false;          // only police DARK hexes
      const esc = cls.replace(/[#[\]]/g, c => '\\'+c);     // .bg-\[\#0f0f0f\]
      return !css.includes(`[data-theme="light"] .${esc}`);
    });
    expect(missing, `uncovered dark classes:\n${missing.join('\n')}`).toEqual([]);
  });
});
npx vitest run tests/light-theme-override-coverage.test.js

Expected: any failure lists uncovered dark classes → either fix them (central override, loop back to the right batch) or, if genuinely intentional, add to SANCTIONED with an inline comment justifying. Re-run → PASS.
