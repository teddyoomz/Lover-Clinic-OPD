WS1 — Anonymous-Surface Lockdown · Implementation Plan

For agentic workers: REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement task-by-task. Steps use HTML checkboxes for tracking. Spec: docs/superpowers/specs/2026-06-10-ws1-anon-surface-lockdown-design.html.

Goal: Remove anonymous-internet read/enumerate access to patient PII (opd_sessions) and anon write to clinic_schedules/chat_conversations, without breaking any patient link (V23-safe).

Architecture: Ship the client/server prep FIRST (128-bit session ids · move ?patient= query to /api/patient-view · migrate webhook chat writes REST→admin SDK), THEN tighten firestore.rules (split get/list + write-to-staff) in one Probe-Deploy-Probe. Rules are verified in the Firebase emulator locally before any prod deploy, then proven on prod with a real anonymous client-SDK adversarial script (Rule Q L2) + real-browser patient flows (Rule Q L1).

Tech Stack: Firebase Firestore security rules · Firebase emulator (@firebase/rules-unit-testing) · @firebase/firestore client SDK · firebase-admin · Vercel serverless · React 19 · Vitest.

Iron-clad: Rule B (Probe-Deploy-Probe on every rules deploy) · Rule Q (real-adversarial verify, no mock-only) · Rule M (data ops local+admin SDK) · V18 (no deploy without explicit "deploy" this turn) · V23 (don't break patient submit) · V1/V9 (don't tighten a rule an unauth path depends on without migrating that path first).
