Q1 — เวลาเจอ order จากการ match ไส้ใน ให้แสดงผลแบบไหน?

A · มินิมัล — แค่ทำให้ order แถวนั้นโผล่ (ใช้ summary เดิมที่ตัดข้อความ max-280px). ถ้า Elonza เป็นรายการที่ 9/10 อาจโดนตัดหายจาก summary → ผู้ใช้เห็นแถวแต่ไม่รู้ว่าตรงเพราะอะไร
B · ดันรายการที่ match ให้เห็น — order แถวนั้นโผล่ + summary จัดให้ตัวที่ค้นเจอ (เช่น Elonza ×10) ขึ้นก่อน/ไฮไลต์ ไม่ว่าจะอยู่ลำดับไหน → เห็นทันทีว่าตรงเพราะอะไร
C · มุมมองรายการสินค้า (flatten) — โหมดผลลัพธ์ใหม่ที่โชว์ "ไส้ใน" ตรงๆ: Elonza ×10 · ORD-20250615 · exp 30/06/2026 · lot …. แรงสุดแต่เป็นของใหม่ทั้งก้อน + test เยอะกว่า

ทั้ง 2 หน้าจะแก้พร้อมกัน: per-branch (OrderPanel) + คลังกลาง (CentralStockOrderPanel) — บั๊คเดียวกัน (Rule P)

Q2 — รูปแบบการแก้วันหมดอายุ

A · แก้ inline ในแถว — กดที่วันหมดอายุ → ช่อง DateField โผล่ในแถวเลย → กด ✓ บันทึก. เร็วที่สุด แต่เผลอแก้ง่าย + ไม่มีที่ใส่เหตุผล/ยืนยัน
B · mini-modal (ตามภาพ) — ปุ่ม ✎ เปิดกล่องเล็ก: โชว์ค่าเดิม → ใส่ค่าใหม่ + เหตุผล → บันทึก/ยกเลิก. กันเผลอ + เก็บ audit ดีกว่า + reuse ได้ทั้ง 2 หน้า (แนวเดียวกับ V144 modal-in-place)

Q3 — เก็บร่องรอยการแก้ (audit) ยังไง?

A · ฟิลด์บน batch อย่างเดียว — expiresAtLegacyValue + expiresAtEditedAt + expiresAtEditedBy. เบา เห็นครั้งล่าสุด แต่ไม่เก็บประวัติหลายครั้ง
B · เอกสาร audit ใน be_stock_adjustments — type 'expiry' เก็บ old→new + เหตุผล + ผู้ทำ + เวลา (เข้าหน้าประวัติปรับสต็อคที่มีอยู่ ไม่ยุ่งกับ movement/conservation)
C · ทั้งคู่ — เอกสาร audit (ครบประวัติ เพราะ "แก้ได้ตลอด") + ฟิลด์บน batch (โชว์ "✎ แก้ล่าสุด …" inline). *ไม่* เขียนลง be_stock_movements (ไม่ใช่การขยับจำนวน → กัน conservation เพี้ยน)

Q4 — ตอนแก้ expiry ของ batch ให้ sync กลับไปที่บรรทัดสินค้าใน order ต้นทางไหม?

A · แก้แค่ batch — บรรทัดใน order (items[].expiresAt) คงเป็นบันทึกตอนนำเข้าเดิม. ง่ายสุด **แต่** ถ้าภายหลังมีคนแก้ order ผ่าน OrderDetailModal (มันcascade expiry → batch) ค่าที่แก้ batch จะโดนทับด้วยค่าเก่า
B · sync สองทาง — แก้ batch แล้วเขียนกลับบรรทัด order ต้นทางด้วย (batch มี sourceOrderId+orderProductId → หา 1:1 ได้) → ตรงกันเสมอ + กันโดนทับภายหลัง (สอดคล้องกับ cascade order→batch เดิม)

★ = ที่ผมแนะนำ. ตอบ Q1–Q4 ได้เลย (เลือก default ทั้งหมดก็ได้). หลังเลือกเสร็จผมจะเขียน spec (HTML + mockup + flow) ให้รีวิวก่อนลงมือ.
