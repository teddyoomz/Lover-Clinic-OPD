## Decisions

1. **เหมาตามจริง = one-shot, track at course level.**
   `assignCourseToCustomer` writes the customer.courses entry with qty
   "1/1 <unit>" sentinel + `courseType: 'เหมาตามจริง'` tag. Save path's
   `deductCourseItems` short-circuits for this tag (zero out regardless of
   deductQty) so the course moves to "ประวัติ" after one treatment. Stock
   deducts directly from be_products batches based on doctor-entered qty.

2. **เลือกสินค้าตามจริง = two-step pick-at-purchase** (NOT
   limit-gated — that attempt was reverted after user feedback).
   `buildPurchasedCourseEntry` returns a placeholder with
   `needsPickSelection: true` + `availableProducts: [...]` + empty
   `products: []`. A new `PickProductsModal` component is the gate: user
   ticks 1+ options + confirms qtys → `resolvePickedCourseEntry` fills
   `products[]` and clears the flag. From that point the course is
   standard (remaining tracking, normal stock deduction, history on 0).

3. **DF Payout Report courseId mismatch (silent ฿0 across prod).**
   Real sales (SaleTab.confirmBuy) store course items with `it.id` only
   (no `courseId`). Aggregator keyed courseIndex by `courseId` alone →
   every prod sale's DF computed to ฿0. Fix: key by BOTH `courseId` and
   `id` (skip empties to avoid '' collision). Inference path same fix +
   `itemType === 'course'` filter to prevent products leaking DF.

4. **DF % rate × full course price × usage weight** per user's spec.
   `computeDfAmount(rate, sub, qty, { courseUsageWeight })` scales
   percent DF by the fraction of course consumed this visit. Weighted
   sum across all visits = full DF (invariant). Baht rate ignores
   weight (per-unit already). Aggregator's `explicitBySale` refactored
   from last-wins Map to array so multi-treatment-per-sale contributes
   correctly. Partial-usage math documented: average of
   `used_qty/total_qty` across all products in the course.

5. **Used-up courses ≠ expired courses.**
   CustomerDetailView used to concatenate `customer.expiredCourses` with
   `usedUpCourses` (remaining=0) in the "คอร์สหมดอายุ" tab. User wanted
   strict separation: expired = date-expired only; used-up traced via
   Purchase History. Fix: `expiredCourses = customer?.expiredCourses`
   only. Treatment-form course column also now filters consumed entries
   (same invariant as activeCourses filter, preventing a consumed
   fill-later course from silently re-appearing as tickable).

6. **Pick-at-treatment first-draft (limit-gated) was wrong.**
   Initial design (reverted commit `f7cb8a8`) treated `เลือกสินค้า
   ตามจริง` as fillLater with min/max bounds. User corrected: it's a
   TWO-STEP pick-at-purchase flow (user picks products + qtys upfront
   via dialog, then course behaves standardly). Revert cost nothing —
   the correct design shipped in the same session (`84f5b0d`). Iron-clad
   A ("bug-blast revert") kept this clean.

## Known limitations (follow-up, not blocking)

- **Pick-at-treatment late-visit** not yet wired. If user buys a
  pick-at-treatment course and closes without picking, the
  `availableProducts` array isn't persisted to `customer.courses` via
  `assignCourseToCustomer`. Needs:
    1. `assignCourseToCustomer` stores `availableProducts` +
       `needsPickSelection: true` on each customer.courses entry when
       `masterCourse.courseType === 'เลือกสินค้าตามจริง'`
    2. `customerCoursesForForm` reads these fields and re-creates the
       placeholder shape so the UI can open PickProductsModal in a
       subsequent treatment.
  In-same-treatment pick + use works end-to-end already.

- **Partial-pick** — the current `PickProductsModal.onConfirm`
  populates `products[]` with only the picked subset. If user wants to
  add MORE picks later, they need to buy another course (no reopen
  flow). Acceptable for MVP; revisit if users complain.

- **`assignCourseToCustomer` courseType overrides for pick-at-
  treatment** don't yet exist (unlike `เหมาตามจริง` which has the
  "1/1 คอร์ส" sentinel logic). Currently pick-at-treatment bought
  courses rely on the in-memory `options.customerCourses` resolution
  via `resolvePickedCourseEntry` without persisting the resolved
  products through an auto-sale assignment.
