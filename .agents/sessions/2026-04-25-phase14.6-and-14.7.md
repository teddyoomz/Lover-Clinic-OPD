## Decisions (non-obvious — preserve reasoning)

### D1 — Inline-flex doesn't push text in inline-text contexts

Round 1 (`e8790ba`) tried `display:inline-flex; align-items:flex-end` on
underline spans to make text sit ON the dotted border. **It didn't work.**
Reason: flex container's height equals text content's natural height — no
extra space for `align-items:flex-end` to push into. The displayed result
was identical to inline-block.

Round 2 (`d77a421`) reverted to `inline-block` and used CSS-injected
`line-height:1 !important + padding-top:6px + padding-bottom:2px` to
collapse the line-box around the glyph. This worked. Verified via
`getComputedStyle` in preview_eval — `lineHeight: 12px` (was 18px),
`paddingTop: 6px`, `paddingBottom: 2px`.

**Lesson**: don't use flex alignment when the container size is content-
driven. Use line-height collapse + padding instead.

### D2 — CSS injection in two places (print window + preview)

The doc-print modal renders preview HTML via `dangerouslySetInnerHTML`
inside a div — no `<head>`, no scoped style scope. Putting CSS only in
`buildPrintDocument` (the print-window builder) means the preview shows
DIFFERENT styling from the print output.

Solution: inject the same CSS rules in BOTH places:
- `documentPrintEngine.buildPrintDocument` — `<head><style>` for the print
  window
- `DocumentPrintModal` — scoped `<style>` block targeting
  `[data-testid="document-print-preview"]` descendants

Same selectors + same `!important` properties = WYSIWYG.

### D3 — `skipCollisionCheck={true}` for customer-page appointments

The shared `AppointmentFormModal` has slot-collision detection (warns if
the room/doctor is already booked at this time). The customer-detail page
has no full-day appointment context — only the N most recent for this
specific customer. Running collision detection there would generate
false-positive warnings (or worse, miss real conflicts that aren't in the
loaded set).

Solution: `skipCollisionCheck` prop on the form. CustomerDetailView passes
`true`; AppointmentTab leaves it `false` (default).

Holiday confirm stays ON for both — useful when picking a future date that
lands on a clinic holiday regardless of context.

### D4 — Generic auto-fill via naming convention

When user picks a doctor in the staff-select dropdown, the form should
auto-fill `doctorLicenseNo` from the picked record's `licenseNo` field.
Hard-coding doctor-specific keys would mean adding new code every time we
add a related field (phone, email, position, EN name, etc.).

Solution: convention-based. `field.key.replace(/Name$/, '') = baseKey`.
Then look for `${baseKey}LicenseNo`, `${baseKey}Phone`, `${baseKey}Email`,
`${baseKey}Position`, `${baseKey}NameEn`, `${baseKey}Department`,
`${baseKey}Signature` in the template's fields[] and populate any that
exist. Empty record values silently skipped.

Works for `doctorName` → doctor-prefixed fields, `staffName` → staff-
prefixed, `assistantName` → assistant-prefixed. Five test cases verify
the contract.

### D5 — Phase 14.7.C (AppointmentTab refactor) deferred

Extracting the form into `AppointmentFormModal.jsx` was already a big
change (550 LoC + import wiring + 4 new tests). Refactoring AppointmentTab
to USE the shared component in the SAME commit risked breaking the
calendar grid (many state hooks entangled — filteredCustomers, monthAppts,
selectedDate, holidays loader, etc.).

Pragmatic split:
- `2728635` ships shared component + CustomerDetailView usage
- AppointmentTab keeps its inline form (works, was tested before)
- Phase 14.7.C will replace AppointmentTab's inline form with shared
  component once tested in production

Both writers produce IDENTICAL payloads to `be_appointments` (verified
via F5.1-3 tests in customer-appointments-flow.test.js). No data shape
divergence to worry about during the gap.

### D6 — Reverted inline-flex with replace_all in single edit

Used `Edit` with `replace_all: true` to swap 77 occurrences of `display:
inline-block;border-bottom:1px dotted #000` to inline-flex (e8790ba), then
reverted same way for d77a421. This is correct usage of replace_all when
the pattern is unique enough — doesn't violate the Edit-fail-silently
trap because the assertion is "all occurrences updated" and the count is
verifiable post-hoc via grep.
