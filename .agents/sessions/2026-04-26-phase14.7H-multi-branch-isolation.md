## Decisions (non-obvious — preserve reasoning)

### D1 — Multi-branch architecture: Option 1 (single project + branchId field)
User asked: "การแยกสาขาต้องแยก database กันหมดแบบ completely?" Three options were enumerated:
- **Option 1**: Single Firestore project + `branchId` field on each branch-scoped doc (ProClinic uses this)
- **Option 2**: Separate Firebase project per branch — physical isolation, federation pain
- **Option 3**: Single project + sub-collection per branch — schema migration

User clarified "เร็ว = response time" (not dev time). All 3 options have equal per-query latency when `branchId` is indexed; Option 1 wins on cross-branch reports + dev time + error rate. User confirmed Option 1.

**Why this matters for future sessions**: this is **Option 1 forever** unless legal-isolation requirements change. Don't second-guess when adding new collections — classify them in `branch-collection-coverage.test.js COLLECTION_MATRIX` (BC1.1 will fail if you forget).

### D2 — V19 narrowed `be_stock_movements` rule (not blanket-open)
The 14.7.F fix narrowed `allow update: if false` to `if hasOnly(['reversedByMovementId'])`. Single-field exception preserves audit immutability for everything else; legitimate reversal-link writes pass.

**Lesson**: any `allow update: if false` rule is a contract with the codebase. **Before adding one, grep `updateDoc(<collection>` + `tx.update(<collection>`** — if the grep is non-empty, narrow the rule to specific fields, don't blanket-block.

### D3 — Branch-future collections deferred (not retrofitted in this session)
Six collections (be_quotations, be_vendor_sales, be_online_sales, be_sale_insurance_claims, be_expenses, be_staff_schedules) have `firestore.rules` support for `branchId` filtering BUT their CRUD UIs don't currently pass `branchId` in their `saveX(data)` payloads (they use `...data` spread).

These are tracked in `branch-collection-coverage.test.js BC2.future` as `scope: 'branch-future'`. Wireup is deferred per-feature: when those tabs see active multi-branch use, add `useSelectedBranch()` in their form modals + thread `branchId` into the payload.

**Why deferred**: scope creep risk. Multi-branch infra is the foundation; per-form wireup is incremental. A single-branch clinic doesn't need them today.

### D4 — Cross-branch transfer flow already built correctly (no refactor needed)
`createStockTransfer` already validates that source batches BELONG to the source branch (`b.branchId !== src` throws). Movements properly attribute: EXPORT_TRANSFER (type 8) carries `branchId: b.branchId` (source), RECEIVE (type 9) carries `branchId: cur.destinationLocationId` (destination). Live integration test against real Firestore confirmed: 10 source units → 7 source / 3 dest after the 0→1→2 state transition.

**The cross-branch transfer system was the most complex thing the user worried about — and it was already correct.** Tests now pin the contract.

### D5 — Listener cluster mirrors 14.7.G pattern
Same shape applied 3 times to close 3 staleness gaps:
- `listenToCustomerSales` → fixes "edit sale in another tab, customer-detail purchase history doesn't update"
- `listenToCustomerAppointments` → fixes nextUpcomingAppt staleness
- `listenToAppointmentsByDate` → fixes multi-admin calendar collision (real risk for production with multiple admins)

The pattern: drop the dep array down to `[customer?.proClinicId]` (or `[selectedDate]` for the date-scoped listener); subscribe in the useEffect; cleanup unsubscribes on unmount. Legacy reload callbacks (e.g. `reloadCustomerAppointments`) kept as no-op shims so existing call sites don't break.
